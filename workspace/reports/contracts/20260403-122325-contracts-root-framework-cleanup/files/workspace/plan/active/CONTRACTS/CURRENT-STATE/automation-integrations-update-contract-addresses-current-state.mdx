## Purpose

These two workflows are the current GitHub Actions wrappers around the contracts pipeline:

- the main workflow runs the pipeline, performs a post-write `--check`, uploads artifacts, and commits changed generated data on success
- the shadow workflow runs the pipeline in `--check` mode only and uploads artifacts on any outcome

## Current Job Steps

### Main workflow current step order

1. Checkout the docs repo.
2. Set up Node.js `20`.
3. Run `npm ci --prefix operations`.
4. Run `node .github/scripts/fetch-contract-addresses.js` with optional `--dry-run` and `--skip-verify`.
5. If generation succeeded, run `node .github/scripts/fetch-contract-addresses.js --check` with optional `--skip-verify`.
6. Upload pipeline artifacts.
7. If generation failed, create or update a GitHub issue from `contract-pipeline-issue-payload.json`.
8. Fail the workflow if generation failed.
9. If generation succeeded, detect whether the repo changed.
10. If files changed and the run was not `dry_run`, commit and push the six generated files.

### Shadow workflow current step order

1. Checkout the docs repo.
2. Set up Node.js `20`.
3. Run `npm ci --prefix operations`.
4. Run `node .github/scripts/fetch-contract-addresses.js --check` with optional `--skip-verify`.
5. Upload pipeline artifacts.
6. If the check failed, create or update a GitHub issue from `contract-pipeline-issue-payload.json`.
7. Fail the workflow if the check failed.

## Current Checkout And Branch Behavior

The current checkout expression in both workflows is:

```yaml
ref: ${{ inputs.use_test_branch == 'true' && vars.TEST_BRANCH || vars.DEPLOY_BRANCH }}
```

Current implications:

- manual runs can choose `TEST_BRANCH` through `use_test_branch`
- the YAML does not define a separate scheduled ref path
- scheduled and `repository_dispatch` runs do not provide manual inputs, so the current workflow text does not explicitly force `TEST_BRANCH` on those event types

## Current Failure Behavior

### Main workflow

- The generate step runs with `continue-on-error: true`.
- A failed generate step can still upload artifacts and create or update an issue.
- The workflow then fails explicitly in `Fail on blocking contracts incident`.
- The post-write `--check` step only runs when the generate step succeeded.

### Shadow workflow

- The check step runs with `continue-on-error: true`.
- A failed shadow check can still upload artifacts and create or update an issue.
- The workflow then fails explicitly in `Fail on blocking contracts incident`.

- The main workflow already uses daily cadence.
