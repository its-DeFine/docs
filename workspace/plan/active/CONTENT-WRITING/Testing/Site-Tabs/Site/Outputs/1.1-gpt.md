## Boundary Rules

**About â†” Solutions**: conceptual system understanding vs product-fit evaluation
Rule: About owns what Livepeer is and how the network works. Solutions owns why a demand-side user should choose it for a real use case. The line is drawn at system explanation vs decision-to-adopt.

**About â†” Developers**: conceptual architecture vs technical implementation
Rule: About owns the network model, actor relationships, and canonical concepts. Developers owns code-facing integration and build patterns. The line is drawn at whether the page helps someone understand the system or implement against it.

**About â†” Gateways**: network role explanation vs gateway operation
Rule: About owns what a gateway is in the network. Gateways owns how to deploy, configure, and run one. The line is drawn at concept vs operation.

**About â†” Orchestrators**: supply-side role explanation vs node operation
Rule: About owns what orchestrators do and why they matter. Orchestrators owns hardware viability, setup, and performance. The line is drawn at understanding the role vs operating the role.

**About â†” LP Token**: conceptual economics vs participation mechanics
Rule: About owns why LPT exists and how it functions in network design. LP Token owns what token holders do with it. The line is drawn at explanation vs action.

**About â†” Community**: governance as system vs governance as social participation
Rule: About owns the governance mechanism conceptually. Community owns the discourse, groups, and participation culture around it. The line is drawn at protocol mechanism vs human coordination.

**Solutions â†” Developers**: API consumption path vs network building path
Rule: Solutions owns the decision layer for demand-side users and the simplest path framing. Developers owns implementation, extensibility, and custom build paths. The line is drawn at deciding to use Livepeer vs building on it.

**Gateways â†” Developers**: infra running vs programmable use
Rule: Developers owns code that consumes or extends Livepeer programmatically. Gateways owns running dedicated gateway infrastructure. The line is drawn at writing against the system vs operating a gateway node/service.

**Gateways â†” Orchestrators**: demand-side infra vs supply-side infra
Rule: Gateways owns job ingress, routing, and demand-side traffic handling. Orchestrators owns compute execution and supply-side performance. The line is drawn at whether the actor routes demand or fulfils compute.

**LP Token â†” Community**: voting mechanics vs governance discussion
Rule: LP Token owns how to vote, delegate, and act as a token participant. Community owns where proposals are discussed, shaped, and followed socially. The line is drawn at formal mechanism vs discourse.

**Resource HUB â†” all role tabs**: shared reference vs role-specific usage
Rule: Resource HUB owns reference material reused across multiple tabs. Role tabs own reference only when it is inseparable from a role-specific workflow. The line is drawn at reusable lookup asset vs role-bound operational context.

**Home â†” all tabs**: routing vs destination
Rule: Home owns first-visit orientation only. The moment content becomes something a user would return to for substance, it belongs in a destination tab.

## Duplication Risk Register

| Topic | Canonical owner | Resolution |
|---|---|---|
| What Livepeer is | About | All other tabs may frame relevance briefly, then link to About for the canonical explanation |
| Hosted path vs deeper build path | Solutions | Developers can assume the decision is made; Solutions owns the choice architecture |
| API quickstarts | Developers | Solutions may mention ease-of-start but must link, not duplicate setup |
| Gateway definition | About | Gateways owns operation, not the canonical system explanation |
| Gateway deployment | Gateways | Developers may reference gateway use programmatically but not operationalise it |
| Orchestrator viability and setup | Orchestrators | About may explain the role; LP Token may reference orchestrators for delegation context only |
| LPT concept | About | LP Token introduces briefly, then moves straight into action |
| Delegation workflow | LP Token | Orchestrators may reference delegation from the operator perspective but do not teach the delegator flow |
| Governance voting mechanics | LP Token | Community owns discourse and participation context, not the formal voting procedure |
| Governance/workstream discussion | Community | LP Token links out when a token holder needs the surrounding discussion context |
| Shared addresses/parameters/glossary | Resource HUB | Role tabs link to canonical references instead of re-listing them |
| Audits/changelog/reference assets | Resource HUB | Role tabs surface only context-specific excerpts, never duplicate the master reference |
| BYOC and custom pipelines | Developers | Orchestrators may mention related workload contexts but do not own build-path documentation |
| Gateway economics vs orchestrator economics | Gateways / Orchestrators (scoped) | Each tab owns its own economics only; About may explain incentives at system level without operational detail |
| Community contribution paths vs code contribution | Community / Developers (scoped) | Community owns broad participation paths; Developers owns technical contribution workflows |
