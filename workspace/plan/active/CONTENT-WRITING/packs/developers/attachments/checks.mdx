# Per-Page Quality Checks – Developers Tab Canonical Reference

Everything a Developers tab page must pass before it ships. Organised by category, each with:
- **What to check** (concrete items)
- **What success looks like** (the bar)
- **Frameworks, rules and tools that help** (where to look, what to run)

**Developers tab default audiences**: `developer` (code-first, deep-protocol, BYOC, OSS contributor) and `builder` (integration-focused, API/SDK consumer, hosted-gateway path). Per-page brief assigns one of these as primary.

**Developers tab PAGE_TYPE mix expected**: `instruction`, `tutorial`, `reference` are dominant. `concept` and `guide` are moderate. `navigation` (portal, navigator). `resource` (compendium, knowledge-hub).

Sources: `plan-canonical.md`, `content-pipeline-canonical.md`, `design-canonical.mdx`, `Frameworks.mdx`, `component-layout-decisions.mdx`, `voice-rules.md`, `section-naming.md`, `veracity.md`, `page-taxonomy-framework.mdx`, `v2-folder-governance.mdx`, `quality-gates.mdx`, `prompt-input-spec.md`, `docs-review-prompt-tiers.md`, `v2/developers/_design/developers-IA.md`.

## 1. FRONTMATTER & TAXONOMY

**What this catches**: Wrong classification, missing fields, deprecated enum values, pages that confuse the pipeline and break downstream systems.

### Checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 1.1 | All **10 required frontmatter fields** present | `title`, `sidebarTitle`, `description`, `pageType`, `audience`, `purpose`, `complexity`, `lifecycleStage`, `keywords`, OG image block – none missing, none empty |
| 1.2 | `pageType` uses **7-type canonical schema** | One of: `concept`, `tutorial`, `guide`, `instruction`, `navigation`, `reference`, `resource`. No deprecated 12-type values (`landing`, `overview`, `how_to`, `quickstart`, `faq`, `troubleshooting`, `changelog`, `glossary`). When migrating a deprecated value, also set `pageVariant` per the §1.1 migration table in Frameworks.mdx (e.g. `quickstart` → `pageType: instruction` + `pageVariant: quickstart`; `faq` → `pageType: reference` + `pageVariant: compendium`; `glossary` → `pageType: reference` + `pageVariant: compendium`; `troubleshooting` → `pageType: instruction` + `pageVariant: troubleshooting`; `changelog` → `pageType: reference` + `pageVariant: changelog`) |
| 1.3 | `pageVariant` valid if present | One of: `overview`, `specification`, `compendium`, `changelog`, `knowledge-hub`, `quickstart`, `troubleshooting`. `overview` is a variant, never a pageType |
| 1.4 | `purpose` uses **15-value canonical set** | One of: `orient`, `explain`, `learn`, `choose`, `evaluate`, `start`, `build`, `configure`, `operate`, `troubleshoot`, `verify`, `integrate`, `optimise`, `reference`, `update`. No deprecated aliases. Developers tab purpose mix skews `start`, `build`, `integrate`, `configure`, `reference`, `troubleshoot` |
| 1.5 | `audience` uses **7-token set** | One of: `founder`, `builder`, `developer`, `gateway`, `orchestrator`, `delegator`, `community`. **Developers tab default: `developer` or `builder`** per page brief. Use `developer` for code-first, deep-protocol, BYOC, OSS pages. Use `builder` for integration-focused, API/SDK, hosted-gateway pages. Not old values like `general`, `everyone`, `internal`, `platform-builder`, `gateway-operator` |
| 1.6 | `complexity` single canonical value | One of: `beginner`, `intermediate`, `advanced`. Developers tab quickstarts skew `beginner`; build pages skew `intermediate`; BYOC and protocol pages skew `advanced` |
| 1.7 | `lifecycleStage` single canonical value | One of: `discover`, `evaluate`, `setup`, `build`, `operate`, `troubleshoot`, `optimise`. Developers tab skews `setup`, `build`, `operate`, `troubleshoot` |
| 1.8 | `veracityStatus` present and honest | One of: `verified`, `unverified`, `stale`. Default for AI-generated or unreviewed content is `unverified`. `verified` only if ALL sections pass their information-type veracity standard. `status: current` requires `veracityStatus: verified` AND zero `{/* REVIEW: */}` flags |
| 1.9 | `industry` array valid if present | Max 2 entries from: `technical`, `finance`, `economics`, `business`, `marketing`, `governance`, `broadcast`, `AI`, `livepeer`. First entry dominates. Developers tab skews `technical`, `AI`, `livepeer` |
| 1.10 | `niche` array valid if present | From: `web3`, `protocol`, `tokenomics`, `AI`, `video`, `streaming`, `hardware`, `infrastructure` |
| 1.11 | `description` well-formed | Subject-first, ≤160 chars, no self-reference ("this page"), UK English |
| 1.12 | OG image block complete | All 5 OG fields present with correct fallback path |
| 1.13 | `keywords` field quality | Keywords are specific, relevant, and searcher-intent-aligned. Not generic. Each keyword must be a term a target reader would actually search for. Bad: `"livepeer", "developer"`. Good: `"livepeer ai api integration", "byoc container deployment", "comfystream realtime ai pipeline", "livepeer studio sdk typescript"`. Cross-reference `v2/developers/resources/compendium/glossary.mdx` |
| 1.14 | **Developer/builder split honoured** | If page uses `developer` register but solves a `builder` problem (or vice versa), reclassify. Developer = code-first, deep-protocol, BYOC, OSS. Builder = integration-focused, API/SDK consumer, hosted-gateway. Mismatch is a frontmatter failure, not a voice failure |

### What success looks like

Every Developers page classifiable by a cold-start agent that produces the same taxonomy a human would. Zero `missing-fields` pages site-wide. No deprecated enum values surviving in any frontmatter. Every developer/builder split is correct.

## 2. VOICE & COPY

**What this catches**: Generic AI slop, wrong register for the audience, US English, hedging, filler, self-referential openers, marketing language adjacent to technical content, prose explanations of code, missing error-state coverage.

### Universal checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 2.1 | **UK English throughout** | No US spellings: optimize→optimise, behavior→behaviour, color→colour, center→centre, labeled→labelled, canceled→cancelled, traveling→travelling, -ize→-ise (where both standard). Full table at the bottom of `voice-rules.md` |
| 2.2 | **Zero banned words** | None of: `effectively`, `essentially`, `basically`, `meaningful`, `significant`, `real` (as intensifier), `various`, `several`, `obviously`, `clearly` |
| 2.3 | **Zero banned phrases** | None of: "This section covers", "This page covers/explains/walks you through", "Understanding X is essential", "It is important to note", "As mentioned above", "and so on"/"etc.", "rather than", "what it takes", "it should be noted", "not just X", "can generate"/"may produce" in value claims, "The reason is straightforward", "among other factors", "low but not zero", "once X is stable", "depends on various" |
| 2.4 | **Zero banned constructions** | No `not [X]` in value statements. No unresolved `if [condition]` in body prose (exception: code blocks, exact config conditions). No `This page [verb]` self-reference. No `can/may [verb]` in value claims |
| 2.5 | **Opening order correct** | Value or outcome before mechanism. Fact before caveat. Reader benefit before system description. Introduction opens with subject, not caveat or self-reference |
| 2.6 | **Paragraph discipline** | One paragraph = one job. Lead sentence states the fact. Final sentence ends on fact, number, or next step – never a hedge |
| 2.7 | **Audience register correct** | Voice matches the page's `audience` token. **Developers tab uses both `developer` and `builder` registers** per page brief. See per-audience rules below |
| 2.8 | **Per-audience prohibited phrases absent** | Each audience has specific prohibited phrases. Developer: no "with just a few lines", "the SDK makes it simple", "easily integrated with", "don't worry about the details", "as you may know". Builder: no "as you know, Livepeer is", "built on blockchain technology", any opening that explains the network before the integration value |
| 2.9 | **No passive value statements** | Every value claim is quantified or concrete. "Cost-effective" → "$0.019/megapixel for text-to-image". "Fast" → "sub-3-second latency at p50" |
| 2.10 | **No hedging openers** | Page does not open with a caveat, disclaimer, or conditional. Goes straight to the point |
| 2.11 | **Terminology locked and consistent** | Uses terms from the locked terminology set (BYOC, NaaP, LIP-92, ComfyStream, PyTrickle etc.). No deprecated or conflicted terms |
| 2.12 | **No em-dashes** | Zero em-dashes in MDX content. Use comma, semicolon, colon, or rewrite the sentence. Hook enforcement active |
| 2.13 | **Entity-led voice throughout** | Every paragraph starts with a system fact, API behaviour, or reader outcome. Never "we", "our", "you will learn", "in this section we explore". Subject-first, always |
| 2.14 | **No hedging verbs in value claims** | No "can help you", "allows you to", "enables you to", "helps you", "may provide". Assert directly: "the SDK returns a base64 image" not "the SDK can return a base64 image" |
| 2.15 | **Description field not self-referential** | Frontmatter `description` does not start with "This page/section/document describes/explains/outlines/covers/walks". Lead with the subject |
| 2.16 | **Deprecated terms absent** | Zero instances of: "Broadcaster" (use Gateway), "Pool worker" (use Pool node), "Combined mode" (use Dual mode), "Hybrid" (use Dual mode), "Transcoder" as synonym for Orchestrator |
| 2.17 | **Universal terms consistent** | Orchestrator, Gateway, Delegator, LPT, Transcoding, Staking match `universal-terms.md` canonical definitions |
| 2.18 | **Spell check passes** | cspell against `tools/config/cspell.json` dictionary. Zero unknown words (add to dictionary if legitimate Livepeer term) |
| 2.19 | **Terms match TERMINOLOGY-COLLATE canonical** | Every Livepeer-specific term matches definition in `TERMINOLOGY-COLLATE/consolidated/glossary-*.md` |
| 2.20 | **Per-tab terminology correct** | Uses terms appropriate to its tab per `TERMINOLOGY-COLLATE/per-tab/glossary-developers.md` |
| 2.21 | **First use of specialised term defined or linked** | Every Livepeer-specific term defined on first use. Reader never encounters undefined term. For developer-register pages: protocol terms (round, active set, ticket) may be referenced terser since the audience has read-the-source competence |
| 2.22 | **Terminology lock respected** | All locked terms used exactly as defined |

### Developer-register checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 2.D1 | **Code-first opening** (developer-register, instruction/tutorial pages) | For `instruction` and `tutorial` pages, the first content beneath orientation is a working code sample, not prose. Prose explains the code, not the other way round |
| 2.D2 | **Function/API/method named in prose has code example or GitHub link** | Every function, API endpoint, SDK method, or CLI flag named in prose either has an inline code example or a link to its GitHub source/release. Naming a function without showing it is a fail |
| 2.D3 | **Versions stated are current** | Every version number (go-livepeer, ai-runner, ComfyStream, SDK npm/PyPI/Go module) is the current release at time of write. State the version explicitly. No "latest" without a number |
| 2.D4 | **Error states and edge cases in main content** | Error states, edge cases, and failure modes belong in the main content, not buried in `<Note>` at the end. A developer page that hides error handling fails this check |
| 2.D5 | **No prose explanations of self-evident code** | Code that speaks for itself does not get a prose paragraph restating it. Comment in the code, or move on |
| 2.D6 | **No marketing language adjacent to technical content** | "Powerful AI pipeline" adjacent to `import livepeer` is a fail. Strip marketing tone. Technical content earns trust through precision, not adjectives |
| 2.D7 | **`<Note>` and `<Info>` not used for primary content** | If content belongs in the main explanation, do not hide it in a Note or Info block. These components are for adjacent context, not core information |

### Builder-register checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 2.B1 | **Integration value before network architecture** | Page leads with what the integration gives the builder's product or users – not what Livepeer is or how the network works |
| 2.B2 | **Exact API endpoint or SDK method named first** | State the exact API endpoint or SDK method first, then the context. Not "you can use the Livepeer SDK to..." but "`livepeer.generate.textToImage()` returns..." |
| 2.B3 | **Working code in most-likely language** | Show working code in the most-likely language for the builder audience: JS/TS first, then Python, then Go. SDK examples preferred over raw curl unless the page is REST-specific |
| 2.B4 | **Full request/response cycle shown** | Show the full request/response cycle, not just "call this endpoint". Include authentication, expected response shape, error response shape |
| 2.B5 | **Prerequisites stated explicitly** | Which API keys, which SDK version, which plan or tier applies – state explicitly at the top |
| 2.B6 | **No node-operator or gateway terminology without translation** | Builder pages do not use orchestrator/gateway/round terminology without translating to integration implications. The builder is shipping a product, not running infrastructure |
| 2.B7 | **No web3 mechanics without framing** | Gas, transactions, on-chain interactions are not assumed knowledge. If the builder must understand them, the page frames why and links to a primer |
| 2.B8 | **No mixing protocol mechanics with integration steps** | "How the protocol works" and "how to integrate" are separate pages. A builder integration page that lapses into protocol explanation is a SPLIT candidate |

### What success looks like

Read any Developers paragraph aloud: developer-register pages sound like a senior engineer wrote notes for another senior engineer. Builder-register pages sound like a staff engineer wrote a working integration guide for another staff engineer. No sentence reads like marketing. Code is primary; prose is supporting.

## 3. SECTION NAMING & HEADINGS

**What this catches**: Generic, vague, or unstable headings that fail to communicate what a section actually contains. Marketing headings on developer pages. Tutorial headings that hide the step.

### Checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 3.1 | Every heading scores **≥20/25** on the naming rubric | 5 dimensions × 5 points: Precision, Depth, Stability, Clarity, Conciseness. **EXEMPT: `Related Pages` heading – do not score, do not include in the Heading Score Table, do not reference in check 3.2 or 3.7. Approved structural element. Treat as invisible to all heading checks.** |
| 3.2 | No banned or weak standalone heading terms | Three tiers per Frameworks.mdx §4.1: **Banned**: `Basics`, `Notes`, `How It Works`, `See Also`, `Conclusion`, `What's Next`. **Avoid**: `Overview`, `Details`, `Information`, `Introduction`, `Summary`, `Options`, `Background`, `Next Steps`, `Further Reading`. **OK**: `Types`, `Modes`, `Profiles`, `Prerequisites`, `Configuration`, `Setup`, `Resources`, `Related`. Banned terms also apply to heading rename suggestions. **EXEMPT: `Related Pages`** |
| 3.3 | No literal contrast labels | No `X vs Y` headings – name the governing concept instead. "Studio API vs Gateway API" → "API Access Tiers" |
| 3.4 | Domain-anchor rule applied | Heading includes the domain noun when needed to be interpretable in isolation |
| 3.5 | Heading names the concept, not examples | "AI Pipeline Endpoints" not "text-to-image, image-to-video, audio-to-text" |
| 3.6 | Title well-formed | 1–3 words, concept-derived, no banned forms |
| 3.7 | Sounds like an expert editorial choice | Would a senior technical editor choose this heading? |
| 3.8 | **Per-pageType naming style applied** | `reference` pages use literal/findability naming (e.g. "Authentication", "Rate Limits"). `instruction`/`tutorial` pages use task-oriented naming (e.g. "Make Your First Inference Call"). `concept` pages use governing-concept naming. `navigation` pages use map language |
| 3.9 | **Per-audience register in headings** | `developer` headings use specialised technical terms (e.g. "FrameProcessor Pattern", "PyTrickle StreamServer"). `builder` headings use integration-outcome terms (e.g. "Authenticate Your Requests", "Handle Rate Limits"). Per `section-naming.md` per-audience guidance |
| 3.10 | **Domain-anchor rule applied** | Heading includes the domain noun when needed to be interpretable in isolation |

### Scoring rubric (per heading)

| Dimension | 5 (best) | 1 (worst) |
|-----------|----------|-----------|
| **Precision** | Names the exact concept | Could mean anything |
| **Depth** | Signals the intellectual level | Surface-level label |
| **Stability** | Survives content changes | Breaks if examples change |
| **Clarity** | Instantly understood by target audience | Requires reading the section to understand |
| **Conciseness** | Minimal words, maximum signal | Wordy or padded |

### What success looks like

A reader scanning only headings on a Developers page builds an accurate plan of what they will do or look up. Site-wide: all headings ≥20/25.

## 4. PAGE STRUCTURE & CONTENT ARCHITECTURE

**What this catches**: Pages trying to do too many things, broken journeys, missing handoffs, content that doesn't serve the reader's actual job. Developers-specific: code samples not testable, error states absent, version drift, mixed developer/builder scope.

### Checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 4.1 | **One purpose, one audience, one job** | Page serves exactly one reader doing exactly one thing. Developer/builder mismatch is a frontmatter/scope failure, not a copy fix |
| 4.2 | **Purpose statement test passes** | Can state: "This page lets the [developer or builder] [concrete action]." Concrete action = make a call, deploy a container, set up auth, integrate webhooks, etc. |
| 4.3 | **PREV_PAGE / NEXT_PAGE adjacency correct** | Reader arriving from previous page has the knowledge needed. Reader leaving to next page knows why. No knowledge gaps. Quickstart → build → guide adjacency must hold |
| 4.4 | **No dead ends** | Every page has a clear next step – next quickstart, next build chapter, related guide, or graduation to Gateways/Solutions |
| 4.5 | **Prerequisites stated or linked** | All prerequisite knowledge explicitly stated at the top: API keys required, SDK version, language runtime, hardware (e.g. CUDA for ComfyStream), credentials |
| 4.6 | **Out-of-scope is clear** | What this page does NOT cover is understood. Per Notion IA Proposal: video streaming aggregator pages link to Solutions platforms (Studio, Streamplace) rather than recreate content |
| 4.7 | **Information type per section is correct** | Each section's information type matches the permitted types for the page's `purpose` (see Purpose→Information Type mapping in Frameworks.mdx §1.6). Developers tab skews `procedural`, `technical`, `factual`, `analytical` |
| 4.8 | **No content duplication from adjacent sections** | Content is not repeated from neighbouring pages. SDK reference belongs in resources/reference; build pages link, do not copy |
| 4.9 | **Section orientation page present** (section-level check) | Every section (Concepts, Get Started, Build, Guides, Opportunities, Resources) has an entry page that orients the reader |
| 4.10 | **Cross-tab links at journey intersections** | At least 3 cross-tab links per page where reader graduations are expected (to Gateways for self-hosted, Solutions for managed video, About for protocol context) |
| 4.11 | **Page answers its question COMPLETELY (Discord test)** | Can you link this ONE page on Discord and know the person gets a complete answer without needing another page? |
| 4.12 | **Page size appropriate for scope** | Concept/guide pages: minimum 5KB substantive content. Pages under 2KB are stubs that should not be in published nav. Gold standard pages are 10-60KB |
| 4.13 | **No TODO/REVIEW comments without tracking** | Zero `{/* TODO` comments in published content. Zero `{/* REVIEW` comments without a linked tracking issue |
| 4.14 | **Flat layout where appropriate** | Build pages use tabs for language variants (TS/Python/Go), not sub-folders. Sub-pages only for genuinely separate journeys |
| 4.15 | **Trade-offs present** | Every concept/architecture/decision page on Developers names at least one trade-off, limitation, or failure condition. BYOC: container size, cold start, debugging cost. ComfyStream: GPU requirement, frame rate ceiling. Studio API: pricing tier, rate limit |
| 4.16 | **Content pass context block completable** | All 15 fields in `content-pass.md` context block can be filled. If any cannot be filled, the page's position in the pipeline is undefined |
| 4.17 | **Code samples – every code block has language tag** (Developers-specific) | Every fenced code block carries a language identifier (`bash`, `typescript`, `python`, `go`, `json`, `yaml`, `dockerfile`). Bare ``` fences are a fail. Cross-reference with check 5.20 (icon and title metadata also required) |
| 4.18 | **Code-first opening** (Developers-specific, instruction/tutorial only) | For pages with `pageType: instruction` or `pageType: tutorial`, the first content beneath orientation is a working code sample, not prose. Cross-reference 2.D1 |
| 4.19 | **Error states and edge cases in main content** (Developers-specific) | Error states, common failure modes, and edge cases live in the main content flow, not buried in a `<Note>` at the end. Cross-reference 2.D4 |
| 4.20 | **Function/API/method named in prose has code example or link** (Developers-specific) | Every function, API endpoint, SDK method, or CLI flag named in prose either has an inline code example or a link to its GitHub source/release. Cross-reference 2.D2 |

### What success looks like

A developer or builder following the journey from quickstart to build to operations never hits a wall. Every page either ships them with a working integration or graduates them to the right next surface (Gateways, Solutions, About).

## 5. LAYOUT, COMPONENTS & TEMPLATE

**What this catches**: Wrong components for the page type, missing required sections, unapproved components, MDX that won't render, inconsistent page structure across the same pageType. Developers-specific: code blocks without language/icon/title metadata, raw Steps instead of StyledSteps, untagged language tabs.

### Checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 5.1 | **Correct template for pageType + pageVariant** | Page uses the structural template defined for its type. See component-layout-decisions.mdx matrix |
| 5.2 | **Required sections present** | Each pageType has mandatory sections. Developers-tab focus: `instruction`/quickstart requires Prerequisites, Steps, Next Steps. `tutorial` requires Prerequisites, Steps, Verification, Related. `reference` requires Reference. `guide` requires Overview + Steps/H2 |
| 5.3 | **Only approved components used** | Components match the "Preferred" column for the pageType. No unapproved components |

# REVIEW EXECUTION GUIDE

## How to run a per-page review (Developers tab)

Based on `docs-review-prompt-tiers.md` Tier 3 (Single Page):

### Pre-flight (before touching the page)
1. **Check for `generated-file-banner`** – if present, read the generation script before logging any findings
2. Read the page's frontmatter – classify against canonical enums. **Read each field value directly from the file. Do not infer `purpose` from `pageType`. Do not infer `audience` from the file path.** Developers-tab risk: assuming `developer` when the page actually serves `builder` (or vice versa) – these have different registers and structure
3. Identify the page's audience, purpose, and pageType – from the actual field values
4. Load the relevant voice rules for that audience (developer or builder)
5. Load the component matrix for that pageType
6. Note the page's position in the journey – read `docs.json` to confirm the navigation sequence

### Review pass order
1. **Frontmatter** (Category 1) – all 10+ fields present, all values canonical, developer/builder split honoured
2. **Structure** (Category 4) – one purpose, one job, code-first opening, error states in main content, Discord test, prerequisites, handoffs, trade-offs
3. **Headings** (Category 3) – score each 20/25+, per-pageType naming, per-audience register, domain-anchor

## REPORTING RULES

**Fixes must be executable.** No "add appropriate X". Specify the exact value, line, or action.

**Definition strings in fixes must be REVIEW placeholders.** Use `{/* REVIEW: [term] – verify definition against [source] before inserting */}`.

**Schema rules must be cited.** Any finding that asserts a schema constraint must cite the specific checks.mdx section.

**Read fields directly.** State the actual frontmatter field value – read it, do not infer it.

**Characterise the error correctly.** When a frontmatter value fails: (a) deprecated alias, (b) wrong field, (c) invalid value.

**Evaluate the current state only.** All findings evaluate the page as it currently exists.

**Result column must match the detail conclusion.**

**Veracity findings: TESTED or NOT-TESTED.** Every code block, command, or factual claim must be labelled.

**Component approval: NOT-TESTED unless the catalog was read.**

**Verify every cited occurrence.** Quote the content at the cited line.

**Link verification.** Use `operations/scripts/validators/content/structure/check-docs-path-sync.js`.

**Navigation sequence.** Check 4.3 requires reading `docs.json`.

**Self-consistency corrections are only complete when ALL instances are updated.**

**Findings must appear in all applicable sections.** No orphaned findings.

**Severity levels.** Use only: CRITICAL / HIGH / MEDIUM / INFO.

**Em-dash prohibition applies to all visible text.**

**Heading rename suggestions** must not introduce BANNED heading terms.

**veracityStatus fix must be internally consistent.**

**First use in Mermaid/diagram labels counts as first use.**

**Dead imports.** Do not flag.

**Deduplication.** Log each finding once in the most specific category.

**Developers-tab specific:** Code blocks without language/icon/title metadata are HIGH findings under check 5.20. Untested code is HIGH under 6.2. Mixed developer/builder scope is CRITICAL under 1.14 / 4.1. Missing error-state coverage on instruction/tutorial pages is HIGH under 4.19 / 2.D4.

| Severity | Meaning | Action |
|----------|---------|--------|
| **CRITICAL** | Breaks downstream phase, produces structurally wrong output, or fails on common page type | Fix before any further work |
| **HIGH** | Incorrect/incomplete on specific cases but does not break downstream | Fix recommended before shipping |
| **MEDIUM** | Inconsistent but does not break downstream consumption | Monitor; fix in next pass |
| **INFO** | Expected variation or minor style preference | Document; no action required |
