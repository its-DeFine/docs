# Per-Page Quality Checks – Delegators Tab Canonical Reference

Everything a Delegators / LP Token tab page must pass before it ships. Organised by category, each with:
- **What to check** (concrete items)
- **What success looks like** (the bar)
- **Frameworks, rules and tools that help** (where to look, what to run)

**Delegators tab default audience**: `delegator`. Pages serving the Governance Entrant or community researcher may flag a secondary audience inline, but the primary token in frontmatter remains `delegator`.

**Delegators tab PAGE_TYPE mix expected**: `concept`, `instruction`, `reference` are dominant. `tutorial` is moderate (first-delegation walkthrough). `navigation` (portal, disambiguation pages) and `resource` (videos, exchanges, blogs) are present.

Sources: `plan-canonical.md`, `content-pipeline-canonical.md`, `design-canonical.mdx`, `Frameworks.mdx`, `component-layout-decisions.mdx`, `voice-rules.md`, `section-naming.md`, `veracity.md`, `page-taxonomy-framework.mdx`, `v2-folder-governance.mdx`, `quality-gates.mdx`, `prompt-input-spec.md`, `docs-review-prompt-tiers.md`, `v2/delegators/_design/delegators-IA.md`.

## 1. FRONTMATTER & TAXONOMY

**What this catches**: Wrong classification, missing fields, deprecated enum values, pages that confuse the pipeline and break downstream systems.

### Checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 1.1 | All **10 required frontmatter fields** present | `title`, `sidebarTitle`, `description`, `pageType`, `audience`, `purpose`, `complexity`, `lifecycleStage`, `keywords`, OG image block – none missing, none empty |
| 1.2 | `pageType` uses **7-type canonical schema** | One of: `concept`, `tutorial`, `guide`, `instruction`, `navigation`, `reference`, `resource`. No deprecated 12-type values (`landing`, `overview`, `how_to`, `quickstart`, `faq`, `troubleshooting`, `changelog`, `glossary`). When migrating a deprecated value, also set `pageVariant` per the §1.1 migration table in Frameworks.mdx (e.g. `quickstart` → `pageType: instruction` + `pageVariant: quickstart`; `faq` → `pageType: reference` + `pageVariant: compendium`; `glossary` → `pageType: reference` + `pageVariant: compendium`; `troubleshooting` → `pageType: instruction` + `pageVariant: troubleshooting`; `changelog` → `pageType: reference` + `pageVariant: changelog`) |
| 1.3 | `pageVariant` valid if present | One of: `overview`, `specification`, `compendium`, `changelog`, `knowledge-hub`, `quickstart`, `troubleshooting`. `overview` is a variant, never a pageType |
| 1.4 | `purpose` uses **15-value canonical set** | One of: `orient`, `explain`, `learn`, `choose`, `evaluate`, `start`, `build`, `configure`, `operate`, `troubleshoot`, `verify`, `integrate`, `optimise`, `reference`, `update`. No deprecated aliases. Delegators tab purpose mix skews `orient`, `explain`, `learn`, `choose`, `start`, `operate`, `verify`, `reference` |
| 1.5 | `audience` uses **7-token set** | One of: `founder`, `builder`, `developer`, `gateway`, `orchestrator`, `delegator`, `community`. **Delegators tab default: `delegator`.** Governance-research surfaces may use `community` only on secondary resource pages (forum context, primary-source links). Not old values like `general`, `everyone`, `internal`, `platform-builder`, `gateway-operator` |
| 1.6 | `complexity` single canonical value | One of: `beginner`, `intermediate`, `advanced`. Delegators tab skews `beginner` and `intermediate`. Advanced pages (governance proposal authorship, treasury allocation analysis) are rare and must be justified in the page brief |
| 1.7 | `lifecycleStage` single canonical value | One of: `discover`, `evaluate`, `setup`, `build`, `operate`, `troubleshoot`, `optimise`. Delegators tab skews `discover`, `evaluate`, `setup`, `operate` |
| 1.8 | `veracityStatus` present and honest | One of: `verified`, `unverified`, `stale`. Default for AI-generated or unreviewed content is `unverified`. `verified` only if ALL sections pass their information-type veracity standard. `status: current` requires `veracityStatus: verified` AND zero `{/* REVIEW: */}` flags |
| 1.9 | `industry` array valid if present | Max 2 entries from: `technical`, `finance`, `economics`, `business`, `marketing`, `governance`, `broadcast`, `AI`, `livepeer`. Delegators tab skews `finance`, `economics`, `tokenomics`, `governance`, `livepeer` |
| 1.10 | `niche` array valid if present | From: `web3`, `protocol`, `tokenomics`, `AI`, `video`, `streaming`, `hardware`, `infrastructure` |
| 1.11 | `description` well-formed | Subject-first, ≤160 chars, no self-reference ("this page"), UK English |
| 1.12 | OG image block complete | All 5 OG fields present with correct fallback path |
| 1.13 | `keywords` field quality | Keywords are specific, relevant, and searcher-intent-aligned. Bad: `"livepeer", "delegator"`. Good: `"how to stake lpt arbitrum", "lpt reward cut", "livepeer delegator earnings", "lpt unbonding period", "choose livepeer orchestrator"`. Cross-reference `v2/delegators/resources/compendium/glossary.mdx` |
| 1.14 | **Earnings/risk pages flagged correctly** (Delegators-specific) | Any page making earnings, yield, or risk claims must use `industry` containing `finance` or `economics`, and `purpose` of `evaluate`, `learn`, or `operate`. Earnings claims hidden inside `concept` pages without correct industry tags are a frontmatter failure |

### What success looks like

Every Delegators page classifiable by a cold-start agent that produces the same taxonomy a human would. Zero `missing-fields` pages site-wide. No deprecated enum values surviving in any frontmatter. Earnings/risk content correctly classified.

## 2. VOICE & COPY

**What this catches**: Generic AI slop, wrong register for the audience, US English, hedging, filler, self-referential openers, technical jargon dumped on non-technical readers, marketing language adjacent to financial claims, "set and forget" framing, naked earnings claims.

### Universal checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 2.1 | **UK English throughout** | No US spellings: optimize→optimise, behavior→behaviour, color→colour, center→centre, labeled→labelled, canceled→cancelled, traveling→travelling, -ize→-ise (where both standard). Full table at the bottom of `voice-rules.md` |
| 2.2 | **Zero banned words** | None of: `effectively`, `essentially`, `basically`, `meaningful`, `significant`, `real` (as intensifier), `various`, `several`, `obviously`, `clearly` |
| 2.3 | **Zero banned phrases** | None of: "This section covers", "This page covers/explains/walks you through", "Understanding X is essential", "It is important to note", "As mentioned above", "and so on"/"etc.", "rather than", "what it takes", "it should be noted", "not just X", "can generate"/"may produce" in value claims, "The reason is straightforward", "among other factors", "low but not zero", "once X is stable", "depends on various" |
| 2.4 | **Zero banned constructions** | No `not [X]` in value statements. No unresolved `if [condition]` in body prose. No `This page [verb]` self-reference. No `can/may [verb]` in value claims |
| 2.5 | **Opening order correct** | Value or outcome before mechanism. Fact before caveat. Reader benefit before system description. Introduction opens with subject, not caveat or self-reference |
| 2.6 | **Paragraph discipline** | One paragraph = one job. Lead sentence states the fact. Final sentence ends on fact, number, or next step – never a hedge |
| 2.7 | **Audience register correct** | Voice matches the page's `audience` token. **Delegators tab default register is `delegator`.** Accessible plain-language, decision-support style |
| 2.8 | **Per-audience prohibited phrases absent** | Delegator-specific prohibitions: "Simply stake your tokens to...", "As a delegator, you will automatically...", any claim about earnings without stating the dependency (orchestrator performance, reward cut, protocol inflation rate, time horizon) |
| 2.9 | **No passive value statements** | Every value claim is quantified or concrete. "You earn rewards" → "You earn LPT rewards each round, sized by your share of the orchestrator's stake and adjusted by their reward cut" |
| 2.10 | **No hedging openers** | Page does not open with a caveat, disclaimer, or conditional. Goes straight to the point |
| 2.11 | **Terminology locked and consistent** | Uses terms from the locked terminology set (LPT, orchestrator, reward cut, fee cut, active set, round, bonding, thawing, unbonding). No deprecated or conflicted terms. Check `decisions/terminology-tracking.md` |
| 2.12 | **No em-dashes** | Zero em-dashes in MDX content. Use comma, semicolon, colon, or rewrite the sentence |
| 2.13 | **Entity-led voice throughout** | Every paragraph starts with a system fact or reader outcome. Never "we", "our", "you will learn", "in this section we explore". Subject-first, always |
| 2.14 | **No hedging verbs in value claims** | No "can help you", "allows you to", "enables you to", "helps you", "may provide". Assert directly: "delegating LPT earns rewards" not "delegating LPT can help you earn rewards" |
| 2.15 | **Description field not self-referential** | Frontmatter `description` does not start with "This page/section/document describes/explains/outlines/covers/walks". Lead with the subject |
| 2.16 | **Deprecated terms absent** | Zero instances of: "Broadcaster" (use Gateway), "Pool worker" (use Pool node), "Combined mode" (use Dual mode), "Hybrid" (use Dual mode), "Transcoder" as synonym for Orchestrator |
| 2.17 | **Universal terms consistent** | Orchestrator, Gateway, Delegator, LPT, Transcoding, Staking match `universal-terms.md` canonical definitions |
| 2.18 | **Spell check passes** | cspell against `tools/config/cspell.json` dictionary |
| 2.19 | **Terms match TERMINOLOGY-COLLATE canonical** | Every Livepeer-specific term matches definition in `TERMINOLOGY-COLLATE/consolidated/glossary-*.md` |
| 2.20 | **Per-tab terminology correct** | Uses terms appropriate to its tab per `TERMINOLOGY-COLLATE/per-tab/glossary-delegators.md` |
| 2.21 | **First use of specialised term defined or linked** | Every Livepeer-specific term defined on first use. Reader never encounters undefined term. Cross-reference 2.D1 below – Delegators tab applies the strictest first-use discipline of any tab |
| 2.22 | **Terminology lock respected** | All locked terms used exactly as defined |

### Delegator-register checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 2.D1 | **Jargon discipline – every protocol term defined on first use** | Every protocol term (LPT, orchestrator, reward cut, fee cut, active set, round, bonding, thawing, claimEarnings, treasury) defined on first use **inline in plain language**. Subsequent use naked. No DeFi jargon (impermanent loss, AMM, slippage, MEV) without inline definition. Acronyms (CEX, DEX, EVM, RPC) defined on first use |
| 2.D2 | **Plain-language economics** | "You receive X% of the rewards the orchestrator earns" not "you receive a proportional share of inflation-adjusted emissions". Every economic mechanism stated in concrete reader terms before any protocol-language description |
| 2.D3 | **Earnings dependency stated** (Delegators-specific, mandatory) | Every claim about earnings, rewards, or yield includes the dependency. Must state at least one of: orchestrator performance, reward cut percentage, fee cut percentage, protocol inflation rate, time horizon, active-set status, gas costs. Example: "Annual yield depends on your orchestrator's reward cut, the protocol inflation rate this round, and how often the orchestrator calls reward()." Naked "you can earn" or "earn passive income" is a fail |
| 2.D4 | **No "set and forget" framing** | Delegation and governance change over time: orchestrators raise reward cuts, exit the active set, miss reward calls. Pages must signal where ongoing engagement is required. Phrases like "set and forget", "passive income", "fully automated" are fails unless immediately qualified with the engagement requirement |
| 2.D5 | **Concrete steps, not vague guidance** | "Stake your tokens" → explain exactly how (open Explorer, connect wallet, select orchestrator, sign bond transaction, confirm). State the actual sequence. State where the user is, what they sign, what each transaction does |
| 2.D6 | **Decision support, not decision avoidance** | The audience is a smart capable decision-maker. Pages should support the decision (here are the trade-offs, here is what to look for, here is how to compare options) rather than refuse to advise (e.g. "everyone's situation is different – you decide"). Trade-offs named, not hidden |
| 2.D7 | **Governance: stakes before mechanism** | For governance pages, explain what is at stake in a vote (treasury allocation, protocol parameter change, LIP adoption) before describing the voting mechanism |

### Delegators-tab interface restrictions

| # | Check | Pass criteria |
|---|-------|---------------|
| 2.D8 | **No CLI commands, no shell syntax, no bash blocks** (Delegators-specific) | Delegators tab pages do not contain CLI commands, shell prompts, `bash` code fences, or terminal screenshots. CLI is not the delegator's interface. **Exception:** contract addresses and copyable on-chain values (e.g. `BondingManager` address, transaction hash, function selector) are permitted on bridging and contract-approval pages where the user must paste the value into a wallet. JSON or hex code blocks for these values are acceptable; they are data references, not commands |
| 2.D9 | **Wallet flows in prose, not in code blocks** | Sequential wallet actions (connect wallet, switch network, approve transaction, confirm transaction) are described in prose with screenshots or `<StyledSteps>`, not in code blocks. The reader does not paste wallet flows into a terminal |

### What success looks like

Read any Delegators paragraph aloud and it sounds like a clear adviser explaining a financial decision to a capable adult. No hedging, no condescension, no jargon dumped without definition. Every earnings claim names its dependency. No "set and forget" framing on pages where ongoing engagement matters. Zero CLI commands.

## 3. SECTION NAMING & HEADINGS

**What this catches**: Generic, vague, or unstable headings that fail to communicate what a section actually contains. Marketing-style section labels on financial pages. Protocol jargon as standalone heading.

### Checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 3.1 | Every heading scores **≥20/25** on the naming rubric | 5 dimensions × 5 points: Precision, Depth, Stability, Clarity, Conciseness. **EXEMPT: `Related Pages` heading – do not score, do not include in the Heading Score Table, do not reference in check 3.2 or 3.7. Approved structural element. Treat as invisible to all heading checks.** |
| 3.2 | No banned or weak standalone heading terms | Three tiers per Frameworks.mdx §4.1: **Banned**: `Basics`, `Notes`, `How It Works`, `See Also`, `Conclusion`, `What's Next`. **Avoid**: `Overview`, `Details`, `Information`, `Introduction`, `Summary`, `Options`, `Background`, `Next Steps`, `Further Reading`. **OK**: `Types`, `Modes`, `Profiles`, `Prerequisites`, `Configuration`, `Setup`, `Resources`, `Related`. **EXEMPT: `Related Pages`** |
| 3.3 | No literal contrast labels | No `X vs Y` headings – name the governing concept instead. "Delegating vs Holding" → "Delegation Outcomes" |
| 3.4 | Domain-anchor rule applied | Heading includes the domain noun when needed to be interpretable in isolation. "Reward Cut" preferred over "Cuts" |
| 3.5 | Heading names the concept, not examples | "Orchestrator Selection Criteria" not "Reward Cut, Fee Cut, Active Set" |
| 3.6 | Title well-formed | 1–3 words, concept-derived, no banned forms |
| 3.7 | Sounds like an expert editorial choice | Would a senior technical editor choose this heading? |
| 3.8 | **Per-pageType naming style applied** | `reference` pages use literal/findability naming (e.g. "Contract Addresses", "Protocol Parameters"). `instruction` pages use task-oriented naming (e.g. "Bond Your LPT", "Switch Orchestrator"). `concept` pages use governing-concept naming. `navigation` pages use map language |
| 3.9 | **Per-audience register in headings** | `delegator` headings use finance/governance register, accessible to non-technical readers. Avoid heavy protocol jargon as standalone heading. "Reward Mechanics" not "BondingManager.reward()". Per `section-naming.md` per-audience guidance |
| 3.10 | **Domain-anchor rule applied** | Heading includes the domain noun when needed to be interpretable in isolation |

### Scoring rubric (per heading)

| Dimension | 5 (best) | 1 (worst) |
|-----------|----------|-----------|
| **Precision** | Names the exact concept | Could mean anything |
| **Depth** | Signals the intellectual level | Surface-level label |
| **Stability** | Survives content changes | Breaks if examples change |
| **Clarity** | Instantly understood by target audience | Requires reading the section to understand |
| **Conciseness** | Minimal words, maximum signal | Wordy or padded |

### What success looks like

A reader scanning only headings on a Delegators page can decide whether the page answers their question. Site-wide: all headings ≥20/25.

## 4. PAGE STRUCTURE & CONTENT ARCHITECTURE

**What this catches**: Pages trying to do too many things, broken journeys, missing handoffs, content that doesn't serve the reader's actual job. Delegators-specific: ordering failures (bond before bridge), missing CEX-buyer onboarding, missing fiat-onramp persona content, mixed delegator/governance scope.

### Checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 4.1 | **One purpose, one audience, one job** | Page serves exactly one reader doing exactly one thing. Yield-seeker and governance-entrant content rarely belongs on the same page – flag and split |
| 4.2 | **Purpose statement test passes** | Can state: "This page lets the delegator [concrete action]." Concrete action = bridge LPT, choose an orchestrator, bond, claim earnings, vote, exit |
| 4.3 | **PREV_PAGE / NEXT_PAGE adjacency correct** | Reader arriving from previous page has the knowledge needed. **Critical Delegators ordering: Bridge before Bond. Choose orchestrator before Bond. Bond before Claim. Unbond before Withdraw.** No knowledge gaps in the sequence |
| 4.4 | **No dead ends** | Every page has a clear next step. Delegators tab pages either deepen delegation knowledge or graduate to a related surface (Governance, Treasury, About for protocol context) |
| 4.5 | **Prerequisites stated or linked** | All prerequisite knowledge explicitly stated. Wallet, ETH for gas, LPT location (mainnet vs Arbitrum), wallet network setup. Reader achieves purpose without external context |
| 4.6 | **Out-of-scope is clear** | What this page does NOT cover is understood. Delegators tab does not own how to run an orchestrator (Orchestrators tab) or how to integrate APIs (Developers tab). That boundary must be visible |
| 4.7 | **Information type per section is correct** | Each section's information type matches the permitted types for the page's `purpose`. Delegators tab skews `procedural`, `conceptual`, `factual`, `evaluative`, `analytical` |
| 4.8 | **No content duplication from adjacent sections** | Content is not repeated from neighbouring pages. Reward cut explained once in concepts, referenced (not re-explained) in instructions |
| 4.9 | **Section orientation page present** (section-level check) | Every section (About LPT, Delegating LPT, Governance & Treasury, Guides & Resources, Technical References) has an entry page that orients the reader |
| 4.10 | **Cross-tab links at journey intersections** | At least 3 cross-tab links per Delegators page where reader graduations or context references occur (to About for protocol context, to Orchestrators where the active set is the topic, to Solutions for fiat-onramp context if applicable) |
| 4.11 | **Page answers its question COMPLETELY (Discord test)** | Can you link this ONE page on Discord and know the person gets a complete answer without needing another page? |
| 4.12 | **Page size appropriate for scope** | Concept/instruction pages: minimum 5KB substantive content. Pages under 2KB are stubs that should not be in published nav. Gold standard pages are 10–60KB |
| 4.13 | **No TODO/REVIEW comments without tracking** | Zero `{/* TODO` comments in published content. Zero `{/* REVIEW` comments without a linked tracking issue |
| 4.14 | **Flat layout where appropriate** | Delegators instruction pages use StyledSteps and prose, not sub-folders. Sub-pages only for genuinely separate journeys |
| 4.15 | **Trade-offs present** | Every concept/economics/governance page on Delegators names at least one trade-off, limitation, or risk. Reward cut increase risk. Orchestrator inactivity risk. Unbonding period lock-up. Gas cost. Active-set churn. No page presents only the upside |
| 4.16 | **Content pass context block completable** | All 15 fields in `content-pass.md` context block can be filled. If any cannot be filled, the page's position in the pipeline is undefined |
| 4.17 | **Earnings dependency present on every earnings page** (Delegators-specific) | Cross-reference 2.D3. Every claim about earnings, rewards, or yield states the dependency: orchestrator performance, reward cut, fee cut, protocol inflation rate, time horizon, active-set status, gas costs. Naked "you can earn" is a fail. The dependency must be visible in the body, not buried in a footnote or a `<Note>` |
| 4.18 | **No "set and forget" framing on operational pages** (Delegators-specific) | Cross-reference 2.D4. Pages covering ongoing delegation, governance, and earnings management must signal where ongoing engagement is required (orchestrator monitoring, reward call windows, governance proposal review). Operational pages must not present delegation as fully passive |
| 4.19 | **L1-stranded and CEX-buyer entry covered** (Delegators-specific) | The Delegators tab serves L1-Stranded and New Crypto User / CEX Buyer personas. Pages and disambiguation must address: "My LPT is on Ethereum mainnet – what do I do?" (bridge) and "I just bought LPT on a CEX – start here" (withdraw + bridge + onboard). Per `v2/delegators/_design/delegators-IA.md` persona research |

### What success looks like

A reader following the Delegators-tab journey from portal to first delegation to ongoing management never hits a wall. The L1-stranded and CEX-buyer personas have a clear path. Every earnings claim is qualified. Every operational page signals ongoing engagement.

## 5. LAYOUT, COMPONENTS & TEMPLATE

**What this catches**: Wrong components for the page type, missing required sections, unapproved components, MDX that won't render, inconsistent page structure across the same pageType. Delegators-specific: bash/shell code blocks (forbidden interface), wallet-flow steps in code blocks, missing trade-off accordions on financial pages.

### Checks

| # | Check | Pass criteria |
|---|-------|---------------|
| 5.1 | **Correct template for pageType + pageVariant** | Page uses the structural template defined for its type. See component-layout-decisions.mdx matrix |

# REVIEW EXECUTION GUIDE

## How to run a per-page review (Delegators tab)

Based on `docs-review-prompt-tiers.md` Tier 3 (Single Page):

### Pre-flight (before touching the page)
1. **Check for `generated-file-banner`** – if present, read the generation script before logging any findings
2. Read the page's frontmatter – classify against canonical enums. **Read each field value directly from the file. Do not infer `purpose` from `pageType`. Do not infer `audience` from the file path.**
3. Identify the page's audience, purpose, and pageType – from the actual field values
4. Load the relevant voice rules (delegator)
5. Load the component matrix for that pageType
6. Note the page's position in the journey – read `docs.json` to confirm the navigation sequence. **Bridge before Bond is the critical ordering**

### Review pass order
1. **Frontmatter** (Category 1) – all 10+ fields present, all values canonical, earnings-page flagging correct
2. **Structure** (Category 4) – one purpose, one job, earnings dependency present, no "set and forget", L1/CEX entry covered, Discord test, prerequisites, handoffs, trade-offs
3. **Headings** (Category 3) – score each 20/25+, per-pageType naming, per-audience register (no protocol-jargon-as-heading), domain-anchor
4. **Voice and copy** (Category 2) – banned words/phrases/constructions, entity-led, no hedging, no em-dashes, UK English, register (delegator), jargon discipline, plain-language economics, earnings dependency, no "set and forget", no CLI commands, deprecated terms, terminology
5. **Layout and components** (Category 5) – gold-standard template, no bash blocks, wallet flows in StyledSteps, data imports for contract addresses, Related Pages or Next Step CTA
6. **Veracity** (Category 6) – claims sourced per authority tiers, protocol parameters cross-checked against live contracts, treasury/inflation numbers current, REVIEW flags, glossary definitions match universal terms
7. **Links and rendering** (Category 8) – everything resolves, page renders, no TODO/TBD, Explorer/subgraph links live
8. **Navigation** (Category 7) – in docs.json, reachable, no dead ends, no stubs in nav, no TO-ADD content in published nav, Bridge before Bond ordering
9. **Completeness** (Category 10) – question coverage, zero-to-hero, persona paths (including CEX/fiat-onramp), claimEarnings warning, vote-detachment explanation
10. **Process** (Category 9) – human sign-off, decisions in registry, gates met, phases ordered, feedback routed
11. **Spelling and typo pass** – scan all visible text

### Verdict scale

| Verdict | Meaning |
|---------|---------|
| **PASS** | Ships as-is. All checks pass |
| **NEEDS WORK** | Specific fixes identified. Fix, re-check affected categories, then re-verdict |
| **REWRITE REQUIRED** | Fundamental scope, audience, or structure problem. Page brief needed before rewrite |

- [ ] Page is above 2KB minimum (5KB+ for concept/instruction)

- [ ] **Every protocol term defined on first use, in plain language** (Delegators-specific)

## REPORTING RULES

**Fixes must be executable.** No "add appropriate X". Specify the exact value, line, or action.

**Definition strings in fixes must be REVIEW placeholders.** Use `{/* REVIEW: [term] – verify definition against [source] before inserting */}`.

**Schema rules must be cited.** Any finding that asserts a schema constraint must cite the specific checks.mdx section.

**Read fields directly.** State the actual frontmatter field value – read it, do not infer it.

**Characterise the error correctly.** When a frontmatter value fails: (a) deprecated alias, (b) wrong field, (c) invalid value.

**Evaluate the current state only.** All findings evaluate the page as it currently exists.

**Result column must match the detail conclusion.**

**Veracity findings: TESTED or NOT-TESTED.** Every operational sequence (bridge, bond, claim, unbond, vote) must be labelled. NOT-TESTED requires a justification and a risk note.

**Component approval: NOT-TESTED unless the catalog was read.**

**Verify every cited occurrence.** Quote the content at the cited line.

**Link verification.** Use `operations/scripts/validators/content/structure/check-docs-path-sync.js`.

**Navigation sequence.** Check 4.3 requires reading `docs.json`. Bridge before Bond is the critical ordering for this tab.

**Self-consistency corrections are only complete when ALL instances are updated.**

**Findings must appear in all applicable sections.** No orphaned findings.

**Severity levels.** Use only: CRITICAL / HIGH / MEDIUM / INFO.

**Em-dash prohibition applies to all visible text.**

**Heading rename suggestions** must not introduce BANNED heading terms.

**veracityStatus fix must be internally consistent.**

**First use in Mermaid/diagram labels counts as first use.** Critical on Delegators tab – every term in a fee-flow or reward-distribution diagram must be defined adjacent to or before the diagram.

**Dead imports.** Do not flag.

**Deduplication.** Log each finding once in the most specific category.

**Delegators-tab specific:**
- A bash/shell code block (other than permitted on-chain values) is a CRITICAL finding under check 5.35 / 2.D8.
- An earnings claim without a stated dependency is a CRITICAL finding under check 4.17 / 2.D3.
- A "set and forget" or naked "passive income" framing is HIGH under 4.18 / 2.D4.
- Missing claimEarnings warning on claim-flow pages is HIGH under 10.8.
- A protocol term used before first-use definition is HIGH under 2.D1.
- Bond-before-Bridge ordering failure is CRITICAL under 4.3 / 7.12.

# SEVERITY DEFINITIONS

| Severity | Meaning | Action |
|----------|---------|--------|
| **CRITICAL** | Breaks downstream phase, produces structurally wrong output, or fails on common page type | Fix before any further work |
| **HIGH** | Incorrect/incomplete on specific cases but does not break downstream | Fix recommended before shipping |
| **MEDIUM** | Inconsistent but does not break downstream consumption | Monitor; fix in next pass |
| **INFO** | Expected variation or minor style preference | Document; no action required |
