## Key Finding — Section Count: 9 vs 12 (Widest Variance Yet)

| Run | Section count |
|---|---|
| Claude Web | 9 |
| ChatGPT | 12 |

This is the largest section count gap seen across all tested tabs. The 3-section difference traces to two specific structural decisions:

### 1. Claude Web S6 design flag was correct

Claude Web's "Go real-time or custom" (S6) carried a `[⚠ Design flag: may need to split]` because it accumulated both evaluation content (BYOC vs hosted trade-offs) and build-path content (PyTrickle / ComfyStream implementation). ChatGPT independently confirmed this by splitting that area into two distinct sections:
- S6: "Build a live interactive media loop" (real-time pattern selection + implementation)
- S7: "Bring your own model or container" (custom compute packaging)

The flag mechanism is working correctly — it correctly predicted that the section needed to split, and the other model's output shows what the split looks like.

### 2. Post-build is 1 section (Claude Web) vs 3 sections (ChatGPT)

| Claude Web | ChatGPT equivalent |
|---|---|
| S7: "Monitor and troubleshoot" (guide) — single section covering all post-build concerns | S10: "Verify release readiness" (instruction) |
| | S11: "Diagnose failures by layer" (guide) |
| | S12: "Tune latency, quality, and cost" (guide) |

ChatGPT's 3-section treatment correctly separates verify (pre-launch), diagnose (active failure), and tune (ongoing optimisation) — these are distinct reader states with different entry conditions. A reader with a production failure should not re-read release-readiness content. Claude Web's single section would serve all three states simultaneously, which is the same overload pattern the design flag rule was designed to catch.

**Collation implication**: Claude Web's S7 should carry a design flag — same logic as S6. Neither run flagged it explicitly, but the ChatGPT structure makes the case clearly.

### 3. ChatGPT added 2 sections with no Claude Web equivalent

- S8: "Build protocol-adjacent tools and extensions" — serving the Protocol Toolmaker persona directly (separate section)
- S9: "Connect auth, events, and internal systems" — systems integration as a distinct build phase

Claude Web handles both within existing sections: Protocol Toolmaker paths through S8 (Understand the protocol) via a routing note from S1; auth/events/callbacks are implied within the build sections. ChatGPT's explicit coverage is more thorough for these paths.

## Key Finding — Persona Granularity: ChatGPT Split Is More Precise

Claude Web treats the AI developer path as one archetype: "AI pipeline builder" (ML/Python background, wants inference at scale). ChatGPT correctly splits this into two distinct archetypes:
- **Interactive Media Builder** — latency-first, real-time product idea, needs a transport + processing pattern
- **Custom Model Porter** — has an existing model/container, needs a packaging + execution path

These are meaningfully different arriving states: one needs a real-time architecture decision, the other needs a custom compute path. The ChatGPT split is more precise and produces a cleaner section structure (S6 and S7 respectively).

Claude Web's design flag on S6 was the indirect signal that these paths needed separation. ChatGPT resolved it by producing separate personas.
