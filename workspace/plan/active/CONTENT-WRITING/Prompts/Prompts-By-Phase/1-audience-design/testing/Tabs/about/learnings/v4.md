# Audience Design Prompt — v4 Learnings
## Tab: About

**Runs**: 2 — Claude Web (run 1), ChatGPT (run 2)
**Note**: No Agent or Perplexity run for this tab. 2 runs is below the 3-run minimum for full collation confidence; findings are treated as indicative rather than definitive. Cross-tab pattern comparisons are available from Orchestrators (4 runs) and Gateways (3 runs).
**Prompt version**: v4
**Date**: 2026-03-23

## v4 Fixes on This Tab

Three v4 structural fixes were tested. About is a distinct tab — no single operator audience, no step-by-step setup path — so each fix produces different evidence than on Orchestrators or Gateways.

### Fix 1 — `pageType: navigation` on disambiguation / S1 section

**Result: 2 of 2 runs confirmed.**

Both runs independently determined that a disambiguation section is required as the first content section, and both assigned `pageType: navigation` with `purpose: orient`.

| Run | S1 title | purpose | pageType | lifecycleStage |
|---|---|---|---|---|
| Claude Web | "Find your place in the network" | `orient` | `navigation` | `discover` |
| ChatGPT | "Which role is mine?" | `orient` | `navigation` | `discover` |

Claude Web made the reasoning explicit: *"the first section had to do hard routing work that role tabs usually do not require. I kept that routing isolated in Section 1 to avoid contaminating the explanatory sections with mixed-purpose navigation."*

ChatGPT derived the same conclusion via the self-identification check: *"A dedicated disambiguation section is required as the first content section. It routes readers to the correct role tab; it does not explain or teach."*

This fix generalises to a cross-audience tab without any ambiguity. 2 of 2 runs, zero variation.

### Fix 2 — LIP-92 source verification rule

**Result: Not directly applicable to About — no LIP-92 reference in the About glossary. Both runs exercised the source verification rule on other terms.**

The About glossary (`glossary-about.md`) does not contain a definition that references LIP-92 directly. However, the source verification rule in the v4 prompt applies to any glossary definition that references a specific external artifact. Both runs found other terms requiring the flag.

**Claude Web** flagged 8 terms as `[verify-live]` or `[verify-against]` across the TERMINOLOGY_LOCK, including:
- `[verify-live: https://explorer.livepeer.org]` — Active Set size (100), governance-controlled
- `[verify-live: https://docs.livepeer.org/delegators/guides/yield-calculation]` — Dynamic Inflation rate (0.00005%) and target bonding rate (50%)
- `[verify-against: https://github.com/livepeer/LIPs — LIP-89]` — Stake-Weighted Voting
- `[verify-against: https://github.com/livepeer/wiki/blob/master/WHITEPAPER.md]` — multiple terms

Claude Web also made explicit glossary correction decisions: excluded `Inflation Model` (deprecated per deprecated-terms.md; used `Dynamic Inflation` instead), excluded `Active Set Election` (overlap with `Active Set` per deprecated-terms.md), excluded `veLPT` (not implemented), excluded `Ephemeral Compute` (definition thin). Each exclusion was documented in the Addendum with a reason.

Additionally, Claude Web flagged a `[verify-live]` value for the Unbonding period — noting that the glossary embeds the specific 7-round value but that this is governance-controlled and must not be asserted as fact. This is exactly correct source verification behaviour for About.

**ChatGPT** embedded verify flags inline in the TERMINOLOGY_LOCK header line rather than in a full table, using the format `[verify-live: Livepeer Explorer]` and `[verify-against: Whitepaper / protocol contracts]`. Coverage was present but less granular — no per-term table, no explicit documentation of exclusion decisions.

**Glossary conflict identified by Claude Web only**: The About glossary marks `Broadcaster (deprecated)` with `Status: current` — this directly contradicts `deprecated-terms.md` which explicitly states "Never use 'Broadcaster' in new documentation." Claude Web flagged this in the Addendum. ChatGPT did not mention it but correctly used `Gateway` throughout, never using `Broadcaster`.

**Summary**:

| Run | Verify flags used | High-staleness values flagged | Glossary conflict noted | Deprecated term exclusions documented |
|---|---|---|---|---|
| Claude Web | Yes — full table, 8 terms | Active Set size, inflation rate, target bonding rate, unbonding period | Broadcaster status: current vs deprecated-terms.md | Yes — Inflation Model, Active Set Election, veLPT, Ephemeral Compute |
| ChatGPT | Yes — inline format, ~8 terms | Active Set, round, dynamic inflation | No explicit mention | Not documented explicitly |

The source verification rule is working on the About tab. No LIP-92 trigger was available to test, but the broader rule behaviour (verify-live for governance-controlled values, verify-against for whitepaper/LIP-sourced claims) fired correctly in both runs.

### Fix 3 — Section load flag in table row

**Result: Not triggered in either run.**

Neither run produced a section that accumulated `purpose: evaluate` and routing/disambiguation work simultaneously. The About disambiguation section (S1 in both runs) was cleanly scoped as `orient / navigation` with no evaluation work attached to it. The conceptual sections that followed were each assigned single-purpose roles (`explain`, `learn`, `update`, `verify`, `choose`).

This matches the pattern from Orchestrators v4 (0 of 4 runs triggered) and Gateways v4 (1 of 3 triggered only when Perplexity overloaded a late section). Clean section design on About — where the tab is explicitly a substrate-and-routing layer — produced no conditions requiring the flag.

The fix is present and functional. It did not fire because the sections were designed correctly.

## Key Findings

### Finding 1 — Disambiguation consensus: first section, `navigation`, no teaching content

Both runs independently concluded that About requires a disambiguation section as Section 1 with `pageType: navigation` and no teaching content embedded in it. This is unanimous despite the runs using different titles and slightly different framings.

Claude Web framed it as a routing decision between three conceptual paths (platform purpose, participant roles, token/governance). ChatGPT framed it as a four-way role-path disambiguation (Orchestrator, Gateway, Delegator, Developer/Builder). Both agree on the structural principle: the reader cannot identify their next step before understanding enough to distinguish the paths, and this disambiguation must happen in a dedicated navigation section rather than within a concept page.

This is the strongest structural consensus of the About v4 runs.

---

### Finding 2 — Audience token disagreement: `community` vs `all at discover`

The most significant inter-run divergence is the audience token assignment.

**Claude Web** derived `all` at `discover`, citing the Audience Definitions table note: *"About serves all audiences at the discover lifecycle stage. A single audience token cannot fully describe About's primary audience."* Claude Web treated the table note as a direct instruction, concluded that no canonical single token applies, and stated this explicitly as the AUDIENCE derivation.

**ChatGPT** assigned `community` as the audience token, acknowledging in the Addendum: *"The brief requires a single AUDIENCE token, but the brief itself also states that About cannot be cleanly represented by one token. I used `community` as the broadest available token, but treated the tab as substantively multi-audience throughout."*

Neither answer is wrong — both runs recognised the multi-audience reality and handled it structurally. The prompt note says "a single audience token cannot fully describe About's primary audience" without specifying a resolution path. Both runs navigated the ambiguity differently but both produced structurally correct multi-audience section designs.

This is not a quality failure in either run. It is a prompt ambiguity specific to About, which is the only tab in the site with this property.

---

### Finding 3 — Section count divergence: 8 (Claude Web) vs 10 (ChatGPT)

| Run | Section count | In range (8–12)? | Notable structural difference |
|---|---|---|---|
| Claude Web | 8 | Yes — low end | Lean design; About as substrate-and-routing layer; depth owned by role tabs |
| ChatGPT | 10 | Yes | Added §9 (verify live) and §10 (where to go next) as explicit standalone sections |

Claude Web justified 8 explicitly: *"About is intentionally lean on section count because it is a substrate tab — it must orient and route, not become a second comprehensive reference. The role tabs carry depth; About carries conceptual clarity."*

ChatGPT's additional sections — "What must be verified live" and "Where to go next" — are defensible additions. The verify-live section addresses a genuine reader need (knowing which values require live verification before acting) and is consistent with the tab's `discover`→`evaluate` journey. The "Where to go next" section gives the final routing handoff explicit treatment. Both are within the section count target range.

The 2-section difference is structurally grounded, not noise.

### Finding 4 — Terminology lock depth: Claude Web significantly more thorough

Claude Web produced a 21-term TERMINOLOGY_LOCK in full table format with per-term definitions, verify flags, and a written justification for exceeding 20 terms. ChatGPT produced an inline comma-separated list of approximately 22 terms with embedded verify flags but no per-term definitions and no justification for exclusions.

For About specifically — which covers the full conceptual substrate including two compute types, four actor roles, tokenomics, governance, and payment mechanics — a thorough TERMINOLOGY_LOCK is more important than on a single-role tab. About cannot anchor correctly if its terminology base is under-specified. Claude Web's table format with explicit exclusion reasoning is the higher-quality approach for this tab type.

This is not an error in ChatGPT's run; the inline format satisfies the brief's minimum requirement. But the evidence suggests that About, as the broadest-scope tab, benefits disproportionately from the extended table format.

---

### Finding 5 — Both runs correctly identified About's unique audience structure

Neither run tried to force About into a single-role persona model. Both produced multi-path persona sets that reflect About's cross-audience nature:

- Claude Web: 5 personas including Role Seeker (primary), Economics Researcher, Protocol Architect, Governance Participant, Curious Observer
- ChatGPT: 4 personas including Fit-check evaluator (primary), System-model seeker, Token and governance checker, Protocol/history researcher

The primary persona in both cases is the reader who has not yet identified their role — the "fit check" or "role seeking" persona. Both runs correctly placed this persona as primary and made the disambiguation section the structural priority.

This is correct prompt execution for a cross-audience tab with no single canonical role.

## Issues Found

### MEDIUM — Audience token resolution not specified for About

**Issue**: The prompt's Audience Definitions table includes an explicit note that "a single audience token cannot fully describe About's primary audience" but does not specify what to do with this — force-select the nearest single token, use a composite, or state `all`. The two runs chose differently (`community` vs `all at discover`).

**Impact**: For the collation phase, this ambiguity means the AUDIENCE field in the About output document will need a human-reviewed resolution before it can be treated as canonical. No downstream structural harm in either run — both handled the multi-audience design correctly in practice.

**Recommended fix**: Add a resolution instruction to the About-specific brief context, or add a note to the Audience Definitions table specifying what a model should output when the note applies: e.g. "For About only: output `AUDIENCE: all (discover)` and treat as substantively multi-audience throughout."

**Priority**: MEDIUM — affects About runs only; does not affect other tabs.

---

### LOWER — ChatGPT Broadcaster glossary conflict: not flagged

**Issue**: The About glossary lists `Broadcaster (deprecated)` with `Status: current`. This conflicts with `deprecated-terms.md`, which explicitly states "Never use 'Broadcaster' in new documentation." Claude Web caught and documented this conflict. ChatGPT did not flag it.

**Impact**: ChatGPT correctly excluded Broadcaster from the TERMINOLOGY_LOCK and used `Gateway` throughout, so no terminology contamination occurred. The issue is that the conflict itself — the glossary's incorrect `Status: current` label — was not surfaced as a finding that would prompt a glossary correction.

**Recommended fix**: No prompt change needed given that 1 of 2 runs caught it and neither run was contaminated by the error. Monitor on next About run or any tab whose glossary has deprecated terms incorrectly marked as current. The source verification rule is working — it caught the conflict in the run with web access and richer verification output.

**Priority**: LOWER — no structural harm; surfaced for glossary maintenance.

---

### LOWER — TERMINOLOGY_LOCK format inconsistency between runs

**Issue**: Claude Web produced a 21-term table with per-term definitions and explicit verify/exclude reasoning. ChatGPT produced an inline comma-separated list with embedded flags but no definitions or exclusion documentation. Both technically satisfy the brief's output requirement (the prompt specifies inline format: `[term1, term2, ...]`), but the table format produces substantially more auditable output for About's broad scope.

**Impact**: For About specifically, the audit trail matters more than on single-role tabs because the terminology base must cover five distinct content pillars. ChatGPT's inline format makes it harder to verify that every high-staleness value was correctly flagged.

**Recommended fix**: No prompt change needed — the prompt's format specification is intentionally minimal to avoid over-constraining models. The difference in output quality is a model capability difference, not a prompt failure. Note for collation: use Claude Web's table as the primary TERMINOLOGY_LOCK reference when the two runs are combined.

**Priority**: LOWER — collation process handles this.

## Run Quality Notes

### Claude Web (run 1)

**Strengths**:
- Most complete TERMINOLOGY_LOCK in the About v4 set: 21 terms, full definition table, explicit exclusion reasoning for each term removed from consideration (Inflation Model, Active Set Election, veLPT, Ephemeral Compute).
- Correctly identified and documented the governance-controlled value problem: active set size, inflation rate, target bonding rate, and unbonding period all flagged `[verify-live]` with specific source URLs.
- Caught the Broadcaster `Status: current` conflict in the glossary and documented it in the Addendum.
- Best audience derivation reasoning: cited the Audience Definitions table note directly and derived `all at discover` with explicit justification.
- Strongest persona path validation: included explicit structural block and knowledge gap analysis for all 5 personas.
- Section design reasoning is the most thorough: every section has both an entry state and exit state that describe concrete decisions or completed actions, not comprehension.
- Addendum flags are substantive and actionable: each flag names the phase, step, issue, and resolution assumption.

**Weaknesses**:
- 8 sections is at the lower end of the target range. While justified by the substrate-tab argument, the runs with 10 sections (ChatGPT) identified genuine reader needs — the verify-live section and the explicit routing handoff — that the 8-section design handles implicitly rather than explicitly. A collated design may warrant 9–10 sections.
- The extended TERMINOLOGY_LOCK format (21 terms, full table) is more verbose than the prompt's minimum specification. This is a strength for About but would be excessive on narrower tabs.

---

### ChatGPT (run 2)

**Strengths**:
- Clean section design: 10 sections all within the target range; no overloaded or ambiguous sections.
- Added two sections absent from Claude Web's design — "What must be verified live" (`verify / reference`) and "Where to go next" (`choose / navigation`) — that serve genuine reader needs and are structurally justified.
- Persona set is well-differentiated: 4 personas with distinct entry vectors and motivations; no persona overlaps.
- Checkpoint self-assessment is complete: all 13 items checked, with specific evidence cited for each.
- Correctly used `Gateway` throughout without ever using `Broadcaster`.
- "Update" purpose used correctly in §8 (protocol history section): purpose `update`, pageType `guide` — correct application of the enum.

**Weaknesses**:
- Audience token (`community`) does not fully capture About's multi-audience nature; acknowledged in Addendum but the tension was not resolved.
- Inline TERMINOLOGY_LOCK format without per-term definitions makes it harder to audit which values were flagged and why. Exclusion reasoning not documented.
- Did not flag the Broadcaster `Status: current` glossary conflict.
- Shorter Addendum than Claude Web — flags are present but less specific about which assumptions were made to resolve which ambiguities.
