## Issues Found

### CRITICAL — Fix before next run

**1. Checkpoint wording still contradicts the self-containment principle**

Flagged by: v2-orchestrators-audience-design.md (addendum), cross-run pattern

The cross-tab gate instruction was rewritten in v2 to allow intentional duplication when the audience needs the content. However, the Checkpoint section still reads:

> "Cross-tab gate applied — no content duplicated from other tabs?"

"No content duplicated" is the opposite of self-containment. The gate instruction and the checkpoint item now contradict each other. Any model that applies the checkpoint literally will flag its own correct work as a failure.

**Fix**: Rewrite the checkpoint item to match the rewritten gate: *"Cross-tab gate applied — no unnecessary duplication; decision-enabling content included for this audience; CROSS-TAB rows only where audience does not need the content at all?"*

---

**2. LIP-92 numbering collision in source documents**

Flagged by: Claude web UI (Phase 0 discrepancy flag, confirmed in addendum)

The deprecated-terms file assigns LIP-92 = treasury reward cut rate. The per-tab glossary assigns LIP-92 = AI model registry and capability discovery mechanism. These are likely two distinct LIPs assigned the same number in two different source documents. The Claude web UI run correctly flagged this as unresolved and did not assert either meaning in precision-dependent content.

This is a source quality issue, not a prompt issue — but the prompt must tell models what to do when two input files contradict each other on a specific claim.

**Fix**: Add to Phase 0 Step 0.2: *"If the glossary and veracity-sources.md or deprecated-terms.md assign contradictory meanings to the same term or identifier, do not assert either meaning in any section that depends on it being accurate. Flag the conflict in the Addendum as: `[CONFLICT: {term} — {source A says X}, {source B says Y} — requires resolution against {authoritative source}]`."*

---

### HIGH — Fix in next iteration

**3. Section 2 consistently overloaded across all four runs**

Flagged by: Claude web UI addendum, Claude Sonnet Phase 2D analysis, Perplexity path validation — three of four runs raised this

Every run assigned Section 2 ("Is this right for me?" / "Entry paths and prerequisites" / "Qualifying themselves") the work of: hardware qualification, LPT/ETH requirement, workload path routing (video vs AI vs dual mode vs pool), active-set-vs-AI-routing disambiguation, and pool node path explanation.

All three runs that flagged it noted that Section 2 "may need to split into two pages during detailed design." This is a consistent signal that the brief is not giving enough guidance on when to split a disambiguation/routing purpose from an evaluation/qualification purpose.

**Fix**: Add to Phase 2C instructions: *"If the section serving purpose `evaluate` is also the section that must handle self-identification routing (i.e. the reader does not yet know which path applies to them), call this out explicitly in the exit state. A single section carrying both routing and evaluation is a design flag: note in the Addendum that it may warrant splitting into two pages at detailed design. Do not add the second page here — flag it."*

---

**4. AI-First Operator persona produced inconsistently**

Appears in: Claude web UI (distinct Persona 4, Total=7), not in Claude Sonnet (AI path folded into Persona 3), not in Perplexity (AI work described as journey stage, not distinct persona), not in v2-orchestrators-audience-design.md (AI path is a section, not a persona)

For a tab with two substantively different workload types, the scoring rubric does not reliably produce a distinct AI workload persona. When AI is folded into an existing persona as a journey stage, the section structure does not give it a standalone setup path — which means AI-first operators (those who arrive specifically for AI inference and have no interest in video) get buried inside a section designed for someone adding AI to an existing video node.

**Fix**: Add a note to Phase 1 instructions: *"If the tab covers more than one workload type with meaningfully different hardware requirements, setup paths, or economic models, treat the audience for each workload type as a candidate persona. A workload-specific persona should be scored separately even if volume is lower — Impact and Depth scores can justify inclusion at lower volume."*

---

**5. Self-identification check produces inconsistent disambiguation decisions**

Three of four runs: "Disambiguation required — routing page needed as first section"
One run (Perplexity): "No disambiguation page required — label maps cleanly; routing callout in Section 2 handles it"

These are structurally different outputs: one produces a dedicated routing section, the other folds routing into Section 2. For this tab, both may be defensible, but the prompt produces both from the same input without guidance on which to choose.

**Fix**: Add to Phase 2C self-identification instructions: *"A disambiguation page (dedicated `guide` section with purpose `choose`) is required when: (a) the audience label maps to 3+ meaningfully different setup paths, and (b) the reader cannot identify their path from the section list alone without reading content. A routing callout within Section 2 is sufficient when there are 2 paths with a single binary decision point (e.g. sufficient stake / not sufficient stake). If unsure, err toward the dedicated disambiguation section."*

### LOWER — Consider in future iteration

**6. Section count variance is very wide**

Perplexity: 8 sections. Claude Sonnet: 14 sections. Claude web UI: 15 sections. v2-orchestrators-audience-design.md: 8 sections.

A difference of 8 vs 15 sections for the same tab and audience is a structural divergence, not a content depth difference. Both 8-section outputs cover all 7 jobs-to-be-done; the 14–15 section outputs split jobs across finer-grained sections. The brief has no guidance on expected section count. This matters when multiple people are running the prompt: the IA designs produced will be incompatible.

**Fix**: Add a recommended range to Phase 2C: *"Aim for 8–12 sections for a full-audience tab. Below 7 likely means multiple jobs are under-served in a single section; above 14 likely means over-splitting into sections that should be combined. Note in the Addendum if you are outside this range and why."*

---

**7. Pool node still gets a full standalone section in some runs**

In Perplexity: Section 12 is a full `instruction`-type standalone section for pool node participation.
In Claude web UI: Section 6 is a full `guide` section for pool joining.
In Claude Sonnet: pool node is a content note within Phase 2D only, not a standalone section.

The scoring rubric reduced the pool node persona's priority, but did not prevent it from generating a standalone section. A Depth=1 or Depth=2 persona can still produce a section if the section serves another persona path too. The content feasibility problem (pool nodes have ~2 paragraphs of real content) will not be solved at the section structure level — it needs a separate post-audience-design check.

**Note**: This is a known out-of-scope issue. Flagged for the future "content feasibility" pipeline step (post-Phase 1, pre-Phase 3), not the audience design prompt.

## Validation Note — Orchestrators Tab Limits

All v2 runs are on the Orchestrators tab. The prompt was contaminated in the early v2 editing pass with Orchestrators-specific examples. Even though those examples were removed from the brief, models with training knowledge of Livepeer will produce better-structured Orchestrators outputs than they would for a less-documented tab. **These results cannot confirm that the generic prompt works across tabs.** The prompt must be run on a different tab (Gateways, Delegators, or Developers) before any conclusion about generic prompt quality can be drawn.

## Section Structure Observations (content quality, not prompt issues)

- **Arriving question convergence**: All four produced an arriving question centred on hardware + earning + "what does it take". This is stronger than v1, which had more variation.
- **Reward Call got its own section in two of four runs** — both Claude runs. The other two buried it in monitoring or operations. Given that missing a Reward Call permanently forfeits that round's allocation, this may warrant a recommendation in the section structure guidance.
- **On-demand categories consistent across all four** — CLI flags, active set size, pricing benchmarks, performance score formula appeared in all four. Reward Call timing and pool directory appeared in three. Governance/cut norms appeared in two. Good convergence.
- **Cross-tab directions consistent** — all four identified the same inbound vectors (Home, About, Community) and outbound routes (Delegators, Gateways, Resource HUB). No outliers.
