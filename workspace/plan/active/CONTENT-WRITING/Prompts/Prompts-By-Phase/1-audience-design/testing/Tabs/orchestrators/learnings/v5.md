## Key Findings

### Finding 1 — Section count: 10 (v5 Agent) vs 12 (v4 collation canonical)

The v5 Agent run produced 10 sections (S1–S10). The canonical v4 collation output settled on 12 sections (S1–S12), with S6 (pool node section) and S8 (AI pipeline setup) as the two additions that drove the count above 10.

| Structural difference | v5 Agent | v4 Canonical |
|---|---|---|
| Dedicated pool node section | No — pool node path folded into S1 disambiguation and S4 configuration | Yes — S6 dedicated section |
| Dedicated AI pipeline section | Yes — S6 in v5 | Yes — S8 in v4 |
| Dedicated pricing and cuts section | Merged into S4 (configure for production) | Yes — S7 dedicated |
| Total sections | 10 | 12 |

The v5 Agent's merging of pool node path into the disambiguation section (rather than a dedicated evaluate+guide section) conflicts with the canonical v4 decision. In v4 collation, pool node was given a dedicated section because the pool node candidate has a distinct decision gate that cannot be served within a setup section designed for solo operators. This is a structural difference that must be reconciled before the v5 output is used as Phase 2 input.

**Verdict**: The canonical v4 collation output (12 sections) takes precedence. The v5 single-run output is an input to, not a replacement of, that canonical decision.
