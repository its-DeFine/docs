## v3 Fixes That Held

All three v4 changes confirmed working:

- **`pageType: navigation` on disambiguation section** — 4 of 4 runs. S1 has `purpose: orient` + `pageType: navigation` in every run without variation. The inconsistency (guide / concept / navigation across v3 runs) is gone.
- **LIP-92 source verification rule** — 3 of 4 runs flagged the glossary's LIP-92 claim as `[verify-against: https://github.com/livepeer/LIPs]`. ChatGPT went further and identified the actual content of LIP-92 via web search (see below). Perplexity missed it (see issue #1). Rule is working for models with web access.
- **Section load flag in table row** — did not trigger in any run. Reason: the cleaner S1 disambiguation structure (now unambiguously `navigation`) meant no section accumulated `evaluate` + routing work simultaneously. Sections that previously overloaded S2 were correctly separated. The fix worked indirectly by removing the condition that caused the flag in v3.

## Key Finding — LIP-92 Test: Best Result Yet

This is the first version across all testing where a model correctly identified what LIP-92 actually covers.

**ChatGPT (run 3)** — addendum: *"The per-tab glossary contains at least one direct source conflict. It assigns LIP-92 to AI capability discovery, but the current LIPs index shows LIP-92 = Treasury Contribution Percentage. I treated the glossary claim as incorrect and excluded that specific mapping from the lock."*

This is the target behaviour: model used web access, went to the LIPs repo, identified the actual LIP, found the conflict, marked the glossary as wrong, and excluded the claim. Full pass.

**Agent (run 1)** and **Claude Web (run 2)** — both correctly marked LIP-92 as `[verify-against: https://github.com/livepeer/LIPs]` without accepting the glossary's claim. Correct but partial: they flagged for verification rather than verifying.

**Perplexity (run 4)** — did not engage with LIP-92 specifically despite having web access. Missed the connection between the glossary's Capability Advertisement definition and the LIP-92 reference. However, Perplexity caught a *different* valid glossary error: the go-livepeer glossary entry incorrectly lists "Broadcaster" as a role — current term is "Gateway" per deprecated-terms.md. That catch was not made by any other run.

**Summary**:

| Run | LIP-92 handling | Other glossary catches |
|---|---|---|
| Agent | `[verify-against]` — flagged, not verified | "Siphon" term internally inconsistent in glossary |
| Claude Web | `[verify-against]` — flagged, not verified | None |
| ChatGPT | ✅ Identified correct LIP-92 content (Treasury Contribution %) | None |
| Perplexity | Missed LIP-92 | go-livepeer glossary entry lists deprecated "Broadcaster" role |

The source conflict rule is working for 3 of 4 models. The Perplexity miss is likely model-specific pattern-matching (it found other glossary issues but didn't connect Capability Advertisement → LIP-92 → verify against LIPs). No prompt change indicated — if this recurs across a non-Orchestrators tab run, revisit.

## Issues Found

### LOWER — Note for next tab run

**1. Perplexity missed LIP-92**

Perplexity has web access and applied the source verification rule elsewhere (deprecated term catch for Broadcaster), but did not specifically engage with the LIP-92 claim in the Capability Advertisement definition. The rule says: *"for any glossary definition that references a specific external artifact — a LIP number... mark it [verify-against]."* Perplexity did not apply this to the Capability Advertisement entry which references LIP-92.

No prompt change needed given 3 of 4 models caught it. Monitor on the next tab.

---

**2. ChatGPT generated a reference section — correct behaviour**

ChatGPT added Section 11 "Operator reference" — `pageType: reference`. ChatGPT noted in its addendum that it kept the section because "the orchestrator audience clearly needs a dense quick-reference surface." This is correct: on high-density tabs with strong reference lookup jobs (parameter names, flag defaults, reward mechanics), identifying a reference section is good prompt execution, not an instruction violation. The reference section serves a distinct reader state (return-visit lookup) that cannot be served by a how-to or concept page.

This is now confirmed as correct and expected behaviour. No prompt change needed.

---

### LOWER — Structural variance (not a problem)

**3. Pool node path handled differently across runs**

- Agent: Pool node handled as a variant sub-path within S4 (setup)
- Claude Web: Pool node flagged as needing a routing callout within S5 (install)
- ChatGPT: No dedicated section — routed via disambiguation + evaluation sections
- Perplexity: **Dedicated S9 "Joining as a pool node"** — `evaluate | guide`, correctly scoped (decision + architecture, not self-service setup)

Perplexity's dedicated pool node section is the strongest design decision — it correctly identifies that pool participation can only be resolved to an evaluate/guide section (the setup requires finding a pool operator, which can't be fully documented in a self-service instruction page). No prompt change needed; this variance is defensible and the scoring rubric handles it (Depth=2 for pool path → moderate section warrant).
