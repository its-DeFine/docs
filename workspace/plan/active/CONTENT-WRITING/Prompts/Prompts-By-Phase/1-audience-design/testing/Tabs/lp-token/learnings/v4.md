## Key Finding — Section Count and Structural Divergence

Both runs are within the 8–12 target (Claude Web: 11, ChatGPT: 10), but they diverge significantly on two structural decisions:

### 1. L1 bridging — dedicated section vs on-demand item

| Run | Approach |
|---|---|
| Claude Web | Dedicated Section 3: "Getting LPT onto Arbitrum" — `build` / `instruction`, `setup` stage |
| ChatGPT | No dedicated bridging section — bridging path appears in on-demand categories only |

Claude Web's rationale: The L1 Stranded persona (LPT on Ethereum mainnet) has a hard-stop blocker — they cannot bond until LPT is on Arbitrum. A dedicated section resolves this blocker with a concrete sequence. Without it, the persona's path is blocked mid-journey.

ChatGPT's rationale (implicit): The bridging action is a one-time prerequisite rather than a distinct phase — treated as a procedural on-demand item rather than a first-class section.

**Collation decision will need to resolve**: does the L1 path warrant a dedicated section (Claude Web), or is it a routed sub-path covered by on-demand content (ChatGPT)? The deciding factor is whether the L1 Stranded persona (consensus 1 — Claude Web only) meets the single-run persona exception threshold. It scores Total=7 with Impact=3 in Claude Web's run, suggesting strong inclusion case despite being single-run.

### 2. Governance depth — 3 sections vs 1 section

| Run | Governance coverage |
|---|---|
| Claude Web | S9: Governance participation (voting + vote detachment) · S10: Submitting a governance proposal · S11: Treasury and ecosystem funding — 3 sections |
| ChatGPT | S9: How Do I Use Stake in Governance and Treasury Decisions? — 1 merged section |

This is the largest structural gap between the two runs. Claude Web treats governance participation, proposal submission, and treasury context as distinct reader jobs with distinct entry/exit states. ChatGPT treats them as one governance section.

The collation will have difficulty resolving this cleanly — with only 2 runs, there is no majority consensus. The deciding factor should be whether proposal submission and treasury context each require a distinct entry state and distinct exit action that cannot be served within a merged section.

## Key Finding — Arriving Question Divergence

| Run | Arriving question |
|---|---|
| Claude Web | "My LPT is sitting idle — how do I stake it to earn rewards, and is there a catch?" |
| ChatGPT | "I have LPT, or I am about to get it — what is the right participation path for me right now, and what is the next safe action?" |

Claude Web's question is yield-first — it frames the tab from the perspective of an idle-token holder wanting passive income. The "is there a catch?" framing surfaces the thawing period risk upfront.

ChatGPT's question is path-first — it frames the tab around participation routing regardless of whether they are a first-time staker, a reallocator, or a governance participant.

The ChatGPT framing better accommodates the full persona set (including governance-only and evaluators). Claude Web's framing better serves the primary persona (Yield Seeker) but underserves the other paths. Collation should synthesise toward the path-first framing while preserving the yield-first motivation that drives the majority of traffic.

## Issues Found

### LOWER — ChatGPT S1 `lifecycleStage: evaluate` instead of `discover` (recurring)

ChatGPT's Section 1 has `lifecycleStage: evaluate`. The correct value for a disambiguation section is `discover` — the reader has not yet made any decision; they are identifying which path applies to them. This is the same error that appeared in ChatGPT's Orchestrators run and was flagged there.

This is now a confirmed recurring pattern: ChatGPT consistently assigns `evaluate` to the disambiguation section. The collation prompt handles this by overriding to `discover`, but the audience design prompt should be clearer. Consider adding a note to Phase 2C: *"The disambiguation section (S1) must have `lifecycleStage: discover` — the reader is identifying their path, not yet evaluating options."*
