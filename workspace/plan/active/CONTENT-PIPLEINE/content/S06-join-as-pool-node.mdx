<Tip>
  Pool participation is the lowest-barrier entry point for GPU owners who want to earn from the Livepeer network without managing on-chain operations, staking LPT, or running protocol infrastructure.
</Tip>

A pool is a single registered orchestrator node backed by multiple GPU contributors. The pool operator holds stake, handles reward calling, sets prices, and routes jobs. Pool nodes provide GPU compute and receive off-chain payouts for completed work.

<Note>
  Pools are community-run. Each pool operates independently with its own terms. Vet any pool before connecting hardware.
</Note>

**Key insight:** AI inference routing is capability-based, not stake-rank-based. Active set membership is not required for AI jobs. An operator with strong GPU hardware but limited LPT earns AI fees as a solo node without needing pool membership. The pool path is most relevant when the operator wants video transcoding income without acquiring LPT, or prefers a hands-off operational model.

## Pool architecture

From the network's perspective, there is one orchestrator with one stake offering pooled capacity. From a pool node's perspective, the GPU is one contributor behind that orchestrator.

The pool operator handles:
- Holding and managing stake
- Setting prices and advertising capabilities
- Receiving work from gateways
- Routing segments to pool node GPUs for processing
- Distributing payouts to pool members

A pool node:
- Runs a transcoding worker process (go-livepeer in transcoder mode, or the pool's own client)
- Processes segments routed to the GPU
- Receives off-chain payouts based on completed work

<Warning>
  There are no on-chain records of individual GPU contributions. Pool reputation and rewards depend entirely on the pool operator's performance and accounting.
</Warning>

### Due diligence checklist

Before connecting hardware, confirm:

<AccordionGroup>
  <Accordion title="Payout terms" icon="calculator">
    How are earnings calculated? Common models include per-segment payment, proportional share of total pool work, or pool-token distribution. Confirm the payout asset (ETH, USDC, or other), frequency (weekly, monthly), and minimum threshold.
  </Accordion>
  <Accordion title="Worker requirements" icon="list-check">
    Legitimate pools provide clear technical requirements: GPU type, minimum VRAM, operating system, and network bandwidth. A legitimate pool never asks for a keystore file or private key. **A request for a keystore or private key is a scam.**
  </Accordion>
  <Accordion title="Supported workloads" icon="microchip">
    Confirm whether the pool handles video transcoding, AI inference, or both. As of early 2026, publicly documented pools primarily handle video transcoding. Check Discord for AI-focused pools.
  </Accordion>
  <Accordion title="Pool maintenance" icon="clock-rotate-left">
    Check when the pool's GitHub repository was last updated. Check Discord presence and community activity. A pool with stale tooling often has weak gateway demand.
  </Accordion>
</AccordionGroup>

## Connecting your GPU

Pool operators each have their own worker client setup. Two common connection models:

### Docker worker (recommended)

Most pools provide a Docker image:

```bash icon="terminal" filename="docker-pool-worker"
# Clone the pool's repository (example: Titan Node)
git clone https://github.com/Titan-Node/Titan-Node-Pool.git
cd Titan-Node-Pool/docker
docker compose up -d
```

The container connects to the pool operator's orchestrator, receives video segments, processes them with the local GPU, and reports back completed work.

### go-livepeer transcoder mode

Run go-livepeer directly in transcoder mode pointed at the pool operator's orchestrator address:

```bash icon="terminal" filename="transcoder-mode"
livepeer \
    -transcoder \
    -orchAddr <pool-orchestrator-address>:8935 \
    -nvidia 0 \
    -maxSessions 10
```

<Warning>
  Pass `-transcoder` only when joining a pool. That keeps the machine in worker mode and leaves on-chain registration with the pool operator.
</Warning>

<CustomDivider middleText="Verify and Track" />

## Verify work is arriving

Once connected, transcoding activity should appear in worker logs within a few minutes during active routing windows:

```text icon="terminal" filename="expected-worker-log"
Received transcode request  segment=0 ...
Transcoded segment          segment=0 ...
```

No activity after 10-15 minutes: verify the pool orchestrator endpoint address, confirm the GPU is visible (`nvidia-smi`), and ask in the pool's Discord channel. Quiet periods occur when network demand is low.

## Track earnings

Pool payouts are off-chain. Each pool uses its own tracking mechanism -- dashboards, Discord bots, or periodic statements. Keep a record of expected versus actual payouts.

## Frequently asked questions

<AccordionGroup>
  <Accordion title="Do pools share LPT inflation rewards with workers?" icon="coins">
    Transcoding fees are paid in ETH. LPT inflation rewards accrue to the staking orchestrator. Some pools share a portion of LPT rewards with workers -- ask the pool operator about their policy.
  </Accordion>
  <Accordion title="Are AI inference pools available?" icon="microchip">
    AI inference requires capability registration tied to the orchestrator node. As of early 2026, publicly documented pools primarily handle video transcoding. Running AI inference and earning from it currently requires a solo orchestrator path, which does not require active-set-level LPT stake.
  </Accordion>
  <Accordion title="Multiple GPUs in a pool?" icon="server">
    Pass multiple GPU IDs when starting the worker (e.g. `-nvidia 0,1,2`). The pool orchestrator sees the worker as a single endpoint with higher capacity.
  </Accordion>
</AccordionGroup>
