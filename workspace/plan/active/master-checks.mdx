## Per-Page Review Requirements

### What review checks are needed, by page type

The canonical taxonomy defines 7 page types. Review requirements split into two tiers: **universal checks** (apply to every page regardless of type) and **type-specific checks** (vary by pageType). Below is the complete breakdown.

---

### Universal Checks (all 7 page types)

These 9 categories apply regardless of `pageType`. Source: `v2/orchestrators/_workspace/canonical/checks.mdx`.

| # | Category | What it checks | Pass/fail criteria |
|---|---|---|---|
| 1 | Frontmatter & taxonomy | All 10 required fields present; canonical 7-type pageType; canonical 15-value purpose; canonical 7-token audience; veracityStatus honest | Zero missing fields; zero deprecated enum values; veracityStatus = verified only if all REVIEW flags cleared |
| 2 | Voice & copy | Banned words absent; banned phrases absent; banned constructions absent; opening order correct; paragraph discipline; audience register; UK English | Zero violations from the banned list; every value claim concrete and quantified; no US spellings |
| 3 | Section naming & headings | Every heading scores ≥20/25 on the 5-dimension naming rubric; no generic structure words as headings | All H2/H3 score ≥20/25; no standalone "Overview", "Types", "Details", "Getting Started" as headings |
| 4 | Page structure & content architecture | One purpose, one audience, one job; adjacency check (prev/next page); no dead ends; prerequisites stated; no content duplication | Purpose statable in one sentence: "This page lets the [AUDIENCE] [concrete action]" |
| 5 | Layout, components & template | Correct template for pageType + pageVariant; required sections present; only approved components used; no TODO/TBD in published content | Zero render errors; zero unapproved components; correct template applied |
| 6 | Veracity & factual accuracy | Every factual claim citable; code/commands tested; no deprecated APIs; numbers are real; REVIEW flags for unverified claims; veracityStatus honest | ≥90% auto-clearance rate against 45-source veracity library; zero open-ended research tasks |
| 7 | Navigation & information architecture | Page in docs.json; navigation matches file system; no structural orphans; audience journey complete | Zero orphaned pages; zero phantom nav entries |
| 8 | Links & rendering | All internal links resolve; all external links live; all snippet imports resolve; all images load; page renders without error | Zero 404s; zero render errors; zero broken imports |
| 9 | Process & governance | Human sign-off recorded; consuming decisions in registry; gate prerequisites met; phase ordering respected | No page ships on agent self-assessment; every correction routes through feedback-routing-map |

**Data/context needed for universal checks:**
- `docs.json` — for navigation and structural claims
- `voice-rules.md` — for per-audience voice rules
- `workspace/plan/active/CONTENT-WRITING/decisions/decision-registry.md` — for locked structural decisions
- `workspace/plan/active/CONTENT-WRITING/decisions/tab-status.md` — for gate status
- `workspace/plan/active/CONTENT-WRITING/Frameworks/veracity-library.md` — 45-source library for factual claims
- Phase 1 audience design output for the tab (for persona fit check)
- Tab map at `v2/[tab]/_workspace/tab-map.md` (for adjacency check)

### Type-Specific Checks

Beyond the 9 universal categories, each page type carries specific structural requirements:

#### `navigation` — pure routing pages

- **Primary information type**: structural
- **Required sections**: routing cards to all sections within the tab; self-identification prompt if multiple paths exist
- **Specific check**: D-NAV-01 compliance — single shared navigation file per tab for multi-path tabs (locked decision)
- **Specific check**: No prose orientation paragraph — orientation is implicit in card value propositions
- **Specific check**: No instructional content — routing only
- **Component requirement**: Card or CardGroup — no content components
- **Data needed**: Tab's arriving question (from Phase 1 output); D-NAV-01 locked decision

#### `concept` — explains how something works

- **Primary information type**: conceptual, narrative, analytical
- **Permitted secondary types**: factual
- **Required sections**: Opening orientation (what mechanism this explains); core mechanism; implications or outcomes
- **Specific check**: No step-by-step procedures — those belong in `instruction`
- **Specific check**: Exit state must be "reader can describe or reason about X" — not "reader can do X"
- **Specific check**: Diagrams and prose together — code blocks only if explaining a mechanism, not demonstrating a procedure
- **Data needed**: What mental model the reader needs; what conceptual gap this fills in the tab journey

#### `tutorial` — end-to-end guided learning

- **Primary information type**: procedural, technical, narrative
- **Required sections**: Prerequisites (explicit); ordered steps with verification; final outcome statement
- **Specific check**: Every step executable; verification step at the end
- **Specific check**: Prerequisites section is complete and accurate — missing prerequisites are a publication blocker
- **Specific check**: No skippable steps — tutorials are end-to-end; on-demand depth goes in guides
- **Component requirement**: `<Steps>` or `<StyledSteps>` — numbered lists are not acceptable for procedures
- **Data needed**: Live system access for step verification; confirmed prerequisite list

#### `guide` — opinionated guidance for a specific outcome

- **Primary information type**: analytical, evaluative, conceptual, narrative
- **Permitted secondary types**: factual
- **Required sections**: The job (what outcome the guide helps achieve); the considerations or options; guidance toward the outcome; next steps
- **Specific check**: Guide takes a position — not "here are your options" but "here is how to decide"
- **Specific check**: No step-by-step procedures as the primary content — those belong in linked `instruction` pages
- **Specific check**: Evidence for evaluative claims must be real and sourced
- **Data needed**: Source evidence for any evaluative claims; the reader's decision or outcome being guided toward

#### `instruction` — step-by-step procedure

- **Primary information type**: procedural, technical
- **Permitted secondary types**: factual, structural
- **Required sections**: Prerequisites (exact, not vague); numbered steps; verification step; next steps
- **Specific check**: Every step tested and confirmed to produce stated output
- **Specific check**: Quickstart variant: first working state achievable in under 30 minutes
- **Specific check**: Troubleshooting variant: symptom-first ordering (not alphabetical, not system-chronological)
- **Component requirement**: `<Steps>` component family — never numbered prose lists
- **Data needed**: Live system access; exact environment for testing; version pinned

#### `reference` — structured lookup

- **Primary information type**: factual, technical, structural, change
- **Permitted secondary types**: none
- **Required sections**: Structured lookup surface (table, ParamField, ResponseField, or similar); source and date for time-sensitive values
- **Specific check**: No narrative framing before reference content — reader arrives knowing what to look up
- **Specific check**: Compendium variant (FAQ, glossary): question-led scan surface with canonical links
- **Specific check**: Changelog variant: version/date anchoring mandatory; breaking vs non-breaking labels in structure
- **Veracity standard**: Very high for all sections — reference pages have the highest factual bar
- **Data needed**: Primary authoritative sources for every value; staleness category for each claim type

#### `resource` — curated discovery

- **Primary information type**: structural, evaluative, narrative
- **Required sections**: What the resource collection is for; curated items with value propositions; filtering or categorisation if more than 5 items
- **Specific check**: Knowledge-hub variant: external content only — no instructional content mixed in
- **Specific check**: No TODO/Coming Soon — a resource page with placeholder items is a dead end
- **Data needed**: Current status of external links; value proposition for each curated item

## Proposed Master Review Pipeline

### A complete, ordered pipeline from "written" to "approved"

This pipeline applies to any single page. The phases are sequential; no phase starts until the preceding phase is complete and human-approved.

---

### Phase 0 — Pre-flight dependency check (before any review work)

**Purpose**: Confirm all inputs required by the review are present. A review run on incomplete inputs produces wasted output.

**Input required**:
- Page file path
- Tab name

**What to verify**:
- [ ] Phase 1 audience design output exists for this tab
- [ ] Tab map exists at `v2/[tab]/_workspace/tab-map.md`
- [ ] Page exists in `docs.json`
- [ ] Terminology is locked for this tab (Phase 3.5 complete)
- [ ] `veracity-library.md` is accessible
- [ ] `veracity.md` status is LOCKED
- [ ] `feedback-routing-map.md` exists

**Tool**: Manual pre-flight check using pack-guide.md for Phase 3 (content-pass pack-guide)

**Output**: Pre-flight pass/fail with specific blocking items named

**Pass criteria**: All dependencies confirmed at their exact paths. Any missing dependency stops the review.

**Current gap**: No automated pre-flight script exists. Pre-flight is a manual human step.

---

### Phase 1 — Structural position check (naming audit — Phase 8 in the pipeline)

**Purpose**: Lock section headings before veracity markers are placed. If headings change after veracity markers are placed, the markers become orphaned.

**Input required**:
- Page file
- Section naming rubric (`workspace/plan/active/CONTENT-WRITING/Prompts/section-naming.md`)

**What to check**:
- Score every H2 and H3 against the 5-dimension naming rubric (Precision, Depth, Stability, Clarity, Conciseness)
- Minimum score: 20/25
- Headings scoring below 20 must be revised before the review continues

**Tool**: Manual review using `Frameworks.mdx §4.1` + `section-naming.md`; Phase 5 of `layout-pass.md`

**Output**: Heading score table per heading (dimension breakdown, not just aggregate); list of revisions required

**Pass criteria**: All H2/H3 headings score ≥20/25. No generic structure words ("Overview", "Details", "Basics") standing alone as headings.

**Current gap**: No automated heading scorer. The `rubric-static-review` skill is not scoped for this. Manual execution only.

**CRITICAL NOTE**: This phase (naming audit = "Phase 8" in the pipeline) must run BEFORE Phase 2 below (veracity = "Phase 7" in the pipeline). This counter-intuitive ordering is locked in the pipeline spec.

---

### Phase 2 — Frontmatter and taxonomy check

**Purpose**: Confirm the page is correctly classified before reviewing content. Wrong classification produces wrong content checks.

**Input required**:
- Page file
- `Frameworks.mdx §1` (canonical enum tables)
- `workspace/plan/active/CONTENT-WRITING/Frameworks/frontmatter-taxonomy-update.md` (deprecated alias migration map)

**What to check** (checks 1.1–1.12 from `checks.mdx`):
- All 10 required frontmatter fields present
- `pageType` uses canonical 7-type schema
- `purpose` uses canonical 15-value set
- `audience` uses canonical 7-token set
- `complexity` and `lifecycleStage` use canonical values
- `veracityStatus` present and honest
- `description` subject-first, ≤160 chars, no self-reference
- OG image block complete

**Tool**: Manual check + `tools/config/frontmatter-taxonomy.js` (with caveat — see Validation Flags)

**Output**: Frontmatter table per field: present?, value, pass/fail, notes

**Pass criteria**: Zero missing fields; zero deprecated enum values; veracityStatus = unverified for any new or AI-generated content.

**Current gap**: `frontmatter-taxonomy.js` uses old 12-type schema. Running it will flag canonical 7-type values as invalid. Human must override the validator output.

---

### Phase 3 — Structure and content architecture check

**Purpose**: Confirm the page does one job for one audience, with correct adjacency to surrounding pages.

**Input required**:
- Page file
- Phase 1 audience design output for this tab (for persona fit)
- Tab map at `v2/[tab]/_workspace/tab-map.md` (for PREV_PAGE / NEXT_PAGE)

**What to check** (checks 4.1–4.10 from `checks.mdx`, Phases 2–3 and 7 of `content-pass.md`):
- Purpose statable in one sentence: "This page lets the [AUDIENCE] [concrete action]"
- PREV_PAGE / NEXT_PAGE adjacency coherent
- Scope is single (one job, not two)
- No dead ends (page has a next step)
- Prerequisites stated or linked
- Information type per section correct for `pageType`
- No content duplication from adjacent pages

**Tool**: `content-pipeline-pass-a` skill invoking `content-pass.md` Phases 2, 3, and 7

**Output**: Structured findings per check; verdict: PASS / NEEDS WORK / REWRITE REQUIRED

**Pass criteria**: Purpose statable in one sentence; adjacency coherent; single scope; no orphan sections; no pre-emptions from NEXT_PAGE.

**Current gap**: Requires tab map to exist. If tab map is absent, adjacency check degrades to PREV_PAGE/NEXT_PAGE fields only. Tab maps do not yet exist for most tabs (gate is locked).

---

### Phase 4 — Voice and copy check

**Purpose**: Confirm the page sounds like the Livepeer docs voice, serves the correct audience register, and contains no prohibited language.

**Input required**:
- Page file
- `voice-rules.md` (per-audience rules)
- Audience token from frontmatter (to select correct audience section)

**What to check** (checks 2.1–2.11 from `checks.mdx`; Phase 5 of `content-pass.md`):
- UK English throughout (no US spellings)
- Zero banned words: `effectively, essentially, basically, meaningful, significant, real` (as intensifier), `various, several, obviously, clearly`
- Zero banned phrases (full list in `content-pass.md §5.1`)
- Zero banned constructions: `not [X]` in value statements; unresolved `if [condition]` in body prose; `This page [verb]`; `can/may [verb]` in value claims
- Opening order: value/outcome before mechanism
- Paragraph discipline: one paragraph, one job; lead sentence states the fact; final sentence ends on fact, number, or next step
- Audience register correct (per-audience prohibited phrases absent)
- No passive value statements

**Tool**: `content-pipeline-pass-a` skill invoking `content-pass.md` Phase 5; `style-and-language-homogenizer-en-gb.js` for UK English

**Output**: Per-rule verdict (PASS / FAIL / PARTIAL) for every voice check; every FAIL quotes exact failing text and writes corrected version

**Pass criteria**: Zero universal blocking violations; zero audience-specific prohibited phrases; every FAIL has a named fix.

**Current gap**: `style-and-language-homogenizer-en-gb.js` catches spelling; it does not catch banned phrases or constructions — those require the prompt-based check. No automated banned-phrase scanner exists at the page level (though `banned-words.txt` and `banned-phrases.txt` exist as source files).

---

### Phase 5 — Information type and veracity preparation

**Purpose**: Tag every section with its information type to determine the veracity standard that applies. This is the prerequisite for Phase 6 (veracity pass).

**Input required**:
- Page file
- `Frameworks.mdx §1.6` (information type definitions and permitted types by pageType)
- `veracity.md` (standard per information type)

**What to check** (Phase 4 of `content-pass.md`):
- For each section: identify information type (`factual`, `technical`, `procedural`, `change`, `evaluative`, `structural`, `analytical`, `conceptual`, `narrative`)
- Confirm information type is permitted for the page's `pageType`
- Identify veracity standard: very high / high / medium / lower
- Flag structural mismatches (information type doesn't fit pageType)

**Tool**: `content-pipeline-pass-a` skill invoking `content-pass.md` Phase 4

**Output**: Per-section table: section heading, information type, permitted for pageType?, veracity standard

**Pass criteria**: Every section has an identified information type; no structural mismatches; information types match permitted types for pageType.

**Current gap**: No automation. Manual execution only. This classification is required input to Phase 6 — if Phase 5 is skipped, Phase 6 has no basis for knowing which claims need verification.

---

### Phase 6 — Veracity pass

**Purpose**: Verify factual accuracy of claims in the page. Place REVIEW markers on unverifiable claims. Set `veracityStatus` in frontmatter.

**Input required**:
- Page file (post-Phase 1 — section headings must be final)
- `veracity-library.md` (45-source library — must be loaded from file, not recalled from memory)
- `veracity.md` (LOCKED — confirm status before running)
- Phase 5 output (information type classification per section)
- `docs.json` (for structural claim verification)
- Tab terminology lock (Phase 3.5 output for this tab)

**What to check** (full `veracity-pass.md` phases 1–7):
- Phase 1: read and map the page; claim list per section
- Phase 2: classify each claim (25 claim types defined)
- Phase 3: apply confidence scoring matrix and auto-resolve rules
- Phase 4: place REVIEW markers before flagged blocks (never inline)
- Phase 5: mixed-type page handling; worst-case status derivation
- Phase 6: set `veracityStatus` in frontmatter; produce Phase 6 summary
- Phase 7: glossary update loop — route verified terminology back to `v2/[tab]/resources/glossary.mdx`

**CRITICAL rules**:
- Governance-controlled values are NEVER auto-resolved (inflation rate, bonding target, unbonding period, active set size, treasury parameters, reward cut rate, fee share defaults)
- Contract addresses are NEVER auto-resolved
- External RPC URLs are NEVER auto-resolved
- Economic data requires staleness date — always flag with re-verify schedule
- REVIEW markers go BEFORE the flagged block, never inline in prose

**Tool**: `veracity-pass.md` prompt; no automated equivalent exists

**Output**: Page with REVIEW markers added; `veracityStatus` updated in frontmatter; Phase 6 summary (section breakdown, REVIEW count, governance flag count, open items list); Phase 7 glossary update

**Pass criteria**: Every governance-controlled value has a REVIEW marker; REVIEW markers placed before blocks (not inline); `veracityStatus` set correctly; Phase 6 summary present and complete; no content changes — veracity pass adds markers only.

**Current gap**: `veracity-pass.md` is marked DRAFT. It requires 1 review agent + 3 test runs + human approval before production use. `veracity.md` and `veracity-library.md` are LOCKED. The glossary update step (Phase 7 of the veracity pass) assumes `v2/[tab]/resources/glossary.mdx` exists — this file may not exist for all tabs. The two framework files it references (`workspace/plan/active/CONTENT-WRITING/Frameworks/veracity.md` and `workspace/plan/active/CONTENT-WRITING/Frameworks/veracity-library.md`) do exist and are complete.

---

### Phase 7 — Layout and component check

**Purpose**: Apply MDX layer to approved content. Template, components, frontmatter completion, render validation.

**Input required**:
- Approved Pass A content (post-veracity pass)
- `layout-pass.md` prompt
- `section-naming.md`
- Template file from `snippets/templates/pages/` (matched to pageType + pageVariant)
- Component registry at `snippets/components/`
