<Tip>
  A node passes end-to-end verification when it is discoverable by gateways, priced within their MaxPrice threshold, and has processed at least one job. Run every check that applies to your configured mode.
</Tip>

Setup and configuration are complete. This page confirms the node is visible to the network and receiving work through the full routing path.

<Note>
  AI-only operators do not need active set membership. AI inference routing is capability-based, not stake-rank-based. Skip this check if you are running AI pipelines without video transcoding.

## Capability advertisement (AI and dual mode)

Check [tools.livepeer.cloud/ai/network-capabilities](https://tools.livepeer.cloud/ai/network-capabilities). Your configured pipelines should appear with the correct warm or cold status.

Also query your node directly:

```bash icon="terminal" filename="check-capabilities"
curl http://localhost:7935/getNetworkCapabilities | jq
```

If your pipelines are missing from the registry after five minutes, check these causes in order:

1. AI runner container not running — verify with `docker ps | grep ai-runner`
2. `aiModels.json` has invalid syntax — validate with `jq . < aiModels.json`
3. Model download still in progress — check logs for download progress messages

<CustomDivider middleText="Pricing Checks" />

## Confirm pricing is within gateway thresholds

Gateways apply a hard MaxPrice gate. An orchestrator priced above a gateway's threshold receives no work from that gateway.

```bash icon="terminal" filename="check-price"
curl -s http://localhost:7935/status | jq '.PricePerUnit'
```

Compare your price against the current distribution on [Livepeer Explorer](https://explorer.livepeer.org/orchestrators). Position in the lower half of active orchestrator prices when starting out.

For AI pipelines, check per-pipeline pricing in your `aiModels.json` against market rates shown on [tools.livepeer.cloud](https://tools.livepeer.cloud).

<CustomDivider middleText="Connectivity Checks" />

## Service URI reachability

Test from a different machine (not the same server):

```bash icon="terminal" filename="port-check"
curl -k https://YOUR_PUBLIC_IP:8935/status
```

Any response — including a JSON error — confirms port 8935 is open and reachable. A connection timeout means gateways cannot route work to the node. Check firewall rules, NAT or port-forwarding configuration, and cloud security group settings.

## Node process health

Confirm the node is running and the CLI port is responding:

```bash icon="terminal" filename="check-status"
curl -s http://localhost:7935/status | python3 -m json.tool
```

Expected: a JSON response showing `nodeType`, `version`, ETH account address, and ETH balance.

<CustomDivider middleText="Performance and Rewards" />

## Performance score

{/* VERIFY: Performance score formula (latency score x success rate) — confirm against go-livepeer source. */}

Performance score is a composite metric (latency score multiplied by success rate) that gateways use alongside stake weight and price when selecting orchestrators. A low score reduces job routing probability.

After receiving initial jobs, verify the score is tracking by checking Prometheus metrics:

```bash icon="terminal" filename="check-metrics"
curl -s http://localhost:7935/metrics | grep -i "score\|latency\|success"
```

Factors that degrade performance score:
- High transcoding latency (GPU saturation or insufficient `-maxSessions` headroom)
- Failed segments (GPU errors, VRAM pressure, codec incompatibility)
- Service URI latency (network distance between node and gateways)

## Reward call verification

Reward calling distributes LPT inflation each round (~22 hours). Confirm automatic calling is enabled:

```bash icon="terminal" filename="check-reward-config"
docker compose config 2>/dev/null | grep -i "reward" || echo "Check startup flags for -reward=false"
```

The startup command should omit `-reward=false`. Its absence means automatic calling is enabled.

After the first full round completes, verify on [Arbiscan](https://arbiscan.io) that a `reward()` transaction appeared from your orchestrator address. Check logs:

```bash icon="terminal" filename="check-reward-log"
grep -i "calling reward\|reward call" /var/log/livepeer/livepeer.log | tail -5
```

Missing reward calls after a round usually indicate one of three causes: the node was offline during round initialisation, the Arbitrum ETH balance is too low for gas, or `-reward=false` is set in the startup command.

<CustomDivider middleText="First Job Confirmation" />

## Confirm first job received

For video transcoding, watch logs for job receipt:

```bash icon="terminal" filename="check-video-job"
grep -i "got segment\|transcode result" /var/log/livepeer/livepeer.log | tail -10
```

Expected output includes `Got segment` and `Transcode result OK`.

For AI inference, watch for inference request processing:

```bash icon="terminal" filename="check-ai-job"
grep -i "RunAI request\|pipeline=" /var/log/livepeer/livepeer.log | tail -10
```

<Note>
  Job volume depends on network demand, your pricing position, and your performance score. A new node with competitive pricing in the active set typically receives its first video job within 24 to 48 hours. AI jobs route by capability and price — first AI job timing depends on gateway demand for the pipelines you advertise.
</Note>

<Warning>
  If no jobs arrive within 48 hours despite passing all checks above, the most common cause is port 8935 not reachable from the internet. Re-test from an external machine. The second cause is price above gateway caps — compare your price against active orchestrators on the Explorer. Third: for video, confirm you are in the active set; for AI, confirm your capabilities appear on tools.livepeer.cloud.
</Warning>
