## Prometheus and Grafana

Enable the Prometheus endpoint with the `-monitor` flag at startup:

```bash icon="terminal" filename="enable-monitoring"
livepeer \
  -orchestrator \
  -transcoder \
  -monitor \
  -metricsPerStream \
  ...
```

Metrics are exposed at `http://localhost:7935/metrics`.

### Fastest path: Docker monitoring stack

Livepeer maintains a Docker image bundling Prometheus and Grafana with starter dashboards:

```bash icon="terminal" filename="docker-monitoring"
docker run --net=host \
  --env LP_MODE=standalone \
  --env LP_NODES=localhost:7935 \
  livepeer/monitoring:latest
```

Grafana is available at `http://localhost:3000` (default credentials: `admin` / `admin`).

{/* VERIFY: Confirm livepeer/monitoring Docker image is still maintained and compatible with current go-livepeer. [source: github.com/livepeer/livepeer-monitoring] */}

For multi-node setups, pass a comma-separated list of nodes:

```bash icon="terminal" filename="multi-node-monitoring"
docker run --net=host \
  --env LP_MODE=standalone \
  --env LP_NODES=node1.example.com:7935,node2.example.com:7935 \
  livepeer/monitoring:latest
```

### Custom Prometheus setup

Add go-livepeer as a scrape target in your existing `prometheus.yml`:

```yaml icon="code" filename="prometheus.yml"
scrape_configs:
  - job_name: 'livepeer-orchestrator'
    static_configs:
      - targets: ['localhost:7935']
    scrape_interval: 15s
    metrics_path: /metrics
```

## Alerting

Configure alerts for the failure modes that cost money:

```yaml icon="code" filename="prometheus-alerts.yml"
groups:
  - name: livepeer-orchestrator
    rules:

      - alert: LowETHBalance
        expr: "livepeer_eth_balance < 0.02"
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "ETH balance below 0.02 - reward calls and ticket redemptions will fail"

      - alert: OrchestratorAtCapacity
        expr: "livepeer_current_sessions_total >= livepeer_max_sessions"
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Session limit reached for 5+ minutes - jobs are being rejected"

      - alert: TranscodeFailureRate
        expr: "rate(livepeer_transcode_failed_total[10m]) > 0.1"
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Transcode failure rate above 10% over 10 minutes"
```

{/* VERIFY: Confirm metric names livepeer_eth_balance, livepeer_current_sessions_total, livepeer_max_sessions, livepeer_transcode_failed_total against current go-livepeer source. */}

## Monitoring AI runner containers

AI inference runs in the `ai-runner` Docker container alongside go-livepeer. Monitor it separately for faster signal on AI-specific issues.

```bash icon="terminal" filename="check-ai-runner"
# Container status
docker ps --filter name=livepeer-ai-runner

# Follow logs live
docker logs -f livepeer-ai-runner

# Resource usage
docker stats livepeer-ai-runner
```

## Log capture

For production nodes, capture logs to a file alongside terminal output:

```bash icon="terminal" filename="log-capture"
livepeer \
  -orchestrator \
  -transcoder \
  -monitor \
  ... \
  2>&1 | tee /var/log/livepeer/livepeer.log
```

Useful search patterns for daily review:

```bash icon="terminal" filename="log-search"
# Reward calls
grep -i "reward" /var/log/livepeer/livepeer.log | tail -10

# Errors only
grep -i "error\|fail\|crit" /var/log/livepeer/livepeer.log | tail -20

# Job activity
grep -i "got segment\|RunAI request" /var/log/livepeer/livepeer.log | tail -10
```

For `systemd`-managed services:

```bash icon="terminal" filename="journalctl-search"
journalctl -u livepeer --since "1 hour ago"
journalctl -u livepeer -p err    # errors only
```
