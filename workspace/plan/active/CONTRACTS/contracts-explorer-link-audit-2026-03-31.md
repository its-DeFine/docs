## Verification Hierarchy

Rank these methods from strongest to weakest when checking whether a published contract is official and current to Livepeer:

1. `controllerRegistered` for controller-managed contracts.
2. Proxy downstream mapping, either `proxyTarget` on the proxy row or `proxyAddress` on the current implementation row.
3. Canonical address source from the docs registry, either `governor-scripts` or `authority-catalog`.
4. Explorer header `Livepeer` label.
5. Explorer `Contract Creator` value, especially `Livepeer: Deployer`.

Notes:

- `Contract Creator` appears on all 90 explorer pages, but it is supporting evidence, not primary address truth.
- `Livepeer: Deployer` appears on 73 of 90 pages. Absence of that label does not prove a contract is wrong if controller or proxy evidence is stronger.
- `Header Livepeer` is the strongest explorer-only signal, but it is not universal. Two active rows fail it: `AIServiceRegistry` and `L1LPTGateway`.
- `AIServiceRegistry` is currently the weakest active row by explorer evidence because it has no header `Livepeer` label and no `Livepeer: Deployer` creator label.
