**Scope:** All documentation surfaces for the template library — `snippets/templates/` (33 files across pages, blocks, and docs-guide subfolders), the GitHub issue/PR templates (`.github/ISSUE_TEMPLATE/`, `.github/pull_request_template.md`), and their catalog and discovery surfaces.

## Source of truth

**Canonical:** The template files themselves — `snippets/templates/**/*.mdx`. These are the actual usable templates.

**Derived (generated):**
- `docs-guide/catalog/templates-catalog.mdx` — GitHub issue/PR templates only (not the page/block templates)
- `docs-guide/catalog/ui-templates.mdx` — page/block templates (no CI, manual only)

**Governance authority:**
- None. There is no template documentation policy defining what templates exist, what metadata they must contain, or how they are discovered.

**Critical missing source of truth:**
- There is no catalog that covers ALL templates in one place. The two existing catalogs cover disjoint sets (GitHub templates vs page templates) and are generated by different scripts with different CI wiring. A contributor asking "what templates are available?" has no single answer.
- There is no template metadata standard. None of the 33 page/block template files have a documented metadata header format (no equivalent of JSDoc for scripts). Template purpose, audience, and page type are embedded in filename and content only.

## Gaps and issues

1. **`templates-catalog.mdx` catalogs GitHub templates, not page templates.** Despite being titled "Templates Catalog," it inventories only `.github/ISSUE_TEMPLATE/*.yml` and `.github/pull_request_template.md`. The 33 page and block templates in `snippets/templates/` are not represented. This naming mismatch is a significant discoverability gap.

2. **`ui-templates.mdx` has no CI trigger.** The generator for page/block templates (`generate-ui-templates.js`) is annotated `@pipeline manual`. No workflow runs it. `ownerless-governance-surfaces.json` declares this surface as `gate_layer: "pr-changed"` and `rollout_state: "autofix"` — but neither condition is currently implemented. The catalog can drift indefinitely as templates are added or renamed.

3. **Duplicate template names (file proliferation pattern).** Several template types exist in both a "canonical" and a "-template" suffixed version:
   - `glossary-tab.mdx` + `glossary-tab-template.mdx`
   - `glossary-consolidated.mdx` + `glossary-consolidated-template.mdx`
   - `openapi-endpoint-page.mdx` (×2: in both `pages/` and `pages/reference-and-api/`)
   - `source-of-truth.mdx` + `source-of-truth-template.mdx`
   - `changelog.mdx` + `changelog-template.mdx`
   - `tutorial-page.mdx` + `tutorial.mdx` (in different subfolders)
   There is no convention distinguishing "template" files from "canonical example" files. Contributors cannot determine which to copy.

4. **`snippets/templates/docs-guide/` is not cataloged anywhere.** The 7 internal docs-guide templates (`framework-page.mdx`, `feature-map-page.mdx`, `tooling-reference-page.mdx`, `policy-page.mdx`, `script-catalog.mdx`, `component-catalog-template.mdx`, `catalog-page.mdx`) have no inventory surface. These are high-value templates for internal governance authoring but are discoverable only by browsing the folder.

5. **`README.mdx` is the only contributor guide — it is a stub.** The file is audience=internal and links to component library and other resources rather than explaining when to use which template or how to copy and fill them. There is no user-facing contributor guide explaining the template system.

6. **No template metadata standard.** No field declares what page type a template produces, its intended audience, or its usage context. Contrast with components (JSDoc headers) and scripts (11-tag JSDoc). Templates rely entirely on filename and content for discoverability.

7. **VS Code snippet integration is incomplete.** `.vscode/templates.code-snippets` is generated by `generate-ui-templates.js` — but without CI wiring, it may be stale. The snippet file provides fast template insertion, but only for contributors who know it exists. `dev-tools.mdx` does not document template snippets (only component snippets).

8. **Both template catalogs share generator infrastructure with non-template concerns.** `templates-catalog.mdx` shares its generator with `workflows-catalog.mdx`. `ui-templates.mdx` shares its generator with `.vscode/components.code-snippets`. Neither template catalog can evolve independently without risking side effects on the other outputs.

## Recommendations

1. **Rename `templates-catalog.mdx` to `github-templates-catalog.mdx`.** The current name is misleading. The file catalogs GitHub issue/PR templates; rename it to accurately describe its scope. Update `generate-docs-guide-indexes.js` output path accordingly, and update `docs.json` navigation.

2. **Wire `ui-templates.mdx` to CI.** Add `generate-ui-templates.js --check` to `check-docs-guide-catalogs.yml`. Add `generate-ui-templates.js --write` to `generate-docs-guide-catalogs.yml` with a path filter on `snippets/templates/**`. This is declared in `ownerless-governance-surfaces.json` but not yet implemented.

3. **Define a template metadata header standard.** Add a comment block to every template file (similar to the `generated-file-banner:v1` pattern) declaring:
   - `@template-type`: the page type this template produces (e.g., `how-to`, `reference`, `changelog`)
   - `@audience`: intended contributor audience
   - `@usage`: one-line description of when to use this template
   This makes templates machine-queryable and enables richer catalog generation.

4. **Resolve duplicate templates.** Audit the 6+ template pairs listed in gap #3. For each pair: keep one canonical version, delete or redirect the other, document the canonical name in the template metadata header. A naming convention: all templates end with `.mdx`, no `-template` suffix needed.

5. **Add `snippets/templates/docs-guide/` to the UI templates catalog.** The generator currently only scans `pages/` and `blocks/`. Extend the scanner to include `docs-guide/` templates, with a separate section in the catalog output.

6. **Write a contributor template guide.** Add `v2/resources/documentation-guide/using-templates.mdx` (or a section in an authoring guide) that explains: what templates are available, how to copy one, what fields to fill in, and how template names map to page types. This is the most immediately useful contributor action.

## Upstream / downstream effects

**Changes upstream (affecting this concern):**
- Adding a template to `snippets/templates/pages/` → `ui-templates.mdx` drifts until manually regenerated
- Adding a GitHub issue template → `templates-catalog.mdx` auto-regenerates on push→main
- Changing `generate-docs-guide-indexes.js` → affects both templates-catalog and workflows-catalog simultaneously

**Changes downstream (this concern affecting others):**
- `docs-guide/catalog/ui-templates.mdx` is consumed by contributors authoring new pages and by agents selecting page structures
- `.vscode/templates.code-snippets` enables fast template insertion — if stale, contributors get outdated template options
- Resolving duplicate templates has upstream effects on any v2 page that currently uses a deprecated template variant
- Splitting the shared generator (recommendation for workflows audit) directly unblocks independent evolution of this catalog
