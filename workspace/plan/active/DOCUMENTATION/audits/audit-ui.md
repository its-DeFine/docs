# UI Documentation Audit

**Date:** 2026-03-23
**Scope:** All published documentation for the custom component UI system — `v2/resources/documentation-guide/component-library/` (32 generated pages across 4 locales), the governance-level `docs-guide/` docs, and contributor-facing UI authoring surfaces.

This audit focuses on the **published output layer** of the component documentation system. The source-layer (JSDoc, registry generation) is covered in `audit-components.md`. The overlap is intentional: this audit traces what contributors and end users see vs what the system actually produces.

## Source of truth

**Canonical:** JSDoc in `snippets/components/**/*.jsx` → `component-registry.json` → `generate-component-docs.js` → 32 v2 published pages.

**The published pages are the UI documentation that contributors and users see.** Their accuracy depends on:
1. JSDoc headers being complete and correct in `.jsx` files
2. `component-registry.json` being regenerated when JSDoc changes
3. `generate-component-docs.js` being re-run when the registry changes
4. The generator template containing correct script path references

**Current failure at step 4:** The generator template has the wrong script path baked in, so all 32 output pages tell contributors to run a non-existent script.

**Missing:**
- No canonical status enum file. The registry uses `stable`, `experimental`, `planned`; the components-catalog uses `⬜ Placeholder` for display; `docs-guide/contributing/mintlify.mdx` has draft content covering native Mintlify but no custom component contributor guide exists in `v2/`.

## Gaps and issues

1. **Stale generator path in all 32 published v2 pages.** Every file has a banner saying: `Generation Script: operations/scripts/generate-component-docs.js`. This path does not exist. The correct path is `operations/scripts/generators/components/documentation/generate-component-docs.js`. The fix must go in the generator template (not the files directly), so the next CI generation cycle fixes all 32 files automatically. This is a documentation correctness defect — contributors who follow the banner instruction will get a "file not found" error.

2. **No contributor guide for using components.** There is no v2 page explaining how to use custom components when authoring docs. Contributors must either know from experience, read `dev-tools.mdx` (draft status), or inspect component source files. `docs-guide/contributing/mintlify.mdx` covers native Mintlify components (in draft), but custom components have no equivalent guide. This is the largest usability gap in the UI documentation layer.

3. **`docs-guide/contributing/mintlify.mdx` is in draft status.** Native Mintlify component reference — covers `<Card>`, `<Tabs>`, `<Accordion>`, etc. — is draft-only. Contributors cannot rely on it as a reference. It has no `lastVerified` gate and no CI freshness check.

4. **Status enum conflict between published pages and catalog.** Published component library pages use `stable`, `experimental`, `planned` (from the registry). `components-catalog.mdx` renders a fifth status: `⬜ Placeholder` (for display of `planned` components). No canonical enum file defines what each status means or how it should be displayed. Contributors and agents see different status labels depending on which surface they consult.

5. **`docs-guide/features/ui-system.mdx` lastVerified is 2026-03-12.** This is the UI authoring system map — the broadest overview of the entire custom component system. At 11 days old relative to the audit date, it may not reflect recent additions to the component library. No CI gate tracks its freshness.

6. **Race condition: `generate-component-registry.yml` and `generate-docs-guide-catalogs.yml` both auto-commit to main.** Both trigger on push→main and both commit to `main`. If a jsx file changes: `generate-component-registry.yml` fires, generates the registry, auto-commits → this push triggers `generate-docs-guide-catalogs.yml` which regenerates component docs → another auto-commit. In practice, the cascade is sequential, but if both fire within the same push event window, they may conflict. This race is the same issue documented in the components audit; it is a workflow-layer defect that affects the UI output layer.

7. **32 locale files (es/fr/cn) have identical defects but are harder to audit.** The 3 locale directories mirror the English component library pages. Any defect in the English generator template propagates to all 32 files equally. The locale files have no independent freshness checks and no locale-specific review process documented.

## Recommendations

1. **Fix the stale banner path in the generator template.** Update the banner string inside `generate-component-docs.js` to reference `operations/scripts/generators/components/documentation/generate-component-docs.js`. Trigger the generation workflow — all 32 files will auto-fix on the next push→main with a jsx change (or via `workflow_dispatch`).

2. **Write a contributor component usage guide.** Create `v2/resources/documentation-guide/using-components.mdx`. Minimum content: how to find components, import patterns, the registry URL, when to use custom vs native Mintlify components, and usage examples for the 5 most common components. Add it to `docs.json` navigation alongside the component library pages.

3. **Promote `mintlify.mdx` from draft to current.** Complete the native Mintlify component reference. Add `lastVerified` and set `status: current`. This is a high-value contributor reference that fills the gap between the custom component guide (step 2) and native components.

4. **Define a canonical status enum.** Create `tools/config/component-status-enum.json` with the four valid values (`stable`, `experimental`, `planned`, `deprecated`) and their display representations (`🟢 Stable`, `🧪 Experimental`, `⬜ Planned`, `🟠 Deprecated`). Reference it from the generator template and the catalog template. Eliminates the `planned` vs `⬜ Placeholder` mismatch.

5. **Resolve the auto-commit race condition.** Consolidate `generate-component-registry.yml` and the downstream catalog generation into one sequential workflow. This is the same recommendation from the workflows audit; it prevents competing auto-commits to main.

## Upstream / downstream effects

**Changes upstream (affecting this concern):**
- Adding/editing a `.jsx` file → cascades to registry → cascades to 32 published v2 pages
- Changing JSDoc fields → changes published component library pages on next push→main
- Modifying `generate-component-docs.js` generator template → affects all 32 output files on next generation

**Changes downstream (this concern affecting others):**
- `v2/resources/documentation-guide/component-library/**` is public-facing content visible to all users — the stale banner path is user-visible
- `docs-guide/features/ui-system.mdx` is the primary agent context for the UI layer — stale = wrong agent guidance
- Status enum decision (recommendation 4) affects: catalog display in `components-catalog.mdx`, health check scripts, and any validator that reads status fields
- Writing the contributor usage guide (recommendation 2) fills a gap identified across tooling, templates, and UI audits
