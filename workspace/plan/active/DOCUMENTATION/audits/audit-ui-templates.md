# UI Templates Documentation Audit

**Date:** 2026-03-23
**Scope:** The `ui-templates` governance surface — `snippets/templates/pages/**`, `snippets/templates/blocks/**`, `.vscode/templates.code-snippets`, and `docs-guide/catalog/ui-templates.mdx`.

This audit is scoped to the **generation and documentation pipeline** for UI templates, not the content of the templates themselves (that is covered in `audit-templates.md`). The focus here is: how are UI templates cataloged, how do contributors discover them, and where does the pipeline break down?

## Source of truth

**Canonical:** Template files at `snippets/templates/pages/**` and `snippets/templates/blocks/**`. These are the authoritative source from which `ui-templates.mdx` and `.vscode/templates.code-snippets` are generated.

**Derived (generated):**
- `docs-guide/catalog/ui-templates.mdx` — catalog page
- `.vscode/templates.code-snippets` — VS Code snippet file
- `.vscode/components.code-snippets` — also generated by same script; shares run

**Governance authority:**
- `tools/config/ownerless-governance-surfaces.json` — declares the surface (but not enforced)
- No dedicated policy file for UI template governance

**Missing:**
- No JSON registry (no equivalent of `component-registry.json` for templates)
- No schema for template metadata
- No template preview route system (catalog references preview routes but none exist)
- The gap between declared governance (`gate_layer: pr-changed, rollout_state: autofix`) and actual CI wiring (nothing) is the defining issue for this surface

---

## Gaps and issues

1. **`generate-ui-templates.js` is marked `@pipeline manual` but `ownerless-governance-surfaces.json` declares `rollout_state: autofix` and `gate_layer: pr-changed`.** These two governance declarations directly contradict each other. The ownerless config says this should be auto-fixed at PR time; the script's own JSDoc says it runs manually. No CI implements either. This contradiction has persisted undetected because neither declaration has a validator.

2. **No JSON registry for templates.** `component-registry.json` provides a machine-readable inventory of all components with metadata. Templates have no equivalent. The generator discovers templates via file system glob at runtime, with no persistent structured record. This means:
   - No agent can enumerate available templates without running the generator or reading the raw file system
   - No schema validates template metadata
   - No freshness tracking is possible without running the generator

3. **`.vscode/templates.code-snippets` and `.vscode/components.code-snippets` are co-generated but cover different domains.** Both are produced by a single `generate-ui-templates.js` run. A contributor updating templates must also regenerate component snippets (or vice versa) to keep both snippet files current. These two outputs have different update triggers and should be independently generated.

4. **No CI trigger means catalog can be arbitrarily stale.** With 33 template files and no generation CI, the catalog can be weeks or months behind. The `status: active` and `maintenance: generated` frontmatter in `ui-templates.mdx` implies it is maintained — but `@pipeline manual` means it is not. The frontmatter is misleading.

5. **No v2 user-facing page for UI templates.** `v2/templates/` directory does not exist. Contributors accessing docs through the public site cannot find a UI templates reference under the documentation guide — only docs-guide internal pages (not publicly surfaced in the main nav) contain this catalog. Compare with component library pages at `v2/resources/documentation-guide/component-library/`.

6. **Template preview routes are referenced in the catalog but do not exist.** The `ui-templates.mdx` generator references preview routes for templates. No `v2/templates/` path exists in `docs.json`. Preview infrastructure for templates is not implemented.

7. **Generator shares its run with component snippet generation, coupling two independent concerns.** A change to template files that requires regeneration also regenerates component snippets. A change to component registry that requires re-running snippet generation also regenerates the template catalog. These are independent surfaces with different update frequencies, incorrectly coupled in one script.

## Recommendations

1. **Wire `generate-ui-templates.js --check` to `check-docs-guide-catalogs.yml`.** Add one step. This implements the `gate_layer: pr-changed` declared in `ownerless-governance-surfaces.json`. The check fails if the catalog or VS Code snippets are out of sync with template files.

2. **Wire `generate-ui-templates.js --write` to `generate-docs-guide-catalogs.yml`.** Add with path filter on `snippets/templates/**`. This implements the `rollout_state: autofix` declared in `ownerless-governance-surfaces.json`. Auto-commit `.vscode/templates.code-snippets` and `ui-templates.mdx` when templates change.

3. **Resolve the `@pipeline manual` vs `rollout_state: autofix` contradiction.** Once CI is wired, update the script's `@pipeline` annotation to reflect the actual pipeline. Remove the manual annotation to avoid future confusion.

4. **Create `tools/config/template-registry.json`.** Generate a persistent JSON registry of templates at build time (parallel to `component-registry.json`). Minimum fields per template: `id`, `name`, `type`, `path`, `description`, `audience`, `pageType`. This makes templates machine-queryable and enables richer catalog generation and agent discovery.

5. **Separate template snippet generation from component snippet generation.** Split `generate-ui-templates.js` so that template snippet generation and component snippet generation are independently runnable steps. Each should have its own CI trigger path filter.

6. **Create `v2/resources/documentation-guide/using-templates.mdx`.** A user-facing page under the published component library that explains the template system, lists available templates, and shows how to use VS Code snippets. Add it to `docs.json` navigation.

## Upstream / downstream effects

**Changes upstream (affecting this concern):**
- Adding a template to `snippets/templates/pages/` → catalog and snippets drift (no CI to detect)
- Changing `component-registry.json` → affects co-generated component snippets in same run
- Renaming template files → old entries persist in catalog until manually regenerated

**Changes downstream (this concern affecting others):**
- `.vscode/templates.code-snippets` is consumed by all contributors using VS Code — stale snippets mean outdated template options at authoring time
- `docs-guide/catalog/ui-templates.mdx` is consumed by agents selecting page structures — stale catalog means agents propose wrong templates
- Creating `template-registry.json` (recommendation 4) would unblock richer tooling: template health checks, orphan detection, type-driven suggestions
- Implementing CI wiring for this surface directly fulfills the declared `ownerless-governance-surfaces.json` contract, moving it from `rollout_state: autofix` (declared but not real) to actual autofix
