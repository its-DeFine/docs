## Source of truth

**Canonical:** `docs.json` — controls all public routing and navigation. `v2/**/*.mdx` files — the actual page content. `v2/index.mdx` — the manually maintained flat index of all linked pages.

**Derived (generated):**
- `docs-guide/catalog/pages-catalog.mdx` — tree view generated from `v2/index.mdx` links that also appear in `docs.json`

**Governance authority:**
- `docs-guide/frameworks/page-taxonomy-framework.mdx` — taxonomy definition
- `tools/lib/frontmatter-taxonomy.js` — machine-readable taxonomy values
- `docs-guide/policies/v2-folder-governance.mdx` — folder lane rules

**Critical gap:** The `frontmatter-contract` surface in `ownerless-governance-surfaces.json` has `rollout_state: "advisory"`. This means frontmatter taxonomy is recommended but not enforced at any gate. The taxonomy framework document defines valid values; the validator exists; but neither pre-commit nor CI blocks pages with invalid frontmatter.

---

## Gaps and issues

1. **Frontmatter schema is advisory, not enforced.** `tools/lib/frontmatter-taxonomy.js` defines canonical values for `pageType` (12 values), `audience` (9 values), `status` (9 values), and `purpose` (10 values). A validator exists (`operations/tests/unit/frontmatter-taxonomy.test.js`). However, `ownerless-governance-surfaces.json` sets `rollout_state: "advisory"` — meaning no gate blocks incorrect frontmatter. Over 703 pages, this is a significant data quality gap. Agents querying pages by type or audience cannot trust the values.

2. **`v2/index.mdx` is manually maintained and is the source for the pages catalog generator.** The generator reads `v2/index.mdx` link list, not `docs.json` directly. If a page is in `docs.json` but not in `v2/index.mdx`, it will not appear in `pages-catalog.mdx`. This means the catalog may systematically exclude pages added directly to `docs.json` without a corresponding update to the index. No validation catches this divergence.

3. **`pages-catalog.mdx` is a tree view only — no page metadata.** The catalog shows the folder hierarchy of pages but includes no frontmatter data (no pageType, audience, status, lastVerified). It is useful for navigation structure but useless for content health queries. Contrast with the scripts catalog (which includes type, concern, purpose per entry) — pages have richer metadata in their frontmatter but none of it surfaces in the catalog.

4. **No navigation alignment check (v2 pages vs docs.json).** There is no validator that reports: "these v2 files exist but are not in docs.json" or "docs.json references paths that don't exist as files." `check-docs-guide-catalogs.yml` checks catalog freshness but not navigation completeness. Orphaned pages (written but never added to nav) and dead links (in nav but file deleted) have no systematic detection.

5. **No `lastVerified` staleness gate at PR time.** `content-health.yml` runs weekly and can flag stale pages, but it is not a PR gate. A contributor can merge a page with `lastVerified: 2024-01-01` without any warning. The taxonomy defines the field; no enforcement exists at commit or PR time.

6. **`v2/index.mdx` has no generation pipeline — it is the only manually maintained catalog source.** Every other major concern (components, scripts, workflows, templates) has either a generated catalog or a structured source. `v2/index.mdx` is an exception: it is the raw source for the pages catalog generator but is itself maintained by hand. If a page is added to docs.json but not to `v2/index.mdx`, it disappears from the catalog silently.

7. **`pages-catalog.mdx` and `v2/index.mdx` serve overlapping purposes via different generators.** Both are page inventories. `v2/index.mdx` is the human-editable flat list. `pages-catalog.mdx` is the generated tree. An agent or contributor looking for "all v2 pages" encounters two surfaces with different formats, different coverage (catalog depends on index), and no clear guidance on which to use.

8. **703 pages with no deletion audit.** When pages are deleted, redirects may or may not be configured. No CI audits for deleted pages without redirects. The only deletion gate is the pre-commit hook deletion guard — which blocks unapproved deletions but does not verify that a redirect was added to `docs.json`.

## Recommendations

1. **Add frontmatter validation to `check-docs-guide-catalogs.yml`.** Run `frontmatter-taxonomy.test.js --check` on changed MDX files in the PR diff. This moves `frontmatter-contract` from `advisory` to `pr-changed` enforcement — soft enough that it only checks touched files, not all 703 pages at once.

2. **Add navigation alignment check.** Write a validator that cross-references `v2/index.mdx` links against `docs.json` entries and reports: pages in `docs.json` missing from `v2/index.mdx`, and `v2/index.mdx` links not present in `docs.json`. Add to `check-docs-guide-catalogs.yml`.

3. **Add frontmatter metadata to `pages-catalog.mdx`.** Extend `generate-docs-guide-pages-index.js` to read frontmatter from each page and emit a table (in addition to the tree view) with `pageType`, `audience`, `status`, and `lastVerified` per page. This makes the catalog useful for content health queries, not just navigation structure.

4. **Add a `lastVerified` staleness warning to the PR gate.** In `check-docs-guide-catalogs.yml`, run a check on changed files: if a modified v2 page has `lastVerified` older than 90 days, emit a warning (not a failure). This is a soft gate that keeps staleness visible without blocking merges.

5. **Consider generating `v2/index.mdx` from `docs.json`.** The index is manually maintained but could be generated from `docs.json` entries. This eliminates the dual-maintenance problem and ensures the pages catalog generator always reflects actual navigation. Implement with an exit-if-no-diff guard so it only commits when the navigation actually changes.

6. **Add a deletion-with-redirect check to the pre-commit hook.** When a file is being deleted from `v2/`, check if a redirect entry exists in `docs.json` for its path. Warn (not block) if none is found. This prevents silent deletion of pages without redirect coverage.

## Upstream / downstream effects

**Changes upstream (affecting this concern):**
- Adding a page to docs.json → should also add to v2/index.mdx (no automated sync)
- Editing frontmatter taxonomy values in `frontmatter-taxonomy.js` → changes what is "valid" across all 703 pages
- Removing a page → pre-commit deletion guard fires; but no redirect validation

**Changes downstream (this concern affecting others):**
- `pages-catalog.mdx` is consumed by agents navigating the docs structure — shallow catalog = incomplete agent context
- `v2/index.mdx` drift from docs.json means the pages catalog silently misses pages
- Frontmatter advisory-only status means any tooling querying page metadata (scripts-catalog, content-health) cannot trust field values
- Enforcing frontmatter at PR time (recommendation 1) will surface existing invalid frontmatter on every PR that touches a page — plan for a remediation batch
