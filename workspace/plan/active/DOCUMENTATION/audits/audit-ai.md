## Source of truth

**Canonical:** `AGENTS.md` is the baseline source of truth for all AI agents. Adapters derive from it but may extend it. `ai-tools/registry/ai-tools-registry.json` is the canonical inventory of the ai-tools surface.

**Derived:**
- `ai-tools/registry/ai-tools-inventory.md` — generated report from the registry
- `v2/**/*-data.json` — AI companion files generated per glossary page

**Governance authority:**
- `docs-guide/policies/agent-governance-framework.mdx` — approved adapter locations, lane rules, workspace boundaries
- `ai-tools/registry/ai-tools-registry.json` — machine-readable source for what exists in `ai-tools/`

**Fragmentation problem:** The baseline + adapter model means the source of truth for AI rules is split across 10 files (AGENTS.md + 9 adapter files). `AGENTS.md` is the authoritative baseline, but there is no automated check that adapters are consistent with it. Parity is enforced by convention, not by CI.

## Gaps and issues

1. **`ai-features.mdx` is a stub with mostly TODOs.** This is the primary contributor-facing documentation for AI capabilities in the docs system. The file exists as draft, with sections about what AI features are available, but most content is placeholder. Contributors and agents trying to understand the AI layer cannot rely on it. It has no `lastVerified` and no CI freshness check.

2. **No `llms.txt`.** `generate-llms-files.yml` and `verify-llms-files.yml` workflows exist (both in the workflows glob). These suggest an `llms.txt` generation pipeline is in place or planned. However, there is no `llms.txt` file in the repo root and no documentation of where it should live or what it contains. The AI-facing site index for LLM crawlers is absent.

3. **`ai-tools-registry.json` has `rollout_state: "migrating"` and `ownerless_ready: false`.** The registry is the source of truth for `ai-tools/` artifacts but is not yet fully governed. `ownerless-governance-surfaces.json` records it as `gate_layer: "pr-changed"` but `rollout_state: "migrating"` means the PR gate is not yet active. The validator (`validate-ai-tools-registry.js`) must be run manually.

4. **No adapter parity check.** `AGENTS.md` defines core rules that all adapters should implement. There is no CI validator that reads each adapter file and verifies it covers the baseline rules. Adapters can drift from the baseline indefinitely without detection. This is especially concerning for `.github/copilot-instructions.md` and `.windsurf/rules/repo-governance.md` — these have unknown freshness and no freshness gate.

5. **AI companion generation triggers only on main — not on PR branches.** `generate-ai-companions.yml` triggers `push.branches: [main]`. Contributors reviewing PRs on `docs-v2` or feature branches cannot see freshly generated companion files — they see the last main-branch generation. For agents reviewing PRs, companion context may be stale. The PR gate (`check-ai-companions.yml`) validates consistency but does not generate fresh companions.

6. **`docs-guide/tooling/ai-tools.mdx` is missing frontmatter delimiters.** As noted in the tooling audit, this governance entrypoint file has bare key-value pairs without `---` delimiters. It is the primary tooling reference for the AI registry but is structurally malformed. Mintlify will not parse its frontmatter.

7. **No v2 user-facing AI documentation.** All AI documentation lives in `docs-guide/` (internal) or `AGENTS.md` (agent-only). There is no v2 public page explaining what AI capabilities the docs system has, which tools integrate with it, or how it works. This is a contributor communication gap.

8. **Lane boundary enforcement is advisory for some lanes.** `ai-tools-registry.json` defines 4 lanes (`templates`, `local`, `workspace`, `exports`). The `workspace` lane is `advisory_only: true`. No CI enforces that workspace-scoped content stays out of export or template lanes. Drift is possible without detection.

## Recommendations

1. **Complete `ai-features.mdx`.** Remove TODO placeholders and write the canonical description of AI capabilities: companion file pipeline, glossary generation, adapter model, skills system, registry. Add `status: current` and `lastVerified`. This is the primary contributor reference and the gap is most visible.

2. **Wire `validate-ai-tools-registry.js --check` to `check-ai-companions.yml`.** Add as a step alongside the companion manifest check. This moves the registry from `rollout_state: migrating` to an active PR gate. Update `ownerless-governance-surfaces.json` accordingly.

3. **Write an adapter parity validator.** A script that reads `AGENTS.md`, extracts its core rule sections, and checks each adapter file for coverage of those rules. Add to `check-ai-companions.yml`. This enforces the baseline → adapter consistency contract.

4. **Add `llms.txt` generation.** `generate-llms-files.yml` exists — determine if it is wired and what it generates. If not wired, wire it. Create the `llms.txt` (and optionally `llms-full.txt`) at the repo root or standard path. Add documentation in `ai-features.mdx`.

5. **Fix `ai-tools.mdx` frontmatter.** Add the missing `---` delimiter. Add `status`, `lastVerified`, `pageType`, `audience` fields. (Same recommendation as tooling audit item 3 — the two audits converge on this fix.)

6. **Consider generating companions on PR branches.** For `check-ai-companions.yml`, add an optional step that generates companions for changed glossary pages in the PR branch (without auto-committing) and uses the result for the freshness check. This gives agents reviewing PRs accurate companion context.

## Upstream / downstream effects

**Changes upstream (affecting this concern):**
- Editing v2 glossary pages → triggers companion regeneration on push→main
- Adding an AI tool to `ai-tools/` → registry drifts until manually updated
- Changing core rules in `AGENTS.md` → adapters drift until manually updated (no parity check)

**Changes downstream (this concern affecting others):**
- `AGENTS.md` and adapters are read at the start of every agent session — stale or inconsistent instructions affect all AI-assisted work in the repo
- `v2/**/*-data.json` companion files are consumed by AI agents reading specific pages — stale companions mean agents have outdated page context
- `ai-tools-registry.json` is the source for what skills, templates, and packs exist — stale registry = agents unable to enumerate available AI tools
- `ai-features.mdx` being a stub means the AI layer is effectively undocumented for contributors — anyone trying to understand or extend it has no starting point
