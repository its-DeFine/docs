## Source of truth

**Canonical:** The `.github/workflows/*.yml` files themselves. Each workflow file is the authoritative source for its trigger conditions, steps, and behavior.

**Derived (generated):**
- `docs-guide/catalog/workflows-catalog.mdx` — inventory extracted from workflow files by `generate-docs-guide-indexes.js`

**Governance authority:**
- `docs-guide/repo-ops/maps/enforcement-map.mdx` — hand-maintained map of what enforces what
- `docs-guide/features/automations.mdx` — hand-maintained feature description

**Missing:**
- No machine-readable workflow registry (no equivalent of `script-registry.json` for workflows)
- No per-workflow inline documentation standard (no JSDoc equivalent for `.yml` files)
- No schedule index (which workflows run on cron and when)

---

## Gaps and issues

1. **`generate-docs-guide-indexes.js` handles both workflows and templates catalogs (mixed concern).** One generator produces two structurally distinct outputs: `workflows-catalog.mdx` (45 workflow entries) and `templates-catalog.mdx` (GitHub issue/PR templates only). This conflation means a change to either catalog's format requires changing the same generator, and CI wiring is entangled. The `generate-docs-guide-catalogs.yml` path filter includes both `.github/workflows/**` and `.github/ISSUE_TEMPLATE/**` — if one surface changes, both catalogs regenerate unnecessarily.

2. **Catalog metadata is shallow.** `workflows-catalog.mdx` lists workflow name, trigger, and a brief description derived from the `name:` field. It does not include: failure modes, dependent downstream workflows, deprecation status, schedule (for cron triggers), or ownership. For an agent trying to understand the CI system, the catalog is insufficient.

3. **Race condition between `generate-component-registry.yml` and `generate-docs-guide-catalogs.yml`.** Both trigger on push→main and both auto-commit to main. `generate-component-registry.yml` auto-commits when `snippets/components/**/*.jsx` changes. This auto-commit triggers `generate-docs-guide-catalogs.yml` (because it matches path filter for `.github/workflows/**` — no, wait: the cascade trigger is the component registry auto-commit touching the `docs-guide/` path which then triggers the other workflow). The actual race: if a jsx file changes on push→main, BOTH workflows fire simultaneously on the same commit, both checking out `ref: main`, potentially producing competing commits. This is documented in the components audit and is relevant here because the workflows are the site of the race.

4. **`enforcement-map.mdx` is hand-maintained with no freshness gate.** There are 43 workflows. The enforcement map describing which workflows gate which operations is written and maintained by hand. When workflows are added, renamed, or deprecated, the enforcement map does not update automatically. No CI validates that the enforcement map matches the actual `.yml` files.

5. **No deprecation tracking.** A `deprecated/` directory for workflows is not tracked in the catalog or enforcement map. Inactive or retired workflows that still sit in `.github/workflows/` are indistinguishable from active ones in the catalog output.

6. **`check-docs-guide-catalogs.yml` triggers on push to both `docs-v2` and `main`.** This means the PR check runs on push-to-main as well as PR, potentially double-checking on merge. Low impact but the trigger scope is broader than necessary — a `pull_request:` event is sufficient for a gate check.

7. **No per-workflow inline documentation standard.** GitHub Actions workflows have no equivalent of the JSDoc 11-tag standard used for scripts. There is no enforced `name:`, `description:`, or annotation standard in workflow YAML. Workflow intent is communicated only by the `name:` field and any comments in the file.

8. **43 workflows with no schedule registry.** Several workflows are cron-scheduled (e.g., `content-health.yml`, `freshness-monitor.yml`, `broken-links.yml`). There is no catalog of scheduled workflows, their cadences, or what they output. Agents and contributors cannot determine at a glance what runs automatically at what frequency.

## Recommendations

1. **Split `generate-docs-guide-indexes.js` into two generators.** Create `generate-workflows-catalog.js` and `generate-github-templates-catalog.js` with separate outputs. Each gets its own path filter in `generate-docs-guide-catalogs.yml`, eliminating unnecessary regeneration when only one surface changes.

2. **Add per-workflow metadata as structured YAML comments.** Define a lightweight annotation standard for workflow files: at minimum `# @description`, `# @trigger`, `# @outputs`, `# @downstream` comment fields. The generator can extract these to enrich the catalog. This mirrors the JSDoc approach for scripts without requiring a schema change to GitHub Actions syntax.

3. **Add a schedule index to the catalog.** Extend `generate-workflows-catalog.js` to detect `on.schedule.cron` fields and emit a separate schedule table in the catalog: workflow name, cron expression, human-readable cadence, what it generates.

4. **Wire `enforcement-map.mdx` to CI validation.** Write a validator that reads the enforcement map and cross-references it against actual workflow files. Fail the PR gate if the map references workflows that don't exist, or if workflows exist that are not in the map.

5. **Add a `deprecated:` annotation to inactive workflows.** Agree on a convention (e.g., `# @deprecated: true — kept for reference; not triggered`), have the generator emit deprecated workflows in a separate section of the catalog with a visual indicator.

6. **Resolve the race condition.** Consolidate `generate-component-registry.yml` and the downstream catalog generation into one sequential workflow so competing auto-commits cannot occur. This is the same fix recommended in the components audit; it must be implemented in the workflow layer.

## Upstream / downstream effects

**Changes upstream (affecting this concern):**
- Adding or renaming a workflow → `workflows-catalog.mdx` regenerates on push→main
- Changing workflow trigger paths → catalog entries change; enforcement-map may become inconsistent
- `generate-component-registry.yml` auto-commit → triggers `generate-docs-guide-catalogs.yml` (cascade)

**Changes downstream (this concern affecting others):**
- `workflows-catalog.mdx` is consumed by agents navigating CI — stale or shallow catalog = incorrect CI context
- `enforcement-map.mdx` is the primary reference for what gates what — stale map causes incorrect contributor guidance
- The race condition in this layer can produce competing commits, causing checkout failures or lost generation outputs for both component and catalog workflows
- Splitting the generator (recommendation 1) would require updating path filters in both `generate-docs-guide-catalogs.yml` and `check-docs-guide-catalogs.yml`
