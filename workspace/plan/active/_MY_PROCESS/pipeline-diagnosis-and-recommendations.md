## THE CORE PROBLEM

The pipeline has excellent frameworks but produces inconsistent results because **it is architecturally an open loop with no enforcement gates**. Gold-standard pages exist alongside pages that violate the pipeline's own rules. The same pipeline, the same prompts, the same voice rules — different quality every time.

**Root causes (in order of impact):**

1. **Bad audience/persona foundations contaminate everything downstream** — Phase 1 outputs were never validated. AI-generated "personas" are generic archetypes, not grounded in real reader behaviour. Every downstream phase inherits this weakness.

2. **No enforcement gates between phases** — a page can move from Phase 2 to Phase 6 without any checkpoint verifying that Phase 2 output was actually correct. The pipeline DESCRIBES gates but never BLOCKS on them.

3. **Voice and terminology rules exist but are not enforced at publish time** — banned phrases still appear in shipped pages. TODOs ship in published content. No CI gate catches these.

4. **The system is open-loop** — corrections found while writing Tab A are applied locally but never propagated back to the frameworks. Tab B starts at the same quality level as Tab A page 1.

5. **No exemplar-driven quality standard** — the pipeline tells agents what NOT to do (banned words, banned phrases) but never shows them what gold-standard output looks like. Agents need to see a finished page that passes every check, not just a list of rules.

### Pattern

The pipeline produces gold-standard output when:
- The audience is single and locked (gateway operator, orchestrator, delegator)
- The page type is operational (guide, instruction)
- An experienced human reviewed the output

The pipeline fails when:
- The audience is "general" or multi-audience
- The page type is conceptual (concept pages have the most voice violations)
- No human reviewed before publication
- Persona work was generic (AI-generated archetypes, not grounded in real Discord/GitHub/forum signals)

## WHAT'S MISSING: THE 6 GAPS THAT BREAK AUTONOMY

### Gap 1: No validated persona foundation

**The problem**: Phase 1 outputs (audience design) were generated by running a prompt across multiple AI systems and collating the results. The collated output defines personas like "Independent GPU operator" or "Product evaluator" — but these are derived from the prompt's structure, not from actual reader research.

**Evidence**: The site-wide persona research index notes that Orchestrators, Delegators, and Founders have NO sourced persona files. Only Gateways has Discord/GitHub/Messari-sourced archetypes. The canonical audience design for About defines 5 "personas" that are really 5 arriving audience types — not people the tab actually serves.

**What's needed**: A Check B validation gate that asks: "Are these personas grounded in actual reader behaviour, or are they AI-generated generics?" Check B should verify:
- Does each persona cite a real entry vector (Discord channel, search term, referral path)?
- Can you name a real person or community thread where this persona appeared?
- Is the persona defined by what they NEED FROM this tab, not where they CAME FROM?

**Impact if unfixed**: Every downstream phase inherits generic personas. Content written "for" an AI-generated persona sounds like AI-generated content. This is the single biggest quality lever.

### Gap 2: No enforcement gates between phases

**The problem**: The pipeline describes phase gates (Phase 1 must be approved before Phase 2 runs, tab-map must exist before content pass runs, etc.) but nothing BLOCKS progression. An agent can run Phase 4 (content pass) without a validated Phase 1 or a completed tab-map, and it will produce plausible output that looks correct but is built on sand.

**Evidence**: `tab-map.mdx` for Orchestrators was scored "8 — Known-good" in the first inventory audit. It does not exist. This hallucination proves the pipeline has no input validation.

**What's needed**: Each phase needs a pre-flight check that is NOT advisory but BLOCKING:
- Phase 2 cannot start without a human-approved Phase 1 output
- Phase 3/4 cannot start without a completed tab-map
- Phase 5 cannot start without all {/* REVIEW: */} flags either resolved or explicitly accepted
- No page ships with `{/* TODO: ... */}` comments — CI lint catches these

### Gap 3: No exemplar-driven quality standard

**The problem**: The pipeline tells agents what NOT to do (47 banned words/phrases, 15 structural anti-patterns, 10 quality criteria). But it never shows what gold-standard output LOOKS like. The agent reads rules and tries to follow them; but rules without exemplars produce technically-compliant-but-lifeless output.

**What's needed**: A curated set of 3-5 exemplar pages that are annotated with WHY they work:
- One concept page (About → protocol/overview.mdx is close)
- One instruction page (Gateways → setup/prepare.mdx)
- One guide page (Orchestrators → ai-inference-operations.mdx)
- One reference page (to be identified)

Each exemplar should have inline annotations: "This opening works because it leads with outcome, not description." "This paragraph succeeds because it has one job." "This heading works because it names the governing concept."

The content-pass prompt should load one relevant exemplar per page type before writing.

### Gap 4: Voice enforcement not automated

**The problem**: voice-rules.md is comprehensive (28-36 lines per audience, 7 audiences fully covered, UK English table, banned words/phrases). But enforcement is manual. "This page explains" still appears in published pages. Banned words still slip through. TODO comments ship.

**What's needed**: A pre-publish lint that runs automatically:
- Scans for all 10 banned words + 10 banned phrases
- Scans for `{/* TODO` and `{/* REVIEW` comments
- Scans for "This page/section/document covers/explains/walks"
- Scans for hedging constructions ("can help", "may provide", "allows you to")
- Fails the page if any are found — no advisory mode, BLOCKING

This is the `checks-pipeline.js --cat 2` that is PLANNED but not built. Until it exists, voice enforcement is honour-system.

### Gap 5: Feedback loop does not exist

**The problem**: Corrections found while writing Tab A pages are applied to those pages but never propagated back to the frameworks, voice rules, or phase prompts. Tab B starts from the same position as Tab A page 1. The system cannot learn.

**Evidence**: plan-canonical.md explicitly states: "The system is open-loop: Layer 6 (Feedback System) of canonical design is not in this plan."

**What's needed**: After completing each tab's content pass, a feedback synthesis step that:
- Captures all corrections made (voice violations found, terminology inconsistencies, structural patterns discovered)
- Updates voice-rules.md with any new banned phrases discovered
- Updates terminology lock with any new terms resolved
- Updates exemplar set if a new gold-standard page was produced
- Records pattern findings that should change the prompts for the next tab

### Gap 6: Veracity pass is specified but untested

**The problem**: veracity-pass.md exists (45KB prompt) but is explicitly flagged DRAFT. It has never been run against a known-good page. The 45-source veracity library exists but has no resolution logic (when two sources conflict, which wins?), no staleness rules (how old is "stale"?), and no parsing logic for {/* REVIEW: */} flags.

**What's needed**: Before scaling, run veracity-pass on ONE completed page (e.g., operator-rationale.mdx), compare output to expected result, confirm it flags claims correctly without over-flagging. Then document:
- Resolution logic: GitHub tagged release > whitepaper > forum post > blog
- Staleness rules: token price = real-time; API version = per release; economic model = annual; protocol parameter = governance-controlled (verify-live)
- Publication gate: pages can ship with `veracityStatus: unverified` only if ALL {/* REVIEW: */} flags are documented in a tracking issue

## RECOMMENDATIONS: HOW TO MAKE THE PIPELINE AUTONOMOUS

### Recommendation 1: Build the exemplar pack FIRST

Before fixing any pipeline infrastructure, create 3-5 annotated exemplar pages. These become the quality standard that everything else is measured against.

**Why first**: Every other fix (better personas, enforcement gates, voice linting) produces output that needs to be compared against something. Without exemplars, "good enough" is undefined.

**What to do**:
1. Select the best page from each of: concept, instruction, guide, reference (candidates identified above)
2. Create an annotated version showing WHY each paragraph, heading, and structure works
3. Store in `.claude/references/exemplars/` (or equivalent)
4. Update content-pass.md to load the relevant exemplar per page type before writing

### Recommendation 2: Rebuild persona foundations from real signals

The current persona work is prompt-derived. Replace it with signal-derived personas:
- Read Discord #local-gateways, #orchestrators, #staking channels for real questions
- Read GitHub issues in go-livepeer, livepeer-js, LIPs for real pain points
- Read Livepeer Forum threads for real governance participation patterns
- Read Messari/Dune for real delegator behaviour patterns

For each tab, define 2-3 personas that are named after real situations, not generic labels:
- Not "Independent GPU operator" → "The RTX 4090 owner who found Livepeer through a mining subreddit and wants to know if they can earn $50/day"
- Not "Token researcher" → "The analyst writing a Messari quarterly who needs the inflation formula and cannot find it"

### Recommendation 3: Build 3 blocking gates

**Gate 1 — Phase 1 output validation (Check B)**:
- Human reads personas and asks: "Would I recognise this person on Discord?"
- If no → iterate personas with real signals
- If yes → approve for Phase 2

**Gate 2 — Pre-content-pass validation**:
- Tab-map exists and is approved
- All Phase 1 personas have unblocked paths through the section structure
- Context block is fully filled (no empty PRIMARY_SOURCES)
- If any fail → block Phase 3/4

**Gate 3 — Pre-publish lint**:
- Zero banned words/phrases
- Zero `{/* TODO` comments
- Zero `{/* REVIEW` comments without tracking issue
- All frontmatter fields populated with canonical values
- If any fail → block publication

### Recommendation 4: Close the feedback loop

After completing each tab:
1. Run a retrospective: what voice violations were found? What terminology was resolved? What structural patterns emerged?
2. Update voice-rules.md, terminology lock, and exemplar set
3. Record in a `pipeline-learnings.md` file per tab
4. Next tab's Phase 1 loads previous tab's learnings

### Recommendation 5: Build the voice linter (checks-pipeline.js --cat 2)

This is the most impactful automation. A script that:
- Reads every MDX file in a tab
- Checks against banned words (10), banned phrases (10), self-referential patterns (5), hedging constructions (5)
- Outputs a structured report: file, line, violation, suggested fix
- Can run in CI as a pre-commit hook

This exists as PLANNED in script-plan.md. Building it immediately provides automated enforcement for the single most common quality failure (voice violations).

### Recommendation 6: Build persona-level voice rules

Current voice-rules.md provides audience-level rules (7 audiences). But within each audience, personas have different registers:

- **Orchestrator audience**: a prospective solo miner needs different language than a returning operator tuning AI pipelines
- **Developer audience**: a hackathon participant needs different density than a protocol contributor
- **Delegator audience**: a first-time staker needs different complexity than a governance participant

For each tab's primary personas, add 3-5 lines of persona-specific voice guidance to voice-rules.md. This is the difference between "technically correct but generic" and "feels like it was written for me."

## THE GOLD-STANDARD PAGE TEST

A page passes the gold-standard test if:

1. **Opening line** — leads with what the reader gets or needs to know, not what the page covers
2. **Single audience** — one persona, named in frontmatter, consistent register throughout
3. **One job** — the page does one thing and does it completely
4. **Entity-led voice** — every paragraph starts with a system fact or reader outcome, never "this page" or "this section"
5. **No hedging** — no "can help", "may provide", "allows you to" for value claims
6. **No banned words** — zero instances of effectively/essentially/basically/meaningful/significant/various/obviously/clearly
7. **No banned phrases** — zero instances of "This page covers/explains", "It is important to note", "As mentioned above"
8. **Paragraph discipline** — each paragraph has one job and ends with a fact, number, or next step
9. **Terminology locked** — every Livepeer-specific term matches the tab's terminology lock
10. **No editorial debt** — zero `{/* TODO` or `{/* REVIEW` comments without tracking issue
11. **Frontmatter complete** — all canonical fields populated with canonical values
12. **Next step clear** — reader knows where to go next

If a page fails ANY of these, it is not gold-standard and should not ship.

## WHAT "AUTONOMOUS" MEANS

The pipeline is autonomous when:
- A human defines the persona (from real signals) and approves the IA
- An agent runs Phases 2-6 without human intervention
- A pre-publish lint catches all voice/terminology/structural violations
- A human reviews only the final output and either approves or sends back with specific corrections
- Corrections from Tab A automatically improve Tab B's starting position

The human's role becomes: persona definition, IA approval, and final sign-off. Everything else is automated and enforced.
