## Issues Found

### CRITICAL — Fix before next run

**1. LIP-92 test is untriggerable with the current input pack**

The input pack `veracity-sources.md` lists LIP-92 as a primary source for "formal protocol changes" but does not state what LIP-92 covers. It reads: `| Formal protocol changes — LIP-1, LIP-73, LIP-89, LIP-91, LIP-92 | [LIPs repo] | primary |` — no definition.

The glossary says: LIP-92 = "AI model registry and capability discovery mechanism." The veracity-sources.md doesn't say anything about LIP-92's content — only that the LIPs repo is authoritative. This means there is no contradiction for the source conflict rule to catch. A model following the rule correctly would see: glossary says X, veracity-sources doesn't disagree, no conflict.

Result across three runs:
- Run 1 and 2: LIP-92 never appears in the lock or output — not triggered at all
- Perplexity: explicitly checked both files, concluded "no conflict — consistent" (actually misread veracity-sources as confirming the glossary entry)

The test cannot work unless the veracity-sources.md explicitly names what LIP-92 covers. The conflict rule in the prompt is correctly written, but the input pack doesn't create the conflict it's supposed to test.

**Fix (prompt, not input pack)**: The input pack is correct. `veracity-sources.md` already names LIP-92 directly in the LIPs repo row: `| Formal protocol changes — LIP-1, LIP-73, LIP-89, LIP-91, LIP-92 | [LIPs repo] | primary |`. A model reading this correctly should recognise: the LIPs repo is the primary source for LIP-92; the glossary's definition is unverified against that source; flag as `[verify-against: https://github.com/livepeer/LIPs]`. The prompt's source conflict rule only told models to look for explicit contradictions — it did not say "absence of a contradiction is not confirmation." Updated in v3 post-run: rule now states that any glossary definition referencing a specific external artifact (a LIP, a contract, a CLI flag) must be marked `[verify-against: {URL}]` if veracity-sources.md lists a primary source for it, regardless of whether an explicit counter-claim exists.

### HIGH — Fix in next iteration

**2. Disambiguation section pageType is unspecified — produces 3 different tokens across 3 runs**

The v3 prompt says the disambiguation section must have `purpose: orient`. It does not specify `pageType`. Result:
- Run 1: `pageType: guide`
- Run 2: `pageType: concept`
- Perplexity: `pageType: navigation`

All three are plausible reads of "routing section." `navigation` is technically correct (pageType = "Routes the reader"), but the prompt doesn't say this. Inconsistency here compounds in downstream phases when the section structure is used to design pages.

**Fix**: Add to the disambiguation section rule in Phase 2C: *"The dedicated disambiguation section has `purpose: orient` and `pageType: navigation` — it routes the reader to the correct path, it does not explain or teach."*

---

**3. Section 2 design load — flagged but not structurally resolved**

All three runs correctly flagged Section 2 in the Addendum. However, the flag is produced as a note and then ignored — the output section table still has Section 2 carrying both evaluation and routing work. The flag instruction says "do not split here" which is correct, but gives no instruction on what the downstream design process should do with this flag.

The flag will be invisible to whoever picks up the output later unless they read the Addendum. It needs to appear in the section table itself, not just the Addendum.

**Fix**: Add to Phase 2C section load check instruction: *"When flagging a section as overloaded, also add a `[⚠ Design flag: may need to split]` note directly in the relevant row of the section structure table — in addition to the Addendum flag."*
