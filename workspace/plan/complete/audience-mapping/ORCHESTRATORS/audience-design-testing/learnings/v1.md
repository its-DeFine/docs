## Issues Found

### CRITICAL — Fix before next run

**1. Cross-tab gate vs self-containment principle directly contradict each other**

Flagged independently by: Claude web, ChatGPT

The brief states both: (a) "each tab is self-contained — every piece of information a reader needs must be present, intentional duplication is correct" and (b) "if another tab owns this content, replace the row with [CROSS-TAB: links, does not duplicate]."

These rules directly conflict. ChatGPT applied the gate too strictly and produced cross-tab rows where the reader needed inline content to proceed. Claude web noted the tension and made a judgement call. Different models will resolve this differently, producing inconsistent outputs.

**Fix**: Replace the cross-tab gate instruction with: *"If another tab is the canonical deep reference, include the minimum decision-enabling content the reader needs to proceed, then link for depth. Do not omit content this audience requires to complete their goal. Only replace a section with a CROSS-TAB row when this tab's audience does not need that content at all to achieve their goal in this tab."*

---

**2. Deprecated term appeared in TERMINOLOGY_LOCK**

Flagged by: review of Claude web output

Claude web included "Pool Worker" in the TERMINOLOGY_LOCK despite `deprecated-terms.md` explicitly replacing it with "Pool node." The brief tells the AI to load `deprecated-terms.md` but does not instruct it to apply those replacements during Phase 0 term selection.

**Fix**: Add to Phase 0 Step 0.2: *"Before finalising the TERMINOLOGY_LOCK, check each term against deprecated-terms.md. Replace any deprecated term with its current equivalent."*

---

**3. TERMINOLOGY_LOCK 5–10 cap too tight for large-glossary tabs**

Flagged by: Claude web (hit 19 before applying cap), Claude Sonnet agent (produced 15)

The Orchestrators glossary has 123 terms with Livepeer-specific meanings. Both Claude runs exceeded the cap before pruning. The cap forces the AI to discard genuinely load-bearing terms (e.g. `reward cut`, `fee cut`, `performance score`, `O-T split`) which then appear in later phases without having been anchored. ChatGPT stayed within the cap but produced a thinner lock (9 terms).

**Fix**: Raise to 10–20 terms. Or split into two tiers: *Core lock* (5–10 terms used to constrain Phase 0 reasoning) and *Extended lock* (up to 20 terms available in Phases 1–4 without requiring fresh introduction).

---

### HIGH — Fix in next iteration

**4. "Secondary navigation page" is undefined in the universal elements exclusion**

Flagged by: Claude web

The Phase 2C exclusion rule lists "portal/landing page, secondary navigation page, resources/compendium section." "Secondary navigation page" has no definition anywhere in the brief, is not in the pageType enum, and is not an obvious concept. One AI flagged it as confusing; others may silently design content that belongs there.

**Fix**: Replace "secondary navigation page" with a concrete description (e.g. "a section-level index or 'Getting started' landing that routes into sub-sections of the tab") or remove the term entirely.

---

**5. Terminology permissions after Phase 0 are underspecified**

Flagged by: ChatGPT

The brief says the per-tab glossary is permitted "in Phase 0 only." But the TERMINOLOGY_LOCK generated in Phase 0 is meant to be used in all later phases. There's no explicit statement of what's permitted in Phases 1–4. One AI inferred correctly; another was uncertain.

**Fix**: Add after Phase 0: *"After Phase 0, permitted terms are: (a) terms in the TERMINOLOGY_LOCK, (b) terms in the Product Context block, and (c) plain-language terms introduced fresh without assuming prior knowledge. The per-tab glossary is not consulted again after Phase 0."*

---

**6. "Blocked at" column conflates two different types of block**

Flagged by: Claude web

The persona path validation table has a single "Blocked at" column, but there are two distinct block types: (a) structural blocks (a required section doesn't exist — fix: add a section) and (b) knowledge blocks (the persona doesn't understand a concept introduced in a section — fix: add content within a section). These require different remedies and should be distinguishable in the output.

**Fix**: Split "Blocked at" into two columns: *"Structural block"* (missing section) and *"Knowledge gap"* (section exists but needs more content for this persona).

---

**7. Checkpoint has no failure path**

Flagged by: Claude web

The checkpoint is a self-attestation — the AI marks its own boxes with [x]. There is no instruction for what to do if a gate fails. In practice, all three runs checked all boxes regardless. A failing gate should produce a correction request, not a rubber-stamp.

**Fix**: Add after the checkpoint: *"If any gate is unchecked, do not produce the output document. Instead, list the failing gates and the specific correction required before the output can be produced."*

---

### LOWER — Consider for future iteration

**8. "Volume" ranking for personas is unverifiable from prompt inputs**

Flagged by: Claude web

The instruction "Rank by volume (most common first)" requires analytics data not provided in the prompt. All models inferred volume from general reasoning about GPU operators — which is exactly the kind of training-data knowledge the brief prohibits. The ranking ended up broadly consistent across runs, but for less well-known audiences it will diverge.

**Fix**: Either remove the volume requirement and replace with "rank by dependency/journey sequence" or explicitly permit: *"In the absence of analytics data, rank by estimated volume based on the Product Context block only — state the assumption."*

---

**9. "Types" in Phase 2B on-demand is underspecified**

Flagged by: Claude web

"List 4–8 types" for on-demand needs doesn't define what a "type" is. The three runs interpreted it as: reference categories (correct intent), but the term is ambiguous. The 4–8 cap is also too narrow for complex tabs — all runs had more than 8 valid on-demand needs.

**Fix**: Replace "List 4–8 types" with "List each category of specific information this audience returns to look up after completing the linear journey. Use the format: *[what they're looking for] — e.g. 'Current CLI flag names and default values'.* Aim for 6–10."*

---

**10. guide vs concept distinction needs examples**

Flagged by: Claude web

The pageType enum defines `concept` as "Explains a mechanism or system" and `guide` as "Practical understanding of a system." The distinction is thin and produced inconsistent usage — some sections that should be `guide` were marked `concept` and vice versa.

**Fix**: Add one concrete example to each enum entry in the Framework Enums section. E.g. `concept: how probabilistic micropayment works; guide: how to set pricing for profitability`.

## Section Structure Observations (content quality, not prompt issues)

- All three produced a **routing/disambiguation section as the first content section** — confirmed correct
- Claude web produced **3 separate quickstart sections** (solo, AI-only, pool worker) — higher specificity than the other two, which had path-branching within a single quickstart
- All three included a **"staying current"/"changelog" type section** — suggests operators have a strong return-visit pattern around protocol changes
- Claude web uniquely included **"Understanding your performance score"** as a standalone section — none of the others did. This may be a genuine gap in the other two outputs
- ChatGPT's arriving question was notably more complex: *"Given my hardware, stake position, and operating goal, what is the right way to join Livepeer supply and get to my first paid work without taking the wrong path?"* — captures the multi-variable decision better than the simpler formulations in the other runs
