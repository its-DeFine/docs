## 8 Root Causes for Session Loss (Community-Identified)

1. **sessions-index.json not updated** — Extension relies on an index file instead of scanning JSONL files directly. Since ~v2.1.30, it stopped being populated reliably.

2. **64KB head/tail buffer limit** — `fetchSessions()` reads only first/last 64KB of each JSONL. Large sessions (images, long messages) have title data outside this window and are silently dropped.

3. **deserializeWebviewPanel ignores sessionId** (#35022) — On restart, the panel receives the saved session ID but passes `undefined` to `setupPanel()`. All restored tabs get wrong/blank content.

4. **600-second webview timeout** (#35005) — Webview discards saved session IDs if the panel was idle >10 minutes before VS Code was closed.

5. **countConcurrentSessions() deletes session files** (#38528, #35367) — Concurrent session counter uses `parseInt` on filenames and `unlink`s files it considers stale, destroying legitimate data.

6. **Cross-version format incompatibility** — Updates introduce new required fields without migrating existing data, making prior sessions invisible.

7. **Path resolution mismatches** — Case-sensitivity, network drives, worktrees, submodules, and directory-scoped binding all break session discovery.

8. **Cross-project cache contamination** (#22215) — Extension caches globally rather than per-project, causing wrong sessions to appear or none at all.
