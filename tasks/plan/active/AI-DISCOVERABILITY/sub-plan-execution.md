## Item 1 — Where do companion `.json` files go?

Two types of companion files, two different homes.

### Tier 1: External data snapshots (CoinGeckoExchanges, MarkdownEmbed)

**Proposed location**: `snippets/data/snapshots/`

```
snippets/
└── data/
    ├── about/              ← existing page-data folders
    ├── developers/
    ├── ...
    └── snapshots/          ← NEW — external API snapshots only
        ├── coingecko-livepeer.json
        └── embed-[url-hash].json
```

**Why `snippets/data/snapshots/` and not adjacent to MDX:**
- Snapshot data is not page-specific — the same CoinGecko data could be used on multiple pages
- External snapshots are regenerated by a cron/CI job, not by the author writing a page
- Separating them from props-extracted files makes the regeneration target clear
- `snippets/data/` is already the established home for non-component data in this repo

**Naming convention**:
- `coingecko-[coinId].json` — one file per coin identifier
- `embed-[url-hash].json` — content + source URL, hash = first 8 chars of SHA256 of the URL

---

### Tier 2: Props-extracted data (SearchTable, ShowcaseCards, CardCarousel)

**Proposed location**: Adjacent to the MDX page that uses the component, in `v2/`

```
v2/
└── developers/
    ├── glossary.mdx                    ← page using <SearchTable data={[...]} />
    └── glossary-data.json              ← companion — full unfiltered data array
```

**Why adjacent (not central):**
- An AI agent, LLM, or crawler traversing a directory sees the `.mdx` + `.json` as a natural pair
- The relationship is obvious without cross-directory resolution
- Follows the same convention used for co-located assets elsewhere in the repo

**Naming convention**: `[page-slug]-data.json` — same slug as the MDX file, with `-data` suffix.

**Note**: These files do not exist yet. They are written at page-authoring time (writer/agent obligation), not by the generation script. The generation script (CDA-5) will eventually auto-extract them from MDX props — until then it is a manual step documented in CONTENT-WRITING/plan.md.

## Item 2 — Where does the generation script go?

**Proposed location**: `tools/scripts/generators/ai/generate-ai-companions.js`

```
tools/
└── scripts/
    └── generators/
        ├── components/
        │   ├── documentation/     ← existing
        │   ├── library/           ← existing
        │   └── discoverability/   ← NOT here (see rationale below)
        ├── governance/            ← existing
        └── ai/                    ← NEW
            └── generate-ai-companions.js
```

**Why `generators/ai/` not `generators/components/discoverability/`:**
- The script covers any page in `v2/` that uses a hook-based component — it is not scoped to the component library itself
- Future AI-layer scripts (context packs, agent resource generators, AI companion manifests) should live alongside it under `generators/ai/`
- `generators/components/` is for scripts that operate on `snippets/components/` — this script operates on `v2/` pages

**Script responsibilities (no execution — reference only):**
1. Read `docs-guide/component-registry.json` for components with `@aiDiscoverability: snapshot | props-extracted`
2. For `snapshot` components: call external API, write to `snippets/data/snapshots/`
3. For `props-extracted` components: parse MDX files that import the component, extract data prop, write `[page-slug]-data.json` adjacent to the MDX
4. Write/update manifest at `docs-guide/catalog/ai-companion-manifest.json`
5. `--check` mode: verify all companions exist and are not stale

**Handoff flag**: Snapshot generation (step 2) requires outbound network. Recommend CI step only. Props extraction (step 3) is safe locally.

### New section to add after "Removed tags"

```markdown
### @aiDiscoverability values

Required on any component that uses React hooks (`useState`, `useEffect`, `useRef`, etc.).
Not required on pure/presentational components.

| Value | Meaning | Companion file location |
|---|---|---|
| `snapshot` | Component fetches external data at runtime — content invisible to crawlers | `snippets/data/snapshots/[source-id].json` |
| `props-extracted` | Component renders props-driven data through interactive UI (filter/paginate/search) — full data set not visible in static HTML | `v2/[section]/[page-slug]-data.json` (adjacent to MDX) |
| `none` | Component uses hooks for UI state only — no content is hidden from crawlers | No companion needed |
```
