## Problem Statement

Mintlify custom components render **client-side only**. Any component that uses React hooks (`useState`, `useEffect`, etc.) produces content that does not appear in the initial static HTML. For an AI-first documentation site, this is a critical discoverability gap:

- AI agents parsing page HTML see empty containers where content should be
- Search crawlers cannot index filtered/paginated data
- LLM training data misses content entirely
- Agent-based doc traversal (e.g., LPD CLI, context loaders) cannot read the content

The trigger was discovering that `SearchTable` — the recommended component for the glossary page — renders its rows entirely client-side. Glossary terms would be invisible to any AI reading the page.

**This is not a niche concern: it applies to 9 components across the entire component library.**

## Architecture

### Principle

> Every component that hides content from static HTML must have a static, machine-readable counterpart.

The counterpart is the canonical data source. The component is the display layer. The MDX page renders both.

### Three mechanisms

#### Mechanism 1 — Static snapshot JSON (Tier 1: externally fetched data)

Generated at deploy time or on a cron. Script calls the API and writes the response + metadata to:

```
snippets/data/snapshots/coingecko-[coinId].json
snippets/data/snapshots/embed-[url-hash].json    ← content + source URL
```

These files are:
- Committed to the repo (stale-acceptable — AI reads the snapshot, not the live feed)
- Regenerated by script (cron or CI step)
- Linked from the MDX page via a `<a href>` or hidden `<link>` to make them discoverable

#### Mechanism 2 — Extracted props JSON (Tier 2: props-driven data)

The raw data already exists in MDX props. A script parses MDX files, extracts the data prop from component invocations, and writes it as a companion file adjacent to the MDX:

```
v2/developers/glossary.mdx              ← <SearchTable data={[...]} />
v2/developers/glossary-data.json        ← full unfiltered data array, auto-extracted

v2/showcase/index.mdx                   ← <ShowcaseCards items={[...]} />
v2/showcase/index-data.json             ← full items array, auto-extracted
```

These files are:
- Auto-generated (pre-commit or CI)
- Never hand-edited
- Linked from the MDX page

#### Mechanism 3 — Governance flag in JSDoc (enforcement)

New field added to component JSDoc for hook-using components:

```
@aiDiscoverability snapshot | props-extracted | none
```

- `snapshot`: requires a companion generated from an external source
- `props-extracted`: requires a companion extracted from MDX props
- `none`: Tier 3 — no content hidden, no companion needed

The registry captures this field. Pre-commit checks that any component flagged `snapshot` or `props-extracted` has a corresponding companion manifest entry.

### Companion manifest

```
docs-guide/catalog/ai-companion-manifest.json
```

Lists every companion file: source component, target MDX pages, companion path, last-updated timestamp, mechanism type. This is the registry's machine-readable index of all AI discoverability coverage.

- Snapshot caching: compare response hash before overwriting (avoid unnecessary churn)
- `--check` mode: verify companions exist and are not older than N days without regenerating

## Open Questions

1. **Naming convention for companion files**: `[page-slug]-data.json` adjacent to MDX, or a central `snippets/data/` directory? Adjacent is more discoverable (AI agents traversing a directory see the pair). Central is more maintainable (one location to audit).
   - **Recommendation**: adjacent to MDX for Tier 2 (discoverability), central `snippets/data/snapshots/` for Tier 1 (external snapshots).

2. **Link mechanism from MDX**: How should the companion be linked for AI discoverability? Options:
   - HTML `<link rel="alternate" type="application/json" href="./glossary-data.json">` in page head (not possible in Mintlify without custom layout)
   - `<a href="./glossary-data.json" style="display:none">` (visible to crawlers, hidden from users)
   - Mintlify `<Note>` block with link (user-visible, crawler-visible)
   - **Recommendation**: `<Note>` block with data download link — honest, discoverable, user-useful.

3. **Staleness policy for snapshots**: How old can a snapshot be before it's a governance failure?
   - CoinGecko data: 7 days acceptable (exchange listings don't change hourly)
   - **Recommendation**: 7-day max for external snapshots, regenerate on deploy.
