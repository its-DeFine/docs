### Notes

**Two-layer purpose translation.** `frontmatter-taxonomy.js` defines 15 canonical purpose values. `rubric-loader.js` defines a separate 10-value `PURPOSE_ENUM` for internal rubric scoring, using pre-Phase-1 labels. `purposeToRubricPurpose()` bridges them. Any script that reads `purpose` from frontmatter directly against `PURPOSE_ENUM` (rather than normalising through `frontmatter-taxonomy.js` first) will silently fail on canonical values like `orient`, `choose`, `evaluate`, `integrate`, `optimise`, `verify`, `update`.

**Audience token split.** The canonical 7-token audience set (`founder`, `builder`, `developer`, `gateway`, `orchestrator`, `delegator`, `community`) defined in `Frameworks.mdx` does not match any existing script enum. `rubric-loader.js` and `usefulness-audience-normalization.json` both use a 9-token set built around `gateway-operator`, `platform-builder`, `general`, `everyone`, `internal`. No migration path exists yet. Scripts that write audience tokens to frontmatter will write old-schema values.

**`overview` is not aliased.** `frontmatter-taxonomy.js` explicitly excludes `overview` from `DEPRECATED_PAGE_TYPE_ALIASES` with a comment stating pages must be explicitly migrated. But `add-pagetype-mechanical.js` rule 10 assigns `overview` as a pageType to `overview.mdx` and `index.mdx` files, then passes it through `normalizePageType()`. This will result in an invalid canonical: `normalizePageType('overview')` returns `valid: false` — causing a thrown error at line 281.

**No script checks `veracityStatus`.** The field added to all 70 orchestrator pages during the 2026-03-24 content pass has zero script coverage. No validator, audit, or remediator reads, validates, or enforces it.

## CHECK & REMEDIATION GAPS

### Script Coverage Gaps

The following frontmatter checks do not exist in any script in the repo but should, based on the audit above.

**`pageType` value validation**
No script enforces that `pageType` is a canonical value from the 7-type set. `lint-structure.js` requires the field to be present but does not check its value. A page with `pageType: quickstart` (deprecated) passes the validator without any warning.

**`audience` value validation against the canonical 7-token set**
No script validates that `audience` matches the current canonical set (`founder`, `builder`, `developer`, `gateway`, `orchestrator`, `delegator`, `community`). `lint-structure.js` requires the field but does not check its value. `rubric-loader.js` and `usefulness-audience-normalization.json` validate against the old 9-token set (`gateway-operator`, `platform-builder`, `general`, `everyone`, `internal`) — these tokens are not canonical.

**`purpose` value validation against the 15-value canonical set**
No script in the active pipeline validates that `purpose` is a canonical value from the 15-value set (`orient`, `explain`, `learn`, `choose`, `evaluate`, `start`, `build`, `configure`, `operate`, `troubleshoot`, `verify`, `integrate`, `optimise`, `reference`, `update`). `assign-purpose-metadata.js` writes old-schema labels directly. `rubric-loader.js` validates against the rubric-internal 10-value set only.

**`veracityStatus` presence and value check**
No script checks for the presence or validity of `veracityStatus`. The field is present on orchestrator pages from the 2026-03-24 pass but is entirely unguarded — no validator flags pages where it is missing or set to an unrecognised value.

**`pageType`/`purpose` combination validation**
`frontmatter-taxonomy.js` defines `PAGE_TYPE_ALLOWED_PURPOSES` mapping each pageType to its permitted purpose values. No script enforces this mapping. A page with `pageType: navigation` and `purpose: build` is invalid per the taxonomy but will pass every existing validator.

**Deprecated audience token detection**
No script flags `audience: gateway-operator` or `audience: platform-builder` as deprecated. Pages using old audience tokens from the `assign-purpose-metadata.js` remediator will silently persist with invalid values. No migration remediator exists for audience tokens.

**`overview` pageType classification error in `add-pagetype-mechanical.js`**
Rule 10 in `classifyFile()` assigns `overview` to `overview.mdx` and `index.mdx`. `normalizePageType('overview')` returns `valid: false` because `overview` is explicitly excluded from the alias map. This will throw at line 281 for any file matched by rule 10. No test covers this path. Field: `pageType`. Validation: absent. Risk: runtime error on classification run.

**`lastVerified` format validation**
`lint-structure.js` parses `lastVerified` as a `Date` but does not validate the format. An invalid date string (e.g. `lastVerified: TBD`) will produce `NaN` for `daysDiff`, which is `> 90` evaluates to false — the stale check silently passes. Field: `lastVerified`. Validation: absent for format errors.

**Absence of `purpose` from `lint-structure.js` required fields**
`REQUIRED_FRONTMATTER` in `lint-structure.js` is `['title', 'pageType', 'audience', 'lastVerified']`. `purpose` is not required. Pages without a `purpose` field pass the validator. This is the field the entire rubric scoring system runs on.

**Deprecated pageType aliases written by `add-pagetype-mechanical.js`**
The script successfully normalises deprecated aliases to canonical values before writing — but its own `SUMMARY_TYPES` counter array and summary printout use the deprecated label names, not the canonical ones. This means the summary output is misleading and the counter keys (`landing`, `quickstart`, etc.) will never increment correctly because the written values are canonical (`navigation`, `instruction`, etc.).
