A Livepeer Gateway is the entry point for your application into the Livepeer decentralized compute network.
It handles job routing, orchestrator selection, payment, and service-level logic - so your app doesn't have to.

There are two ways to access a gateway:

- **Use a hosted gateway** - connect to an existing provider's gateway via API or endpoint. No infrastructure to run.
- **Run your own gateway** - self-host a `go-livepeer` gateway node for full control, custom pricing, and direct network access.

  <Step title="How to evaluate performance">
  <Tip>
    The Livepeer Explorer is adding a Gateway view - you'll be able to compare active gateways
    by throughput, reliability, and orchestrator relationships directly. Coming soon.
    <br />
    A preview is available at: https://explorer-arbitrum-one-git-feat-add-g-10dba1-livepeer-foundation.vercel.app/gateways
  </Tip>

  Until the [Explorer](https://explorer.livepeer.org/gateways) gateway view is live, evaluate providers on:

  **Latency** - For real-time AI (Daydream-style pipelines), end-to-end frame latency matters more than throughput. Ask about p50/p95 inference times.

  **Orchestrator pool quality** - A gateway is only as good as the orchestrators it routes to. Ask which orchestrators a provider uses, or check `/getNetworkCapabilities` on a self-hosted node to inspect the pool directly.

  **Model availability** - Not all gateways support all AI models. Verify that the pipelines you need (e.g. `text-to-image`, `image-to-video`, ControlNet, ComfyStream) are available before committing.

  **Pricing transparency** - Hosted providers bundle gateway margin into their pricing. Self-hosting lets you pay orchestrators directly and inspect pricing per-model with `maxPricePerCapability`.

  **Reliability** - Look for uptime history, failover behaviour, and whether the provider publishes SLA commitments.
