# Content, Copy, Voice, Style & Layout — Consolidated Audit

**Scope**: Full repo audit of all documentation guidelines, copy rules, voice registers, styling standards, layout contracts, terminology governance, and page structure rules.
**Date**: 2026-03-29
**Sources scanned**: 140+ files across `docs-guide/`, `workspace/plan/`, `ai-tools/`, `snippets/`, `v2/`, `tools/config/`

---

## Summary

### Voice & Tone
- 7 audience-specific voice registers defined and locked: gateway, orchestrator, developer, builder, delegator, community, founder
- Universal banned words: effectively, essentially, basically, meaningful, significant, real, various, several, obviously, clearly
- 40+ banned phrases codified — "This section covers", "Understanding X is essential", "It is important to note", and 37 more
- Entity-led voice required: subject is system/node/user — never the documentation itself
- Exit-state over topic: headings name what reader can do, not what section is about
- UK English enforced throughout (organise, colour, behaviour, centre, licence/license)
- Opening order: value/outcome before mechanism, fact before caveat, reader benefit before system description

### Copy Rules
- Three-tier enforcement: blocking (merge blockers), warning-only (rewrite by default), manual sign-off (human decides)
- Master test: "Does this sentence give the reader something they can act on?"
- One paragraph = one job; lead sentence states fact; final sentence ends on fact/number/next step — never a hedge
- Banned constructions: `not [X]` in value statements, `if [condition]` in body prose, `This page [verb]`, `can/may [verb]` in value claims
- Description field: one-sentence outcome statement, not table of contents, under 160 characters
- Comparative framing banned as entry point — define the thing before comparing it
- `{/* REVIEW: */}` flags block merge if unresolved in decision-critical positions

### Page Structure & Layout
- 12 page types governed: landing, overview, guide, quickstart, how_to, reference, tutorial, concept, resource, faq, troubleshooting, glossary, changelog
- Each type has required sections, preferred components, and components to avoid
- Page composition: Header (frontmatter + optional CTA) → Introduction (overview) → Body (H1+ sections) → Footer (optional related pages)
- Anti-patterns: no more than one major layout element (Table, Steps) per page unless nested; no duplicate layout elements
- Section headings: governing-concept descriptors only — no questions, no literal labels, no numbers, no comparators, under 3 words
- Related Pages footer: Card layout, 1-3 word title, one sentence max 10 words description

### Styling & UI
- CSS Custom Properties only — JavaScript theme objects (ThemeData) deprecated
- Colour system: 9 CSS variables with light/dark mode values, primary accent #3CB540 (Jade Green)
- No hardcoded colours, no inline style objects with literal hex values
- 50+ governed JSX components across 5 categories: elements, wrappers, displays, integrators, scaffolding
- Component registry and usage map maintained in `docs-guide/config/`

### Terminology
- 480+ terms across 9 tab-specific glossaries (home, about, solutions, developers, gateways, orchestrators, LP token, community, resources)
- Consolidated master glossary with cross-tab matrix and external verification links
- First-use jargon rule: define coined terms before they appear in headings, tables, or diagrams
- Section naming rubric locked: name the governing concept, not the examples it contains

### Content Architecture
- 3 content layers: Product/Positioning → Operational How-to → Deep Reference
- Role-based IA: Home/About (orientation), Community (participation), Developers/Gateways/Orchestrators/Delegators (role workflows), Resources (references)
- Publishable vs workspace distinction enforced via folder governance (`_workspace/` never routed)
- 15 purpose values defined (orient, explain, choose, start, build, configure, operate, troubleshoot, optimise, integrate, verify, compare, reference, learn, update)

### Quality & Governance
- Pre-commit validators enforce structure, component layout, and frontmatter compliance
- Quality gates: P0/P1 blocking, P2 advisory/blocking, P3 advisory
- Copy review packets produced for each tab (Gateway, Orchestrator — dated 2026-03-16/17)
- Research-backed framework sourced from 6 external standards: Diataxis, Google Technical Writing, Microsoft Style Guide, Red Hat Modular Docs, DITA, public SKILL.md standards

### Good Page Patterns Observed
- Entity-led voice: systems and users as subjects, never the documentation
- Concrete numbers, dates, amounts, specifications
- Decision trees and comparison tables
- Governing-concept headings, professional terminology, under 3 words
- Proper component usage per page-type contract

### Universal Rules (All Audiences, Locked)

**Entity-led voice** — The subject is the system, node, user, or action. Never the documentation.
- ✅ "The orchestrator submits a reward call each round"
- ❌ "In this section, we explain how orchestrators submit reward calls"

**Exit state over topic** — Section heading/opening names what reader can do, not what section covers.
- ✅ "Choose an orchestrator based on reward cut, uptime, ETH fee history"
- ❌ "Orchestrator Selection Criteria" (as heading above vague body prose)

**Precision over breadth** — Say exactly what is true. If the number is unknown, flag it.
- ✅ "Unbonding period is 7 rounds (~7 days)"
- ❌ "Unbonding typically takes several rounds, which may vary"

**No forward-looking uncertainty about shipped features** — If it is live, state as fact.
- ✅ "Governance proposals require 33% quorum to pass"
- ❌ "Governance proposals may require quorum depending on settings"

**Opening order** — Value/outcome before mechanism; fact before caveat; reader benefit before system description.

**Paragraph discipline** — One paragraph, one job. Lead sentence states fact. Final sentence ends on fact, number, or next step. Never a hedge.

### Per-Audience Voice Registers (7 Registers, Locked)

**Gateway** — Peer-to-peer technical. Operator-to-operator. Senior engineer to senior engineer. Lead with gateway output (earnings, routing performance). Use concrete numbers. Assume networking/CLI/gas competence. State commands imperatively. Prohibited: "Don't worry, this is easy to configure" / "As a gateway operator, you will..." / "This document will guide you"

**Orchestrator** — Operational, earnings-forward, hardware-aware. Lead with earnings, performance, operational outcomes. Use hardware-specific language (VRAM, CPU cores, NVLink, bandwidth). Quantify everything: "8 GB VRAM minimum for SDXL, 24 GB for high-res". Distinguish paths by hardware profile. Prohibited: "Your orchestrator will automatically..." / "The network rewards you for..." / "Easy to set up"

**Developer** — Code-first, minimal prose. Code example first, prose explanation second. Function signatures, types, interfaces are primary content. Reference actual codebase with GitHub links. Be explicit about versions, breaking changes. Include error states and edge cases in main content. Prohibited: "Easily integrated with your existing..." / "With just a few lines of code..." / "The SDK makes it simple to..."

**Builder** — Integration-focused, outcome-before-mechanism, API-centric. Lead with what integration gives builder's product/users. State exact API endpoint or SDK method first. Show working code in JS/TS first. Full request/response cycle. State prerequisites (API keys, SDK version, plan tier). Prohibited: Opening that explains network before integration value.

**Delegator** — Accessible, plain-language, decision-support. Lead with outcome (what staking/delegating/voting gives). Explain protocol terms on first use, then use freely. Use plain language. State actual steps. No code blocks or CLI commands. No DeFi jargon assumed. Prohibited: "Simply stake your tokens to..." / unquantified earnings claims.

**Community** — Inclusive, participatory, substantive. Warm but not effusive. Concrete and actionable. Route to role-specific sections rather than duplicating content. State what participation requires (time, skills, tools, credentials). Prohibited: "Join our thriving community!" / "Together, we can..." / "The community is the heart of..."

**Founder** — Executive summary, outcome and economics before mechanism. Direct, credible, numbers over adjectives, no hype. Lead with business outcome. State unit economics with numbers: "70% lower than AWS MediaConvert". Frame technical details as business implications: "sub-3-second latency → usable for live commerce". Prohibited: "Powered by blockchain technology..." / "The future of decentralised..."

### Three-Tier Enforcement

| Tier | Meaning | Example |
|---|---|---|
| **Blocking** | Merge blocker or CI blocker — must fix | Banned words, banned phrases, banned constructions |
| **Warning-only** | Rewrite by default, keep accurate exceptions | Opening order, description field format, comparative framing |
| **Manual sign-off** | Human review decides | Operator benefit accuracy, canonical terminology, product truth |

---

### The Master Test

Before any sentence ships:

> Does this sentence give the reader something they can act on, trust, or use to make a decision, stated directly and in the right order for the page?

If no: rewrite or delete.

---

### Banned Constructions (Blocking)

**`not [X]` in value statements** — Rewrite as positive assertion. Delete contrast.
- ❌ "This is not just a simple relay"
- ✅ "The gateway routes, load-balances, and negotiates pricing"

**`if [condition]` in body prose** — Do not make reader resolve a knowledge gap the page should close. Exception: code blocks, prerequisite lists, exact configuration conditions.
- ❌ "If you have enough VRAM, you can run transcoding"
- ✅ "Transcoding requires 8 GB VRAM minimum"

**`[value statement] - if [condition]`** — Split into two sentences. Value first. Requirements second.

**`rather than`** — Delete contrast clause. State the positive directly.

**`This page [verb]`** — Delete. Start with the content.

**`consistently` without a number** — Add the threshold or rewrite.

**`can/may [verb]` in value claims** — Assert directly or delete the claim.

---

### Description Field (Warning-Only)

The frontmatter `description` is a one-sentence outcome statement, not a table of contents.
- Open with reader outcome or fit — not "this page" or "overview of"
- Target 160 characters or fewer
- Avoid announcement framing ("The three alternatives to...")

---

### Comparative Framing (Warning-Only)

Do not make comparison the entry point when the thing itself has not been defined.
- ❌ "Unlike centralised CDNs, Livepeer..."
- ✅ Define what Livepeer does, then compare where it genuinely helps the decision

---

### Structural Rules

- One paragraph, one job
- Lead sentence states the most useful fact for the reader
- Final sentence ends on fact, number, or next step — never a hedge
- Tables and diagrams stand alone; duplicate prose is deleted
- High-value URLs and commands appear in body copy, not only inside accordions
- `{/* REVIEW: */}` flags are internal only — never rendered; block merge if unresolved in decision-critical positions

### Section Heading Rules

**DO**: Use governing-concept content descriptors, technical/professional descriptors, under 3 words.

**DON'T**: Questions, question-frame words (What, How, When, Why), sentences, quantified/numbered framing, numbers, comparators (Differences, vs), literal labels, generic descriptors, taxonomy mismatches.

| Bad Heading | Good Heading |
|---|---|
| How Pools Work | Pool Mechanics |
| What You Earn | Rewards / Incentives / Profitability |
| Before You Start | Setup Checklist / Requirements |
| What Changes | Deployment Modes / Node Modes |
| Three Setup Paths | Deployment Types |
| The Two Workload Types | AI Runtime Pipelines |
| When to use this setup | (resolve in body content — not a heading) |

---

### Section Naming Rubric (Locked)

Core rule: **Name the concept, not the examples.**

| Label Class | What it does | Verdict |
|---|---|---|
| Governing-concept label | Names the concept those examples instantiate | Use |
| Content descriptor | Names what the section is about | Use |
| Literal label | Names the visible examples directly | Penalise |
| Comparator label | Names the comparison, not content | Reject |
| Generic descriptor | Valid umbrella, too weak | Penalise |
| Taxonomy mismatch | Wrong conceptual category | Reject |

Weak-label words to penalise: `types`, `profiles`, `modes`, `models`, `overview`, `guide`, `basics`, `essentials`, `foundations` — unless genuinely classifying.

- No more than one major layout element (Table, Steps) per page unless nested
- No duplicate layout elements of the same type
- Related Pages description must not repeat or rephrase the title
- TODO/TBD/Coming Soon placeholders blocked on all page types

### Production-Grade Approach: CSS Custom Properties

Mintlify uses CSS Custom Properties defined in `style.css` at repo root. This is the ONLY production-grade approach.

- CSS Custom Properties (`--variable-name`)

### What Makes These Good

1. Entity-led voice — systems and users as subjects, never documentation
2. Concrete numbers — VRAM specs, quorum percentages, funding amounts, contract addresses
3. Decision support — tables, trees, and matrices that help readers choose
4. Governing-concept headings — professional terminology, under 3 words
5. Proper components — Tabs, Accordions, Steps, Cards used per page-type contract
6. Clear frontmatter — status, lastVerified, pageType, audience, purpose all populated

### What Makes These Bad

1. **Self-referential language** — "This page explains...", "Here's a guide...", "In this section..."
2. **Placeholder content** — Routed pages with `{/* Content to be written */}` comments only
3. **Scattered comments** — Incomplete refactoring visible to anyone reading MDX source
4. **Emoji bullets** — Using emoji as content markers violates voice guidance
5. **Vague openers** — "Get started on your journey by reading about Livepeer..."
6. **Generic approach** — Trying to serve all audiences without voice register differentiation
7. **No concrete specifics** — Missing numbers, dates, requirements, or actionable detail
