# Livepeer Release Updater

## Canonical Source

- Canonical release data module: `snippets/data/globals/latestRelease.jsx`
- Legacy compatibility shim: `snippets/automations/globals/globals.jsx`
- Legacy MDX wrapper: `snippets/automations/globals/globals.mdx`

Only `snippets/data/globals/latestRelease.jsx` should be written by automation.
The legacy files exist for compatibility and must remain thin re-exports.

## Automation Contract

- Workflow entrypoint: `.github/workflows/update-livepeer-release.yml`
- Shared workflow implementation: `.github/workflows/data-refresh-governance.yml`
- Script entrypoint: `.github/scripts/update-livepeer-release.js`

The script:

- accepts optional `--version <tag>`
- fetches `livepeer/go-livepeer` latest release when no override is provided
- writes exactly `latestVersion` and `latestReleasePageUrl`
- exits non-zero on fetch, parse, or write failure

## Manual Run

```bash
node .github/scripts/update-livepeer-release.js
node .github/scripts/update-livepeer-release.js --version v0.8.10
```

## Output Shape

```jsx
export const latestVersion = "v0.8.10";
export const latestReleasePageUrl =
  "https://github.com/livepeer/go-livepeer/releases/tag/v0.8.10";
```

Do not reintroduce `latestVersionUrl` into the canonical module. If an older
consumer still needs that name, it must come from the legacy shim only.
