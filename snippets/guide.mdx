# Snippets Guide

This file is the canonical human-authored governance guide for `snippets/`.

Use it for:
- folder purpose and placement rules
- how to create new snippet items
- generated versus manual ownership rules
- the machine-readable folder metadata consumed by `snippets/snippets-registry.mdx`

Do not hand-edit `snippets/snippets-registry.mdx`. Regenerate it from this guide and the live filesystem.

## Scope

`snippets/` is the shared import namespace for reusable docs infrastructure content. It is where the repo stores:
- shared MDX composables
- governed data modules and generated feed outputs
- static assets used by docs pages and components
- authoring templates
- the governed component library subtree

The generated root registry intentionally ignores:
- `snippets/_workspace/`
- `snippets/automations/`

Those folders still exist for ongoing migration and working-state reasons, but they are not part of the canonical steady-state snippets taxonomy.

## Root Contract

The governed root documentation surface is exactly two files:
- `snippets/guide.mdx`
  The only manual source of truth for snippets taxonomy, rules, and folder descriptions.
- `snippets/snippets-registry.mdx`
  Fully generated from the live `snippets/` tree plus the metadata block embedded in this guide.

No other root-level snippets governance file should compete with these two files.

## Folder Taxonomy

### `assets/`

Use `assets/` for static, non-code files that are consumed by docs pages, metadata, or generators.

Put here:
- images
- logos
- videos
- downloadable files
- raw workflow exports or support artifacts that should be retained but not executed as live code

Do not put here:
- JSX or MDX data modules
- page composables
- templates
- active generator scripts

### `components/`

`components/` is governed separately.

Use:
- `snippets/components/README.md`
- `docs-guide/frameworks/component-framework-canonical.mdx`
- `v2/resources/documentation-guide/component-library/overview.mdx`

This root guide only points to that governance. It does not restate component placement rules.

### `composables/`

Use `composables/` for MDX imported into other MDX pages.

Put here:
- shared page sections
- page fragments reused across routes
- imported external-doc MDX fragments that are rendered as page content

Do not put here:
- JSX data exports
- raw assets
- one-off page-local files that should live beside their route instead

### `data/`

Use `data/` for shared data modules, generated feed outputs, reference datasets, snapshots, variables, and similar non-component content inputs.

Put here:
- generated `.jsx` feed outputs
- reference maps
- snapshots and companion data
- shared variables and content constants
- solution-specific data grouped by source family

Do not put here:
- page composable MDX
- static images/files
- n8n workflow definitions unless they are explicitly being retained as archived data assets

### `templates/`

Use `templates/` for copy-ready authoring scaffolds.

Put here:
- block templates
- page templates
- internal docs-guide template sources

Do not put here:
- active page content
- generated data
- runtime components

## Placement Rules

When adding something new:

1. If it is an `.mdx` file imported into another `.mdx` page, place it under `composables/`.
2. If it is a data module or generated content input for pages/components, place it under `data/`.
3. If it is a static file or retained workflow export, place it under `assets/`.
4. If it is a reusable UI/runtime component, do not decide from this file; route it through component governance under `components/`.
5. If it is a starter scaffold for future authoring, place it under `templates/`.

If a file does not fit one of those lanes, stop and resolve taxonomy first. Do not create ad hoc top-level buckets.

## Generated vs Manual Ownership

Follow these rules:
- Manual guidance lives in `guide.mdx`.
- The registry is generated and must be overwritten, not edited.
- Generated data files in `snippets/data/**` should be refreshed by their owning scripts/actions rather than patched by hand unless the owning workflow is intentionally being replaced.
- Files under `assets/` may be manual or generator-owned; the deciding factor is the producer, not the extension.
- `_workspace/` is for working state, reports, and archival material, not canonical steady-state governance.

## What Does Not Belong Here

Do not introduce:
- a third root governance file for snippets
- duplicate folder-framework docs at the snippets root
- active generator scripts under `snippets/`
- ambiguous holding folders that hide ownership instead of clarifying it
- component governance duplication in this file

## Maintenance Workflow

When the snippets structure changes:

1. Move or create the real folder in `snippets/`.
2. Update the folder metadata block in this guide.
3. Regenerate `snippets/snippets-registry.mdx`.
4. Run the snippets root governance validator and the relevant script governance checks.
5. Update any live docs that describe the snippets root contract if they would otherwise drift.
