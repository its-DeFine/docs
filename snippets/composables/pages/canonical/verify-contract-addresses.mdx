## Browser Verifier

Use the tool below to query the blockchain directly from your browser.

<Accordion title={<CustomCardTitle variant="accordion" icon={<Icon icon="info-circle" color="var(--accent)" />} title="How this verifier works" />}>
  This verifier checks the published contracts data against live chain and explorer information.

  <StyledSteps>
    <StyledStep title="Look up a contract by name" icon="terminal">
      Select a Livepeer contract from the dropdown. For contracts registered in the on-chain Controller, the verifier calls
      `getContract()` via a public Arbitrum RPC and compares the returned address against the canonical reference. For
      standalone and bridge contracts outside the Controller, it verifies the address directly against explorer data.
    </StyledStep>
    <StyledStep title="Verify an address you already have" icon="magnifying-glass">
      Paste an address and the verifier checks whether it is a contract, whether source code is verified, and whether the
      address matches a published Livepeer contract on either Arbitrum One or Ethereum Mainnet.
    </StyledStep>
  </StyledSteps>

  <Note>
    RPC calls are only made after you click a button. If public endpoints are unavailable, the verifier falls back to showing a
    `cast` command that you can run locally.
  </Note>
</Accordion>

## No-Trust On-Chain Verification

Every Arbitrum One controller-managed contract can be verified independently without trusting this documentation or the browser
 verifier.

<Accordion title={<CustomCardTitle variant="accordion" icon={<Icon icon="key" color="var(--accent)" />} title="See the manual verification steps" />}>
  <StyledSteps>
    <StyledStep title={<span>Call <code>getContract</code> on the Controller</span>} icon="terminal">
      Query `getContract(keccak256("<ContractName>"))` on the Controller at `0xD8E8328501E9645d16Cf49539efC04f734606ee4`
      on Arbitrum One (`42161`). The returned address is the current controller-registered implementation or proxy address.

      ```bash title="Controller getContract Example" icon="terminal"
      # Example: verify the BondingManager proxy address
      cast call 0xD8E8328501E9645d16Cf49539efC04f734606ee4 \
        "getContract(bytes32)(address)" \
        $(cast keccak "BondingManager") \
        --rpc-url https://arb1.arbitrum.io/rpc
      ```

      A valid result returns a non-zero address:

      ```txt title="Example Controller Response" icon="terminal"
      0x35Bcf3c30594191d53231E4FF333E8A770453e40
      ```

      If the output is `0x0000000000000000000000000000000000000000`, the contract name is misspelled or not registered on this chain.

      <InlineDivider margin="0.5rem 0" />
    </StyledStep>

    <StyledStep title="Cross-check against the canonical reference" icon="list-check">
      Confirm that the returned address matches the current row on
      [Contract Addresses]({contractsRoutes.reference}), then inspect the row's explorer link and source-code link.

      <InlineDivider margin="0.5rem 0" />
    </StyledStep>

    <StyledStep title="Inspect on the explorer" icon="triangle-exclamation">
      Paste the address into [Arbiscan](https://arbiscan.io) or [Blockscout](https://arbitrum.blockscout.com) and confirm:

      - The address is a contract, not an EOA.
      - The contract source is verified.
      - The explorer label and transaction history are consistent with Livepeer deployment activity.

      <Danger>
        An address with no verified source, no contract bytecode, or no relationship to the published controller lookup result is
        not a valid Livepeer protocol contract address.
      </Danger>
    </StyledStep>
  </StyledSteps>
</Accordion>
